# Editor-Driven `AGENTS.md` Section Regeneration

## Summary

The Agent Editor becomes the only thing that ever rewrites four named sections of `AGENTS.md` (`## Folder Structure`, `## Skills & Tools`, `## Knowledge Bases`, `## Workflows`). On every editor write — `put`, `delete`, `move`, `create-sub-agent` — those four sections are re-rendered in place from the live S3 tree and DB state; everything else in the file is preserved verbatim. All other auto-rewrite pathways (skill mutations, KB mutations, customize render, folder-bundle-importer, eval-agent-provisioning) stop touching `AGENTS.md`. Bootstrap remains the one carve-out: it seeds the initial file at agent creation and never auto-writes again.

---

## Problem Frame

`AGENTS.md` is the always-loaded workspace map. It is the first context the agent and any human operator see, so its shape and accuracy directly determine how legible the workspace is.

Two failure modes coexist today:

**The rendered shape is wrong.** The `renderAgentsMap` function in `packages/api/src/lib/workspace-map-generator.ts` produces a folder tree only two levels deep, hard-coded to assume every workspace contains exactly `CONTEXT.md + docs/`. It never walks the actual S3 prefix, so root-level folders like `skills/`, `memory/`, `review/`, `events/` are invisible, and anything nested below the workspace's `docs/` line is silently dropped. The live `earnest-falcon-947` / `jovial-narwhal-612` / `resilient-otter-384` workspaces are further drifted — their `AGENTS.md` files carry a flat bullet-list "Generated by repair-agent-workspace-blueprint" snapshot from an external script that has no source in this repo, so they have already lost the section structure the generator expects.

**Multiple writers race over the file.** `regenerateWorkspaceMap` re-renders the entire `AGENTS.md` on skill add/remove, KB add/remove, customize-render, folder-bundle-import, eval-agent-provisioning, and bootstrap. The editor never triggers it on file CRUD, but every adjacent system does — and each rewrite stomps any hand-edited prose. There is no stable surface on which an operator can write Quick Navigation rows, ID conventions, or token-management guidance and expect it to survive the next skill toggle.

The combined effect: the file that should be the workspace's most authoritative artifact is the one most likely to be both visually wrong and silently overwritten.

---

## Actors

- A1. **Operator (admin SPA user)**: uses the Agent Editor to create, move, rename, and delete workspace files. Today triggers no map regeneration; under this proposal, becomes the sole driver of `AGENTS.md` section updates.
- A2. **Agent Editor backend** (`packages/api/workspace-files.ts`): receives `put`, `delete`, `move`, `create-sub-agent` actions and writes to S3. Under this proposal, it becomes the single integration point that re-renders the four derived sections after every successful write.
- A3. **Runtime agent (Strands container)**: reads `AGENTS.md` at boot via the manifest. Continues to write to non-map paths (memory, artifacts) as it does today. Never writes to `AGENTS.md`.
- A4. **Bootstrap (`handlers/bootstrap-workspaces.ts`)**: writes the initial seed of `AGENTS.md` and related map files at agent creation. Carve-out: runs once per agent, never again.

## Requirements

**Section-scoped rewriter**

- R1. The Agent Editor backend must expose a single internal helper that takes an existing `AGENTS.md` document plus the agent's tenant/agent identifiers and returns a new document with exactly the four named sections re-rendered in place: `## Folder Structure`, `## Skills & Tools`, `## Knowledge Bases`, `## Workflows`.
- R2. Section boundaries are detected by the `## <Heading>` line and end at the next `---` horizontal rule or the next `## ` heading, whichever comes first. The helper must replace only the content between those boundaries; the heading line itself and the trailing divider (if present) are preserved.
- R3. When a named section is absent from the existing document, the helper must append it at the end of the file in the canonical order above, preceded by a `---` divider. This is what allows already-drifted files to gain the sections on their first editor write.
- R4. All content outside the four named sections must round-trip byte-identical. The helper performs no whitespace normalization, no trailing-newline adjustment, no header reflow on preserved regions.

**Folder Structure rendering**

- R5. `## Folder Structure` is rendered as a fenced code block containing a tree drawn from a recursive walk of the agent's S3 workspace prefix. The root line is the agent's slug followed by `/`; descendants use the `├──`, `└──`, `│   `, `    ` box-drawing convention shown in `/Users/ericodom/projects/workspace-blueprint/CLAUDE.md`.
- R6. The tree must show every folder discovered in the workspace, not a hard-coded skeleton. Reserved root folders (`memory/`, `review/`, `events/`, `skills/`) appear when present; specialist workspace folders appear at their actual depth; nested subfolders under any workspace are rendered up to the full depth of the discovered tree.
- R7. Each top-level entry may carry an optional inline annotation following a `← ` marker. Annotations come from two sources: (a) a small fixed map in code for known paths (`AGENTS.md`, `CONTEXT.md`, and the reserved root folders `memory/`, `skills/`, `review/`, `events/`), and (b) for specialist workspace folders that contain a `CONTEXT.md`, the annotation is the H1 of that `CONTEXT.md` (or the first prose line below it if the H1 is generic). Folders that match neither rule render with no annotation.
- R8. The tree must not include system files such as `.DS_Store`, `.gitkeep`, or any path with a leading dot in its basename.

**Other derived sections**

- R9. `## Skills & Tools`, `## Knowledge Bases`, and `## Workflows` are rendered from the same DB queries `regenerateWorkspaceMap` uses today; the table shapes (columns, escaping, truncation) carry forward unchanged so the visible content does not regress.
- R10. When a derived section has no rows (no skills assigned, no KBs assigned, no workflows configured), the section renders its empty-state line ("No skills assigned.", etc.) as today. The section itself is never omitted.

**Editor integration**

- R11. The section-scoped rewriter must run on the server side as the final step of every successful `put`, `delete`, `move`, and `create-sub-agent` action whose target is an agent workspace (`target.kind === "agent"`). Non-agent targets (templates, spaces, computer, user) are unaffected.
- R12. The rewriter runs after the primary S3 write/delete/move succeeds; failures inside the rewriter must surface in the action's response and must not corrupt the existing `AGENTS.md` (atomic full-file replace, no partial writes).
- R13. The existing `regenerate-map` action stays as a manual escape hatch and invokes the same section-scoped rewriter (not the legacy full-file render).

**Retirement of background rewriters**

- R14. `regenerateWorkspaceMap` calls must be removed from: skill assign/unassign mutations, KB assign/unassign mutations, customize-render flows, `folder-bundle-importer`, and `eval-agent-provisioning`. After this change, none of these pathways modify `AGENTS.md`.
- R15. The legacy whole-file `renderAgentsMap` function and its `renderContextRouter` companion may be deleted once R14 leaves them unused, except for the subset of rendering logic shared with the new section-scoped helper (table rendering for skills/KB/workflows).
- R16. The `regen-all-workspace-maps.ts` script and its post-deploy `migration-precheck` invocation, if any, are retired or rewritten to invoke the new section-scoped rewriter against every agent.

**One-time normalization**

- R17. Normalization of a drifted `AGENTS.md` is operator-triggered, per-agent, from the admin UI. The trigger may extend the existing `rematerialize` action or be a new "Normalize `AGENTS.md`" affordance scoped to the four derived sections plus the blueprint's hand-authored sections. The server-side normalizer replaces the existing flat content with a freshly-seeded document that contains the four derived sections plus the blueprint's hand-authored sections (What This Is, Quick Navigation, etc.) drawn from a canonical template. The three currently-drifted live agents (`earnest-falcon-947`, `jovial-narwhal-612`, `resilient-otter-384`) are normalized by the operator running this action against each of them once. No automatic post-deploy or CI sweep performs normalization on the operator's behalf.

**Bootstrap unchanged**

- R18. Bootstrap continues to write `AGENTS.md`, `CONTEXT.md`, `IDENTITY.md`, `SOUL.md`, `USER.md`, and any other seed files at agent creation. The initial `AGENTS.md` it writes must contain all four derived sections (so the section-scoped rewriter has well-formed boundaries from the first editor save) and the blueprint's hand-authored sections (so operators have a starting place to edit).
- R19. After bootstrap completes, no code path other than the Agent Editor may write to `AGENTS.md`. The `rematerialize` action (operator-triggered re-seed when an agent has drifted past repair) is the one operator-controlled exception and is unchanged.

## Acceptance Examples

- AE1. **Covers R1, R2, R4.** Given an `AGENTS.md` with sections `## What This Is`, `## Folder Structure`, `## Quick Navigation`, `## Skills & Tools`, and a hand-edited `## Token Management` block at the end. When the operator saves a file edit through the editor, then `## Folder Structure`, `## Skills & Tools`, `## Knowledge Bases`, and `## Workflows` are re-rendered and the other four sections (including the hand-edited Token Management) round-trip byte-identical.

- AE2. **Covers R3.** Given an `AGENTS.md` that contains only `## What This Is` and `## Folder Structure` (no Skills, KB, or Workflows sections). When the operator saves any file edit, then the document gains a `## Skills & Tools`, `## Knowledge Bases`, and `## Workflows` section appended at the end in that order, each preceded by `---`.

- AE3. **Covers R5, R6.** Given a workspace whose S3 prefix contains `skills/account-health-review/`, `memory/`, `review/`, `earnest-falcon-947/CONTEXT.md`, and `earnest-falcon-947/skills/renewal-prep/`. When the rewriter runs, then the `## Folder Structure` code block shows all of those paths in a single recursive tree, with `earnest-falcon-947/skills/renewal-prep/` rendered as a third-level child of the `earnest-falcon-947/` entry.

- AE4. **Covers R8.** Given a workspace prefix that contains `.DS_Store` files at multiple depths and a `.gitkeep` placeholder. When the rewriter runs, then none of those dotfiles appear in the rendered tree.

- AE5. **Covers R11, R14.** Given an agent whose Skills table in `AGENTS.md` is out of date relative to the DB (e.g., a skill was added via the admin Customize page yesterday). When no editor save happens, then `AGENTS.md` stays stale — the skill addition does not retroactively edit the file. When the operator subsequently saves any file edit through the editor, then `## Skills & Tools` is re-rendered and the new skill appears.

- AE6. **Covers R12.** Given an editor `put` succeeds in writing the target file but the subsequent section-scoped rewrite throws (e.g., S3 transient error reading `AGENTS.md`). When the response returns, then it surfaces a 500 with a descriptive error AND the previously-stored `AGENTS.md` is unchanged on S3.

- AE7. **Covers R17.** Given the live `earnest-falcon-947` workspace whose `AGENTS.md` is in the flat bullet-list "Generated by repair-agent-workspace-blueprint" shape. When the one-time normalization pass runs, then the file is replaced with a document that has all four derived sections rendered from current state plus the blueprint's hand-authored sections seeded from the template, and the next editor save against that workspace preserves the seeded hand-authored sections verbatim.

## Success Criteria

- An operator can hand-edit any non-derived section of `AGENTS.md` (Quick Navigation, ID & Naming Conventions, Token Management, free-form prose) and that edit survives every subsequent skill toggle, KB assignment, customize-render, and bootstrap-rematerialize that ever runs against the agent. The only way it gets touched is if the operator opens the editor and overwrites it themselves.
- The `## Folder Structure` block in any agent's `AGENTS.md` accurately reflects the S3 prefix as of the last editor save: every workspace folder, every nested subfolder, every reserved root folder, with no dotfiles and no hard-coded skeleton lines.
- The three live drifted agents (`earnest-falcon-947`, `jovial-narwhal-612`, `resilient-otter-384`) end up with a well-formed `AGENTS.md` after the one-time normalization, and their first subsequent editor save produces an idempotent (byte-identical) `AGENTS.md` to demonstrate the round-trip works.
- A downstream implementer can hand this doc to `ce-plan` without needing to ask: which sections are derived, what triggers rewrite, what's preserved, what happens to the existing background callers, or how bootstrap behaves.

## Scope Boundaries

- The other blueprint sections (`## What This Is`, `## Quick Navigation`, `## Cross-Workspace Flow`, `## ID & Naming Conventions`, `## File Placement Rules`, `## Token Management`, `## Skills & Tools Available` table) are **not** derived. Bootstrap seeds them once from a template; thereafter operators hand-edit and the editor preserves them. Deriving any of these from data is out of scope and a candidate for a future brainstorm.
- Runtime / skill writes to non-map paths in the workspace (e.g., `memory-retain` writing `memory/*.md`, `artifact-builder` writing finished documents, Hindsight reflections) are out of scope. They do not touch `AGENTS.md` today and are unchanged here.
- Removing or restructuring the `agent_skills` table or any other DB-side skill catalog is out of scope. The four derived sections still read from current DB state during this work.
- `CONTEXT.md`, `IDENTITY.md`, `SOUL.md`, `USER.md` are out of scope. This brainstorm is about `AGENTS.md` only. (`renderContextRouter` may need a parallel teardown when its callers are removed, but its shape is not redesigned here.)
- Editor concurrency, locking, and conflict resolution when two operators save simultaneously are out of scope. Last-writer-wins matches today's behavior.
- Live re-rendering during typing or save-on-blur in the admin SPA is out of scope. The rewriter runs server-side per write action.
- The manifest regeneration system (`regenerateManifest`) is a separate concern, runs in addition to the section-scoped rewrite on every editor write, and is unchanged.

## Key Decisions

- **Editor is the single sync point.** Every other auto-rewrite pathway is removed. Operators see the rendered Skills/KB/Workflows tables lag behind DB state between editor sessions; this is the explicit trade-off for never having hand-edited prose stomped.
- **Bootstrap is the one carve-out.** Initial seed at agent creation runs as today; the principle "only the editor writes" is scoped to lifetime updates, not creation.
- **The principle is scoped to the four derived sections.** Runtime memory writes, skill artifact writes, and bootstrap remain unaffected. "Editor-only" does not mean "agent runtime becomes read-only against its own workspace."
- **Section boundaries are heading-anchored.** Replacing content between `## <Heading>` and the next `---` or `## ` is the contract. If an operator deletes a section's heading, the rewriter cannot find it and will re-append (R3) rather than stomp a different region.
- **One-time normalization is a separate operation from the editor change.** The three drifted live agents need a fix to seed well-formed sections before the editor-driven rewriter can do its job; this is called out so it does not get lost during implementation.
- **Annotations are source-driven, not hardcoded prose.** Reserved root folders use a small fixed map; specialist folders self-describe via their `CONTEXT.md` H1. This keeps the tree honest — when a specialist's role changes, the annotation updates on the next editor save with no separate metadata to maintain.
- **Normalization is operator-triggered, not automatic.** Per-agent admin UI button rather than a post-deploy sweep. The trade-off is operator memory (three agents × three clicks) for safety — an automatic sweep would silently stomp any prose an operator had added to a drifted file in the meantime.

## Dependencies / Assumptions

- The agent's S3 prefix is the authoritative source of truth for the folder tree. (Verified — `regenerateWorkspaceMap` already reads from S3 via `discoverWorkspaceFolders`.)
- The DB tables `agent_skills`, `agent_knowledge_bases` + `knowledge_bases`, and `routines` + `tenant_workflow_catalog` continue to back the three non-tree derived sections. (Verified against `workspace-map-generator.ts` lines 328–389.)
- A canonical `AGENTS.md` template containing the blueprint's hand-authored sections (What This Is, Quick Navigation, Cross-Workspace Flow, ID & Naming Conventions, File Placement Rules, Token Management) exists or will be written under `packages/system-workspace/` for bootstrap to seed from. The exact location and content is a planning-time decision.
- The `regen-all-workspace-maps.ts` post-deploy script, the `migration-precheck` CI gate, and any other auto-trigger of `regenerateWorkspaceMap` outside the file paths enumerated in R14 will be discovered during planning and retired or repointed.
