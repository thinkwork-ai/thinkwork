### P1 — needs follow-up

- **F-1 (correctness):** `packages/agentcore-pi/agent-container/src/server.ts:618` — `validateDirectTools` is silently skipped when `validatedConfigs.length === 0`. Plan §006 U4 specified fail-loud on directTools mismatches; today a typo never surfaces until configs eventually land. **Fix:** split the gate so `validateDirectTools` runs whenever `args.mcpJsonConfig.directTools.length > 0`; gate ONLY proxy registration on `validatedConfigs.length > 0`.
- **F-3 (reliability):** `packages/agentcore-pi/agent-container/src/server.ts:1478` — Cleanup queue drains MCP `transport.close()` serially with no per-step timeout. The new `DirectToolsValidationError` throw site widens the blast surface — a misspelled directTools entry against a tenant with a wedged MCP server now blocks the 500 response until Lambda timeout. **Fix:** wrap each `await fn()` in a 5s `Promise.race` timeout at both drain sites (`assembleTools`-throw path + run-loop `finally`).
- **F-4 (testing × security):** `packages/agentcore-pi/agent-container/tests/mcp-connect.test.ts` — toolWhitelist + registry-population ordering security invariant is documented but not test-asserted. A future refactor that moves `registry.register()` above the `.filter()` in `mcp-connect.ts` would silently make whitelisted-out tools addressable through U5's proxy `call` mode (authorization bypass). **Fix:** add a test that builds with `toolWhitelist: ["allowed"]`, fakes `tools/list` returning `["allowed", "hidden"]` against a real `McpToolRegistry`, and asserts `registry.get(server, "hidden") === undefined`.
- **F-5 (testing × security × reliability):** `packages/agentcore-pi/agent-container/tests/server.test.ts` — No integration test asserts cleanup queue drains + `handleStore.size === 0` on the `DirectToolsValidationError` path. Three reviewers converged. Current tests only assert the throw fires; the cleanup invariant rides on code review alone. **Fix:** add a `handleInvocation`-level test that fires `DirectToolsValidationError`, asserts `statusCode === 500`, asserts `bundle.handleStore.size === 0` after, and asserts at least one registered cleanup closure was invoked.
- **F-7 (kieran-typescript):** `packages/agentcore-pi/agent-container/src/mcp-proxy.ts:138` — `buildMcpProxyTool` returns `AgentTool<any>` which throws away typebox `Static<...>` narrowing. U5's discriminator-narrowing dispatch will pay the tax in hand-rolled `as McpProxyParams` casts. **Fix:** change return type to `AgentTool<typeof McpProxyParamsSchema>` (the schema's TSchema type). The test file's `as never` casts (KT4) become unnecessary and should be rewritten to exercise the schema-accepted shapes.
- **F-AGENT-1 (agent-native):** `docs/plans/2026-05-23-006-feat-agentcore-pi-mcp-proxy-tool-plan.md` — Plan claims operators OR skills can author `mcp.json`, but the Pi runtime has no workspace-write tool. `bootstrap-workspace.ts` is one-way S3 → local at handler entry; nothing pushes back to S3. **Fix:** either (a) amend the plan's Scope Boundaries to state explicitly that agent self-configuration of directTools is deferred to a separate workspace-write workstream, or (b) commit a `write_workspace_file` tool in U5 that round-trips writes to both `/tmp/workspace` AND the agent's S3 prefix. Memory rule `feedback_filesystem_is_the_agent`: if the filesystem IS the agent, the agent must be able to write to it.
- **F-A1 (adversarial):** `packages/agentcore-pi/agent-container/src/server.ts:1357` — Pre-existing bootstrap-failure-swallow is now amplified: `mcp.json` is a security-relevant allowlist substrate that this PR places on the same broken substrate. Tenant A's `mcp.json` could appear during tenant B's invocation if the S3 sync fails silently on a warm container. **Fix:** fail-closed when bootstrap fails on a warm container. Worth a dedicated PR — pre-existing risk that this work elevates from "skill content" to "security boundary."

### FYI (anchor 50 — advisory)

- **F-8:** `mcp-json.ts` trims `server`/`tool` fields; registry compares raw MCP names. Spurious validation failures if an MCP server emits whitespace-padded names.
- **F-9:** `McpToolRegistry.register()` silently overwrites duplicate `(server, tool)` entries with different schemas. Confused deputy if a misbehaving MCP server returns duplicates.
- **F-13:** Proxy parameter schema has no `oneOf` discriminator — U5's live body may silently dispatch the wrong branch on `{ list: true, call: {...} }`. Encode `Type.Union([...])` exclusivity in U5.
- **F-AGENT-2:** No agent-readable way to introspect current `mcp.json`. U5 should annotate `mcp({ list: true })` results with `direct: true|false` so the agent sees zero-hop vs proxy-routed.
- **A6:** `McpToolRegistry.get/entries/search` returns mutable references; callers can corrupt the registry. Lock down before U5 puts the registry behind a model-visible interface.
- **R4:** `readMcpJson` swallows all `readFile` errors as "missing file" — EACCES / EISDIR / EMFILE / ELOOP silently demote to empty config. Narrow the catch to `ENOENT`.
- **M1–M4:** Maintainability cluster — speculative `BuildMcpProxyToolOptions` fields, unused exports (`McpProxyParams`, `McpProxyMode`), `McpJsonConfig` index signature with no consumer, two-branch proxy registration that could be always-emit-with-empty-registry. Revisit after U5/U6 lands.
