Connectors wire external services into the Thinkwork thread system. When a Slack message arrives, a GitHub issue is filed, or a Google Calendar event triggers — a connector turns that external event into a thread and routes it to the right agent.

Connectors are bidirectional: agents can also send messages, create issues, reply to emails, and post updates back to the originating system through the same connector.

## Built-in connectors

### Slack

The Slack connector enables two flows:

**Inbound** — A user messages a Slack app (direct message or mention in a channel). The connector creates a `SLACK-` thread, assigns it to the configured agent, and the agent's response is posted back to Slack in real time.

**Outbound** — An agent or automation can post messages to Slack channels using the `slack_send_message` tool (available via the Slack skill pack).

Setup requires:
1. Creating a Slack app in your workspace with the required OAuth scopes
2. Configuring the Slack connector in the admin app with your app credentials
3. Setting the event subscription URL to your API Gateway endpoint

Required OAuth scopes: `chat:write`, `im:history`, `im:read`, `app_mentions:read`, `channels:history`

```
Slack message
    → Slack Events API
    → API Gateway /connectors/slack
    → Connector Lambda (validates signature, creates thread)
    → AgentCore invoked
    → Response posted back to Slack
```

### GitHub

The GitHub connector handles:

- **Issues** — New issues become `GH-` threads; agents can triage, label, comment, and close
- **Pull requests** — PR events create threads; agents can review, suggest changes, approve
- **Issue comments** — Mentions (`@agent-name`) in issue comments trigger agent responses
- **Repository events** — Push events, workflow failures, and custom webhooks

Setup requires a GitHub App installed on your organization or repository, with a webhook pointed at your API Gateway endpoint.

Required permissions: `issues:write`, `pull_requests:write`, `contents:read`

### Google Workspace

The Google Workspace connector includes:

- **Gmail** — Inbound emails create `EMAIL-` threads; agents can reply, forward, label
- **Google Calendar** — Event invites and reminders can trigger automations
- **Google Drive** — Documents can be referenced in agent context (requires Knowledge Base integration)

Setup uses OAuth 2.0. The connector handles the OAuth flow and stores tokens in the credential vault.

<Aside type="note">
Google OAuth requires a verified domain and an approved OAuth app for production use. For development, you can use test users and an unverified app with the appropriate disclaimer.
</Aside>

## Credential vault

All connector credentials are stored in the **credential vault** — a KMS-encrypted table in Aurora Postgres. Credentials never leave your AWS account.

The vault stores:

| Credential type | Example |
|-----------------|---------|
| OAuth access + refresh tokens | Slack, Google |
| Webhook signing secrets | Slack, GitHub |
| API keys | Custom connectors |
| Service account JSON | Google Workspace (service accounts) |

Credentials are encrypted at rest using a KMS key dedicated to the vault (created automatically during deploy). The Lambda functions that handle connector events have IAM permissions to decrypt only the credentials they need.

### Rotating credentials

Credentials with refresh tokens are refreshed automatically before expiry. For credentials that require manual rotation (API keys, webhook secrets), use the admin app:

1. Navigate to **Connectors → [Connector Name] → Credentials**
2. Click **Rotate**
3. Enter the new credential value
4. The vault updates and all active connector functions pick up the new value within 60 seconds (Lambda environment variable cache TTL)

## OAuth flows

For connectors that use OAuth (Slack, Google), Thinkwork handles the full OAuth dance:

```
Admin app: "Connect Slack"
    → Redirect to Slack OAuth authorize URL
    → User approves
    → Slack redirects to /connectors/slack/callback
    → Lambda exchanges code for access + refresh tokens
    → Tokens stored in credential vault
    → Connector marked as "active"
```

The OAuth callback URL is `https://<api-gateway-url>/connectors/<connector-id>/callback`. This URL must be registered with the external service as an allowed redirect URI.

## Connector recipes

A connector recipe is a template for building custom connectors. The `examples/connector-recipe/` directory in the Thinkwork repo contains a complete working example with:

- A Lambda handler for inbound events (webhook validation, thread creation)
- A skill pack with outbound tool definitions
- A Terraform module for the connector infrastructure
- An admin app plugin for the connector configuration UI

See [Authoring Guide: Connectors](/guides/connectors/) for the full walkthrough.

## Connector routing

When a connector creates a thread, it uses the **connector routing rules** to determine which agent handles it. Routing rules are evaluated in order:

| Rule type | Example |
|-----------|---------|
| Keyword match | If message contains "billing" → billing-agent |
| Channel/source | If Slack channel = #engineering → eng-agent |
| User mapping | If GitHub user = @alice → alice-agent |
| Default | Catch-all agent for the connector |

Routing rules are configured per connector in the admin app. If no rule matches, the thread goes to the connector's default agent.

<Aside type="tip">
For simple deployments, set a default agent on each connector and skip routing rules entirely. Add rules later when you need to fan work out to specialized agents.
</Aside>

<Aside type="note">
Looking to connect agents to external **tool servers** (MCP)? See [MCP Servers](/concepts/mcp-servers/) for connecting agents to third-party MCP servers that expose tools like CRM lookups, task management, and more.
</Aside>

## Thread metadata from connectors

Connector threads carry metadata about their origin:

```json
// Slack thread metadata
{
  "channelId": "C08ABC123",
  "messageTs": "1712345678.123456",
  "teamId": "T08XYZ456",
  "userId": "U08DEF789",
  "username": "alice"
}

// GitHub thread metadata
{
  "repoFullName": "acme/backend",
  "issueNumber": 1234,
  "eventType": "issues.opened",
  "actor": "bob",
  "labels": ["bug", "priority-high"]
}
```

The agent receives this metadata in its context on every turn, so it can tailor responses (e.g., addressing the Slack user by name, linking back to the GitHub issue).
