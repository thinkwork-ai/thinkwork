Connectors are ThinkWork's boundary with external systems. They watch tools such as Linear, Slack, GitHub, Google Workspace, CRMs, support queues, and internal APIs; normalize external activity into ThinkWork records; and write results back without exposing provider credentials to the runtime that performs the work.

The default owner for connector-origin work is a [Computer](/concepts/computers/). A connector claims an external item, records provenance, hands the work to the target Computer, and the Computer decides whether to handle it directly or delegate to a managed Agent, routine, or hybrid workflow. That ownership model keeps the user's durable workspace, credentials context, thread history, and delegated workers in one place.

## Connector shapes

ThinkWork uses the word connector for several related surfaces. They share the same trust boundary, but they enter the system differently.

| Shape              | What it does                                                                         | Current example                                  |
| ------------------ | ------------------------------------------------------------------------------------ | ------------------------------------------------ |
| Tracker connector  | Polls an external work system and reconciles matching items into Computer-owned work | Linear issues with the `symphony` label          |
| Integration        | Receives app, channel, or webhook events and can post replies back                   | Slack, GitHub, Google Workspace, custom webhooks |
| MCP tool connector | Gives managed Agents or Computers remote tools they can call during work             | Tenant-registered MCP servers                    |

All three keep ThinkWork's internal model intact: threads are the visible work record, Computers own continuity, managed Agents are delegated workers, credentials stay tenant-scoped, and every execution path leaves audit metadata.

## Computer-first routing

The connector path now starts with an explicit target Computer.

```text
External system
    -> connector claim / execution provenance
    -> Computer task and event
    -> Computer-owned thread
    -> optional managed-Agent or routine delegation
    -> provider writeback
```

That is different from the older "pick an agent for this connector" model. Direct managed-Agent, routine, and hybrid-routine targets can still exist as advanced automation paths, but they are not the main setup path for user-owned work. The normal product story is external system -> Computer -> delegated execution.

This matters because a Computer can preserve the workspace around the work item: local files, repository state, CLI setup, prior context, and future browser or desktop state. Managed Agents remain useful, but as hired workers behind the Computer rather than as the durable owner of the external item.

## The shipped proof: Symphony and Linear

The first production tracker connector is the Symphony Linear path. A tenant admin configures a Linear connector in [Admin: Symphony](/applications/admin/symphony/) with:

- a Linear team and label filter;
- the checkpoint label `symphony`;
- a Linear credential for issue reads and state writeback;
- a target Computer;
- a GitHub credential and repository settings for draft PR creation.

When a fresh Linear issue matches the connector, ThinkWork records a connector execution, creates a `connector_work` task/event for the Computer, opens a Computer-owned thread, delegates coding work, opens a draft PR, comments back on Linear, and moves the Linear card to `In Review` after the PR exists.

See [Connector Lifecycle](/concepts/connectors/lifecycle/) for the data flow and [Run the Symphony Linear Checkpoint](/guides/symphony-linear-checkpoint/) for the operator checklist.

## Credentials and trust boundary

Connector credentials are tenant-scoped secrets resolved by the connector runtime only when it needs to call the external provider. A managed Agent or Computer receives the normalized work context; it does not receive raw provider tokens by default.

The vault stores credentials such as:

| Credential type         | Example                        |
| ----------------------- | ------------------------------ |
| API keys                | Linear, custom providers       |
| OAuth tokens            | Google Workspace, Slack        |
| Webhook signing secrets | Slack, GitHub, custom webhooks |
| GitHub credentials      | PR-producing connector work    |

<Aside type="caution">
  A connector is a trust boundary. Inbound events must be authenticated or
  reconciled from a trusted API, outbound writeback must use tenant credentials,
  and duplicate provider deliveries must not create duplicate visible work.
</Aside>

## Idempotency and provenance

Connector-origin work is intentionally split into several durable records:

- **Connector** — configuration for one tenant/provider/target.
- **Connector execution** — provenance for one external item or poll result.
- **Computer task/event** — the Computer-owned handoff and audit stream.
- **Thread** — the visible work artifact for operators and users.
- **Delegation or turn** — the managed worker execution detail, when the Computer delegates.
- **Outcome metadata** — provider writeback, branch, PR URL, cleanup state, or failure detail.

This split lets the same external issue be polled many times without creating duplicate branches, duplicate PRs, duplicate Linear comments, or duplicate threads.

## Known limits

- **Linear is the only shipped tracker proof.** Slack-facing Computers, GitHub Issues, Jira, Salesforce, Zendesk, and other connector types should use this model, but they are follow-up plans.
- **Routing is explicit in v0.** A connector targets a chosen Computer. Automatic actor-to-Computer matching and team queues are not part of the current proof.
- **Symphony is the first connector experience, not the whole platform.** The broader connector framework still needs future connector types, custom recipe hardening, and richer catalog UX.
- **Advanced direct targets still exist.** They are useful for automation and internal paths, but Computer is the happy-path owner for user-facing connector work.

## Under the hood

Connector configuration and execution records live in `packages/database-pg/src/schema/connectors.ts` and `packages/database-pg/src/schema/connector-executions.ts`. The GraphQL contract lives in `packages/database-pg/graphql/types/connectors.graphql`, with resolvers under `packages/api/src/graphql/resolvers/connectors/`.

The Linear runtime path is implemented in `packages/api/src/lib/connectors/runtime.ts` and is invoked by `packages/api/src/handlers/connector-poller.ts`. The Symphony admin surface lives at `apps/admin/src/routes/_authed/_tenant/symphony.tsx`, with setup defaults and display helpers in `apps/admin/src/lib/connector-admin.ts`.
