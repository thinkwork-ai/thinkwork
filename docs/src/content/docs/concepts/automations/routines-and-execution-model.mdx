A single "invoke the agent with this prompt" is the simplest automation. Many useful automations are multi-step: fetch data, classify it, draft responses, wait for human approval, act on the results. ThinkWork calls these **routines**, and they're backed by AWS Step Functions state machines.

The design goal for routines is that each step is independently observable, retriable, and auditable — while still belonging to one coherent run that lands in one `AUTO-` thread.

## The execution model

Every routine is a Step Functions state machine. States can be:

- **Agent invocations** — call AgentCore with a prompt, wait for the response, continue.
- **Tool invocations** — call a specific tool or connector directly (not through an agent).
- **Branching** — `Choice` states that route based on prior output.
- **Waits** — `Wait` states or the human-in-the-loop pattern via the Inbox.
- **Parallel fan-out** — `Map` states that run the same step against many inputs.
- **Failure handlers** — `Catch` blocks that route failures to retry logic or explicit error states.

Step Functions handles the state transitions, retries, timeouts, and dead-letter behavior. ThinkWork provides the integration layer: every state's output becomes the next state's input, and every agent-invocation state threads through the same thread + memory system as interactive chat.

## A worked example

Consider a real routine: "process every new customer support email that arrived overnight."

```
Start
  │
  ▼
FetchOvernightEmails  (Lambda: queries the email system for unread items)
  │
  ▼
ForEachEmail  (Map state, concurrency=5)
  ├─ ClassifyEmail       (Agent: "is this urgent, routine, or spam?")
  ├─ Choice: isUrgent?
  │    ├─ Yes → DraftUrgentReply  (Agent: generate a tight, pre-approved reply)
  │    │         │
  │    │         ▼
  │    │      RequestApproval     (Inbox wait: operator approves or revises)
  │    │         │
  │    │         ▼
  │    │      SendReply           (Tool: post the reply via email connector)
  │    └─ No  → LogAndSkip        (just record that it was seen)
  │
  ▼
RecordRunMetrics  (Lambda: write summary to the routine's AUTO thread)
  │
  ▼
End
```

Several things are worth noticing:

- **Every agent invocation is a state.** They're not bundled into one giant "run the agent" step — each invocation is discrete, with its own input, output, and retry policy.
- **The `RequestApproval` state uses the Inbox.** A Step Functions "Task Token" is issued and persisted as an inbox item; an operator approving the item resumes the state machine with their decision.
- **Parallelism is explicit.** The `Map` state fans out up to 5 emails at a time; this is a state-machine-level parallelism guarantee, not an in-agent one.
- **Failures are trapped at each step.** If `ClassifyEmail` fails three times in a row for one email, that email's branch fails but the others keep going.

## The AUTO thread

The whole run has one `AUTO-` thread attached. Every agent-invocation state writes into that thread's turn log — so a single thread contains the Classify turn, the Draft turn, the approval event, and the send result. An operator debugging "why did this routine do the wrong thing on email 3?" opens that thread and walks the turn timeline.

For routines with very high fan-out (the Map state processes 10,000 items), ThinkWork supports a "run-per-item" mode where each Map iteration gets its own child `AUTO-` thread with the parent routine's thread as its `parentThreadId`. This keeps any one thread from drowning in a thousand turns.

## Retries and failure handling

Each state can configure:

- **Retry policy.** Intervals, back-off multiplier, max attempts, which error categories to retry on.
- **Catch handlers.** Route specific errors to fallback states (e.g., a classification failure routes to a "human classifies this one" state rather than failing the whole run).
- **Timeouts.** Per-state timeout beyond which the state errors out.
- **Dead-letter.** Failed executions can route to an SQS dead-letter queue for manual replay.

ThinkWork's Terraform module ships opinionated defaults for these — retries on `ThrottlingException` and `ServiceUnavailable`, no retry on `AccessDeniedException`, 15-minute per-state timeout — but each routine can override them per step.

## Where routines are defined

Two authoring surfaces:

- **Admin app Routines UI.** A visual builder for common patterns (the support-email example above would be authored here). Under the hood it generates an Amazon States Language (ASL) document and provisions a Step Functions state machine.
- **IaC.** Complex routines with custom Lambda steps or unusual branching are authored directly as Terraform modules, versioning the ASL JSON in the repo.

The IaC path is the right choice when the routine is going to live for years and needs code-review discipline. The UI path is the right choice for operational flows that will change quarterly.

## Observability

Three complementary views of a routine run:

1. **The AUTO thread** — narrative timeline with every agent turn and every operator decision.
2. **The Step Functions execution** — state-by-state status, input/output per state, retry counts, duration per state. Open via the admin app's Runs surface or the AWS console.
3. **CloudWatch metrics** — aggregate health across many runs (success rate, p95 duration per state, retry rate, dead-letter count).

For debugging one bad run, the thread + execution views are enough. For tracking routine health over weeks, CloudWatch is the one to watch.

<Aside type="tip">
  If a routine is failing intermittently, check the state-level retry metrics before the agent's output. A state that's retrying on every invocation means the upstream tool or connector is flaky, not that the agent is broken.
</Aside>

## Known limits

- **No pause-and-edit.** Once a Step Functions execution starts, you can't edit its state-machine definition mid-run. The ASL version running is the version that started.
- **No backward compatibility guarantee on routine edits.** Editing a routine in the admin app publishes a new ASL version. In-flight executions finish on the old version; new invocations use the new. If a step is removed, any execution waiting on that step fails.
- **Map state max concurrency is 40.** Step Functions limits inline `Map` concurrency. For higher fan-out, fan through SQS to independent state machines.
- **Human-in-the-loop waits have a 1-year max.** A task-token-based Inbox wait can live up to 1 year, but not beyond. Longer waits need periodic re-issue.

## Under the hood

- **State machines.** Provisioned by `terraform/modules/automations/` — one module per routine shape, or per-tenant custom instances.
- **ASL storage.** Routines authored in the admin app store their ASL JSON in `routines` Aurora table; a publisher Lambda diffs the stored ASL against the Step Functions state machine and updates on change.
- **Integration layer.** `packages/api/src/lib/routines/` — helpers for composing agent-invocation states, Inbox wait states, and connector-tool states.
- **Task token persistence.** Inbox items for human-in-the-loop waits store the Step Functions task token; resolving the inbox item issues `SendTaskSuccess` or `SendTaskFailure` to resume the execution.
- **Thread integration.** Each agent-invocation state passes the routine's thread id through its input, so every turn lands on the same thread.
- **Child thread pattern.** High-fan-out routines use `thread.parentThreadId` to link per-item child threads to the parent routine run.
