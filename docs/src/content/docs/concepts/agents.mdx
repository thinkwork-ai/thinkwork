An agent in ThinkWork is the harness's decision-making layer — a named entity that receives messages in a thread and produces responses. Agents read assembled context (the harness's [Perception phase](/architecture/#the-ppaf-agent-loop)), decide what to do (Planning), call tools (Action), and write back into the thread, where the turn's outcome becomes the Feedback the next turn perceives. Reliability and Security operating guarantees land here: templates pin model + guardrail behavior so two agents on the same template produce reproducible work, and capability grants determine which tools a given agent can ever fire.

ThinkWork supports two kinds of agents — **managed** and **connected** — and the difference is about *who owns the inference runtime*, not about what capabilities the agent has.

## Managed vs. connected

**Managed agents** are the default. ThinkWork runs them on **Bedrock AgentCore** inside *your* AWS account. Strands is the default managed runtime; Pi is a second managed runtime substrate for agents that you explicitly route to it. You get a consistent execution model, thread integration, streaming, auditability, and cost tracking without building your own orchestration.

The important distinction: managed does not mean vendor-hosted. ThinkWork makes running an agent easy; the harness, state, memory, guardrails, audit, and data still live inside your deployment boundary. Your tenants' data never leaves your AWS account.

**Connected agents** let you plug an existing runtime — your own Python service, a LangChain or LlamaIndex app, a custom container, a third-party hosted agent — into the ThinkWork thread and control plane. ThinkWork still handles auth, thread persistence, real-time updates, and the admin UI. You provide the inference logic.

A connected agent registers a webhook. When a message arrives in a thread assigned to it, ThinkWork POSTs the thread context to your endpoint; your runtime produces a response in the standard message format and posts it back.

<Aside type="note">
  Connected agents are the right choice when you already have agent infrastructure and want ThinkWork's thread system, connectors, memory, and admin surfaces *on top of* it without migrating your runtime. They're the wrong choice when you want the managed benefits (guardrail integration, per-turn cost capture, fleet-wide template rotation) — those only apply to the managed path.
</Aside>

Both kinds of agents show up identically in the admin app, the mobile app, the thread list, and the audit log. An operator sees "Agent X replied in thread Y" the same way regardless of where the inference ran.

## Managed runtime choices

Managed agents can run on either runtime:

- **Strands** — the stable default runtime and the path existing agents use unless changed.
- **Pi** — a parallel AgentCore runtime for testing and operating Pi-backed agents without leaving the ThinkWork harness.

The selected runtime is stored on the agent and template records. The dispatcher reads that selector at invocation time and routes the turn to the matching AgentCore runtime. Threads, memory, workspace files, connectors, audit, and client delivery stay the same across both runtimes.

See [Runtime Selection](/concepts/agents/runtime-selection/) for the operator model.

## What a managed agent has

Every managed agent is a record that references:

- A **template** (see below) — model, guardrail, inference parameters, memory behavior.
- A **system prompt** — the agent-specific instructions, stored on the agent record.
- Optional **skill packs** — bundled capabilities the agent can call.
- Optional **knowledge bases** — documents the agent retrieves from at turn time.
- Optional **connectors + MCP servers** — external tool surfaces the agent is allowed to use.

The agent itself is thin. Most of what determines how it behaves lives on the **template** it references.

## Templates: where the real configuration lives

Templates are the reuse layer for agent configuration. Rather than setting model, guardrails, budgets, and tool config per agent — and watching them drift over time — you set them once on a template and assign agents to it.

When you update the template, every agent on it picks up the change on the next invocation. That's how ThinkWork handles fleet-wide model rotations, guardrail policy tightening, or budget cap changes: one write, many agents.

See [Templates](/concepts/agents/templates/) for the live-configuration layer and [Skills](/concepts/agents/skills/) for the reusable-capability layer.

## How a turn runs (managed path)

When a message arrives in a thread assigned to a managed agent:

```
Thread message → API Gateway → runtime dispatcher
                                        │
                                        ▼
                              Load agent + template
                                        │
                                        ▼
                              Choose Strands or Pi
                                        │
                                        ▼
                              Assemble context
                              (history + docs + memory + wiki)
                                        │
                                        ▼
                              Bedrock Converse (guardrail applied)
                                        │
                                        ▼
                              Runtime loop: tool calls, reasoning, response
                                        │
                                        ▼
                              Write turn + messages back to thread
                                        │
                                        ▼
                              AppSync subscription → client
```

The selected runtime runs the inner agent loop — tool calling, multi-step reasoning, and response generation. ThinkWork provides everything surrounding it: thread state, memory access, template-driven model config, guardrail enforcement, cost capture, connector-provided tools.

See [Managed Agents](/concepts/agents/managed-agents/) for the runtime model in depth.

## Agent lifecycle

An agent record moves through a small state machine:

```
created → active → suspended → archived
```

- **created** — Agent exists but has never been invoked.
- **active** — Agent is accepting messages.
- **suspended** — Agent is paused. Usually set by the budget reconciler when a spend cap is hit, but operators can also suspend manually.
- **archived** — Agent is retired. Read-only; visible for audit but won't receive messages.

Suspended agents stop *on the next invocation boundary*. An in-flight turn when suspension happens finishes; subsequent messages get a "this agent is suspended" error until an operator resumes it.

## How agents relate to the rest of the system

Agents don't own threads, memory, or connectors — they *use* them. Every concept is composable:

- **Threads** — agents act inside threads, not outside them. See [Threads](/concepts/threads/).
- **Memory** — agents read document knowledge, recall long-term memories, and read their own compounding-memory wiki. They don't define what memory is. See [Memory](/concepts/knowledge/).
- **Connectors** — MCP tool connectors expose external capabilities the agent can call during a turn. See [Connectors](/concepts/connectors/).
- **Control** — guardrails wrap agent invocations; budgets cap their spend; audit records everything they do. See [Control](/concepts/control/).

<Aside type="tip">
  When you start with ThinkWork, read the concepts in this order: [Threads](/concepts/threads/), [Agents](/concepts/agents/), [Memory](/concepts/knowledge/), [Connectors](/concepts/connectors/). Once those four are clear, Control and Automations layer cleanly on top.
</Aside>

## Under the hood

### Model selection

Managed agents use Amazon Bedrock for inference. Any model available in your region through Bedrock can be referenced, but you must [request model access](https://docs.aws.amazon.com/bedrock/latest/userguide/model-access.html) in the AWS console first for each model. Model id is set at the template level, not per-agent — that's how a fleet-wide rotation is one update.

### Connected-agent webhook contract

Connected agents register a webhook URL. ThinkWork POSTs a payload with the thread id, canonical message history, optional retrieved memory, and channel-specific metadata. The connected runtime is expected to respond with a message in the same shape.

The exact contract and auth model are covered in [Admin: Agent Invites](/applications/admin/agent-invites/), which also documents how to issue an invite token that authorizes a BYOB runtime. For the GraphQL shape of the agent object itself, see the [API reference](/api/graphql/).
