An agent in ThinkWork is the runtime actor that reads a thread, plans the next step, calls tools, and writes the result back. The current managed-agent model is centered on a **single tenant platform agent** rather than a roster of separate top-level managed agents.

The tenant platform agent supplies the baseline identity, model, budget, sandbox, system prompt, and workspace. [Spaces](/concepts/spaces/) supply the local workroom: project context, membership, tools, memory, automations, email address, and optional runtime overrides.

<Aside type="note">
  If older docs, URLs, or records mention `/agents` or individual managed-agent
  ids, treat them as legacy compatibility surfaces. Current managed work flows
  through the tenant platform agent plus Space context.
</Aside>

## The current model

ThinkWork separates three layers:

| Layer                 | Purpose                                                            | Operator surface                                    |
| --------------------- | ------------------------------------------------------------------ | --------------------------------------------------- |
| Tenant platform agent | Tenant-wide baseline identity and instructions                     | [Admin — Tenant Agent](/applications/admin/agents/) |
| Space                 | Contextual workroom with local policy, tools, memory, and channels | [Admin — Spaces](/applications/admin/spaces/)       |
| Sub-agent folder      | Specialty behavior delegated by the platform agent                 | Tenant Agent → Sub-agents                           |

This keeps the operational model small. Operators tune the platform agent once, create Spaces for distinct work contexts, and add sub-agent folders only when a specialty needs its own instructions.

## What the tenant platform agent owns

The tenant agent owns the broad defaults:

- Name and role.
- Default Bedrock model.
- Default monthly budget.
- Default sandbox behavior.
- Baseline system prompt.
- Baseline workspace files.
- Folder-based sub-agent definitions.

These are tenant defaults, not project details. If a Space needs a different model, guardrail, budget, paused state, or sandbox mode, set a Space runtime override and leave the tenant default stable.

## What Spaces own

A Space gives the platform agent its immediate context. Spaces can carry:

- Workspace files and runbooks.
- Public or private membership.
- Built-in tools and MCP server availability.
- Knowledge bases and Space-scoped memory.
- Automations.
- Email triggers.
- Runtime overrides.

When a thread starts in a Space, ThinkWork invokes the tenant platform agent with the Space context rendered into the runtime workspace. The thread still has one agent identity for audit and cost accounting, but the behavior is shaped by the Space.

## Sub-agents

Sub-agents are authored as folders under the tenant agent's `subagents` workspace area. A sub-agent folder contains the instructions and supporting files for a specialty the platform agent can delegate to.

Use sub-agents for durable specialties such as research, triage, QA, drafting, or customer onboarding. Use a Space when the difference is context, access, channel, memory, or runtime policy.

## Managed and connected runtimes

Most tenants use the managed runtime path: ThinkWork runs the tenant platform agent on Bedrock AgentCore inside your AWS account. The harness, state, memory, guardrails, audit, and data stay inside your deployment boundary.

Connected agents still exist for bring-your-own-runtime integrations. They are useful when you already operate an external agent service and want ThinkWork's threads, auth, audit, and control plane around it. Connected agents are registered through [Agent Invites](/applications/admin/agent-invites/), not through the tenant platform agent page.

## How a turn runs

When a user message, email, automation, or integration event creates work:

```text
Thread event
  -> resolve tenant and Space
  -> load tenant platform agent
  -> apply Space runtime overrides or inherit tenant defaults
  -> render tenant + Space workspace context
  -> invoke managed runtime
  -> write messages, turns, cost, audit, and tool events back to the thread
  -> publish updates to admin and mobile clients
```

The Space can change what context and tools the agent sees. It does not create a separate top-level managed-agent identity for every workroom.

## Templates after the Space architecture

Templates remain useful authoring infrastructure, especially for reusable workspace and tool patterns. They are no longer the main day-to-day surface for tuning managed-agent identity across a fleet of top-level agents.

For normal operation:

- Configure the baseline in [Admin — Tenant Agent](/applications/admin/agents/).
- Configure local context and overrides in [Admin — Spaces](/applications/admin/spaces/).
- Use templates where the UI still exposes reusable authoring or migration flows.
