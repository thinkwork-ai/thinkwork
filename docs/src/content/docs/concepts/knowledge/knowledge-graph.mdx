---

The current open source Thinkwork docs focus on two implemented memory patterns:

- document knowledge through Bedrock Knowledge Bases
- long-term memory through AWS AgentCore LongTerm memory by default, or Hindsight when configured

A **knowledge graph** is the next likely direction for structured knowledge, not a fully documented product surface today.

## Why it matters

A knowledge graph would complement document retrieval by modeling entities and relationships directly, such as:

- customers and accounts
- services and dependencies
- incidents and linked decisions
- people, teams, and ownership

That could make retrieval more relationship-aware than document chunk search alone.

## How to think about it today

Treat this as roadmap context, not a deploy-and-use feature set in the current docs.

If you're evaluating Thinkwork now, plan around:

- threads as the system of work
- document knowledge for source material
- memory for retained context

Then watch the roadmap for structured knowledge graph capabilities.
