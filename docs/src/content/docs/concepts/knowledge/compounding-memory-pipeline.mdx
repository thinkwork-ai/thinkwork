A memory lands in your agent's head. Some time later, a wiki page exists that weaves that memory together with everything else your agent knows about the same subject. The pipeline is what happens in between.

This page walks through that journey in plain terms, then gets technical toward the end for anyone who needs to modify or debug the pipeline itself.

## The story

### Something memorable happens

You tell your agent about a new restaurant you tried. The agent decides it's worth keeping and writes a short memory note — just a sentence or two summarizing what happened.

That note lands in the raw memory store. It's the source of truth, but it isn't organized. It's just one of thousands of notes sitting in a timeline.

### A compile job gets queued

Not long after, a small background bit of code notices there are new memories to think about. It drops a compile job on the queue.

Compile jobs also get created in two other ways. An operator can explicitly trigger one for a specific agent ("go compile everything you know, right now"). And a nightly lint pass schedules jobs for mentions that have accumulated enough evidence to promote into real pages.

### The compile worker wakes up

The job gets picked up by the compile worker. The worker does one thing: turn a batch of new memories into updates for wiki pages.

It starts by asking, "what memories haven't I seen yet?" The worker keeps a bookmark called a **cursor** — the last memory it processed. It asks the memory store for everything that's changed since the bookmark, in order, up to 50 at a time.

### The planner reads everything in context

The worker hands the new memories to the **planner** — an LLM whose only job is to look at the memories alongside the agent's existing pages and say "here's what should change."

The planner sees:
- The new memories, in full.
- A summary of the pages that already exist in this agent's scope — titles, aliases, short summaries.
- Any unresolved mentions the agent has been holding (names that showed up but didn't justify a page yet).

The planner is explicitly biased toward caution. It knows that creating a spurious page is expensive — once a bad page exists it takes work to get rid of — while holding an alias as "unresolved" costs basically nothing. Given ambiguity, it holds.

### The planner produces a plan

The plan is a small JSON document with five sections:

**Page updates.** "This existing page should have its Visits section rewritten given this new memory."

**New pages.** "There's enough evidence for a new page about Taberna dos Mercadores. Here's a title, an initial set of sections, and some aliases."

**Unresolved mentions.** "Someone named 'Chef João' showed up in this batch. I don't have enough context to give him a page, but I'll note the mention — if he shows up three more times in the next thirty days, we should promote."

**Promotions.** "Remember 'Bairro Alto' that's been held as unresolved for a while? The new memories push it over the threshold. Promote it to a real topic page, here are the sections."

**Page links.** "Austin, Texas should link to Austin Kids Activities. Both pages exist; here's the context."

The planner never writes anything to the database. It just produces the plan.

### The section writer patches one section at a time

For every section the plan wants to update, the compile worker calls a second LLM — the **section writer**. Its job is narrower: take the existing section markdown, the rough draft the planner proposed, and the specific memories the planner cited. Produce the new markdown for just this one section.

Before calling the section writer, the worker asks "is this actually a meaningful change?" If the proposal is nearly identical to what's already there, the worker skips it entirely. No point burning tokens polishing prose that already says the same thing.

### Provenance gets written with deliberate care

Every section carries a list of which memories were used as sources. The rule is strict: **only the specific memories the planner cited for this section**. Not "all the memories in the batch." Not "every memory that looks related."

This rule exists because breaking it destroys the system. An earlier version of the compiler fell back to "use the whole batch as sources" when the planner omitted citations. The result was nonsense: a memory about Austin weather got written as a source for unrelated CRM pages that happened to be in the same batch. The reverse-lookup (`MemoryRecord.wikiPages`) then returned those CRM pages when you tapped the weather memory. Search surfaced them. Everything downstream broke.

The fix was to stop the fallback. Zero cited memories means zero source rows. Blank provenance is always preferable to wrong provenance.

### The cursor advances

Once the plan applies cleanly, the worker moves its bookmark forward to the last memory it just processed. If anything failed mid-batch, the bookmark stays where it was — the next worker will re-read the same memories safely.

### The worker loops

The worker keeps grabbing the next batch of 50 memories and running them through the planner + section writer until it runs out of new memories, hits a hard cap (25 new pages per job, 100 section rewrites per job), or Bedrock rate-limits it. Then the job finishes, records its metrics, and exits.

If the cursor still isn't at the end, a subsequent job picks up where this one left off. Large backfills naturally spread across several jobs.

## What can go wrong

**Bedrock throttling.** The planner and section writer both call Claude through Bedrock. Under load, Bedrock returns "too many requests" errors. The job fails, but because the cursor only advanced on successful applies, no memories are lost — the next invoke picks up where the throttle hit.

**JSON truncation.** If the planner produces more output than its token budget allows, the JSON comes back malformed. Parsing fails, the job fails, the cursor holds. The mitigation is a large `maxTokens` budget (currently 24,000) — enough for dense batches without being wasteful.

**Runaway new pages.** A hallucinating planner could try to create dozens of spurious pages in one batch. A hard cap of 25 new pages per job stops that cold. The job exits with a `cap_hit` flag in its metrics; the next job continues.

**Cross-scope writes.** Before writing, the worker double-checks every page update references a page in the job's `(tenant, owner)` scope. If the planner hallucinates a foreign page id, the worker silently skips it rather than corrupting another agent's wiki.

**Cursor precision drift.** Memories are timestamped with microsecond precision in Postgres, but JavaScript `Date` only carries milliseconds. The cursor query truncates the database side to milliseconds so a memory at `xx:yy:zz.689019` doesn't appear to be "after" a stored cursor of `xx:yy:zz.689` forever. Without this, the drain loops on the tail record indefinitely.

## Cost

One full compile of an agent with a few hundred memory units runs around $0.50 to $1.00 in Bedrock costs. That's end-to-end — planner + section writer, all batches. Per-batch cost is typically $0.04 to $0.12 depending on how much the planner has to say.

The metrics JSON on each compile job row records `input_tokens`, `output_tokens`, and a rough `cost_usd` estimate. Watch that number when you're tuning prompts.

## Under the hood

### Triggers

- **Post-retain** — `maybeEnqueuePostTurnCompile` in `packages/api/src/lib/wiki/enqueue.ts`. Checks `tenants.wiki_compile_enabled`, confirms the adapter is Hindsight, inserts a job row with dedupe key `${tenantId}:${ownerId}:${floor(epoch_s/300)}`, fires `Lambda.invoke` with `InvocationType: "Event"`.
- **Admin mutation** — `compileWikiNow(tenantId, ownerId)` resolver.
- **Nightly lint sweep** — `packages/api/src/handlers/wiki-lint.ts` enqueues promotion jobs for mentions that cross the threshold.

### Provenance rule

```ts
// packages/api/src/lib/wiki/compiler.ts
function resolveCitedRecords(
  byId: Map<string, ThinkWorkMemoryRecord>,
  ids: string[] | undefined,
): ThinkWorkMemoryRecord[] {
  if (!ids || ids.length === 0) return [];
  const out: ThinkWorkMemoryRecord[] = [];
  for (const id of ids) {
    const r = byId.get(id);
    if (r) out.push(r);
  }
  return out;
}
```

No fallback to batch-wide sources. This function is the single most important correctness invariant in the pipeline.
