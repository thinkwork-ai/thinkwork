External task connectors turn work items owned by an external system — LastMile Tasks today, other task systems next — into native ThinkWork threads. When a user is assigned work in the external system, ThinkWork opens a dedicated thread for it. Inside the thread, the task renders as a mobile-first GenUI card built from a bounded block grammar. Users can review, update, and discuss the task in ThinkWork while the external system remains the source of truth.

This is a deliberate product position: **ThinkWork owns the work experience; the external provider owns the work record.**

<Aside type="note">
For the broader connector model, start with [Connectors Overview](/concepts/connectors/). For MCP tool servers that agents call during a turn, see [MCP Tools](/concepts/connectors/mcp-tools/).
</Aside>

## How it fits

ThinkWork is not a task system. It does not store statuses, workflow IDs, audit history, or assignments as canonical state. The external provider is authoritative for all of that. ThinkWork stores the linkage, the rendered projection, and the thread that carries the conversation.

That split drives the whole architecture:

- **Threads** remain the unit of work — an external task becomes a thread with `channel = TASK`.
- **Agents** can still participate in the thread, chat with the user, and call tools.
- **A bounded block grammar** (not free-form markdown) drives every provider's task card. One mobile renderer, many providers.
- **An adapter seam** isolates provider-specific logic (event normalization, write tool mapping, OAuth) behind a single TypeScript interface.

LastMile Tasks is the first and only adapter in production. Linear is the committed next target. The adapter seam is provider-agnostic by design.

## What the user sees

> _Screenshot placeholder: external task detail view — pinned task card above the chat timeline._

External tasks live in the same inbox as chat threads. Opening one routes to a dedicated detail screen that composes a **pinned task header** above the **chat timeline**, so the user can act on the task and talk about it in the same place.

The pinned header renders an `ExternalTaskCard`. The card is built entirely from blocks — there is no markdown, no iframe, and no per-provider UI. The current block types are:

| Block | Renders |
|---|---|
| `task_header` | Title, description, provider badge, optional updated timestamp, edit pencil (when the task advertises an edit capability) |
| `field_list` | Labelled field rows — status, priority, assignee, due date, and any provider-specific fields |
| `badge_row` | Inline pills for tag-like fields |
| `activity_list` | Webhook-driven audit rows ("Status changed to In Progress", "Assigned to Eric"), newest first |
| `action_bar` | A row of primary actions (update status, assign, edit fields) |
| `form` | Bottom-sheet edit form built from the provider's field schema |
| `section` | A titled group of nested blocks |
| `empty_state` | Fallback when a block has no data |

Buttons and the pencil icon open bottom-sheet modals for status changes, assignments, and field edits. On submit, the change is routed through the adapter seam (see [Save-edit round-trip](#save-edit-round-trip)) and the card re-renders with the new envelope.

### Activity timeline vs chat timeline

<Aside type="note">
The task card's **Activity** section and the **chat timeline** below it are two different feeds. The activity section shows webhook-driven audit rows. The chat timeline shows user and assistant messages only. Audit rows are deliberately kept off the chat transcript so the conversation stays clean.
</Aside>

This is a design decision worth calling out because it surprises people. Both feeds live on the same thread, and both are derived client-side from the same `messages` table, but they filter on different criteria:

- **Chat timeline** — messages with `role ∈ {user, assistant}`.
- **Activity section** — messages with `role = system` and `metadata.kind = "external_task_event"`.

If you need to render something durable in the chat transcript from an external system, write a system message with a different `metadata.kind` so the activity filter ignores it.

### Realtime updates

When an external system fires a webhook, the activity row appears on the card within about two seconds. That path uses AppSync subscriptions; there is no pull-to-refresh. Push notifications fire on top of that for a narrow set of events (see [Push notification policy](#push-notification-policy)).

## The envelope and the block grammar

Every external task is ultimately rendered from a single shape, the `ExternalTaskEnvelope`:

```ts
type ExternalTaskEnvelope = {
  _type: "external_task";
  _source?: { provider: string; tool: string; params: unknown };
  item: NormalizedTask;
  blocks: TaskBlock[];
  _refreshedAt?: string;
};
```

The envelope is the contract between the server and the mobile card. Everything the card needs to render is inside it: the normalized task fields, the blocks to lay out, and an optional refresh timestamp. The mobile renderer does not care which provider produced it.

`NormalizedTask` carries the task's core fields (id, title, description, status, priority, assignee, due date, URL, updated-at), a `capabilities` object that tells the UI which actions to show (`updateStatus`, `assignTask`, `commentOnTask`, `editTaskFields`, …), a list of `fields` with curated options for selects, a list of `actions`, and an optional `forms.edit` / `forms.comment` form schema. Provider-specific extras live under `extensions`; the raw provider payload is preserved under `raw`.

The block types in the table above are the complete grammar — there is no "render arbitrary HTML" escape hatch. Adding a new provider does **not** mean adding new blocks; it means returning the existing blocks with different content.

## The adapter seam

Every provider implements a single TypeScript interface, `ExternalWorkItemAdapter`:

```ts
interface ExternalWorkItemAdapter {
  readonly provider: TaskProvider;

  verifySignature(req): Promise<boolean>;
  normalizeEvent(rawBody): Promise<NormalizedEvent>;
  normalizeItem(raw): NormalizedTask;
  buildFormSchema(item): TaskFormSchema;
  buildBlocks(item): TaskBlock[];

  executeAction(args): Promise<ExternalTaskEnvelope>;
  refresh(args): Promise<ExternalTaskEnvelope>;
}
```

Eight methods. Three responsibilities:

1. **Inbound translation** — `verifySignature`, `normalizeEvent`, `normalizeItem` turn a provider webhook into a normalized event and a normalized task.
2. **Presentation** — `buildFormSchema` and `buildBlocks` produce the form and the block list the UI renders.
3. **Outbound actions** — `executeAction` performs writes through the provider's API; `refresh` re-fetches live state.

The rest of the pipeline — connection resolution, OAuth token refresh, thread upsert, realtime broadcast, push notifications — is adapter-neutral and runs the same for every provider.

```
Provider API
    → adapter.refresh()
    → NormalizedTask
    → blocks + envelope
    → mobile ExternalTaskCard
```

## Webhook ingestion

The webhook pipeline is the heart of the integration. When the external system fires an event, ThinkWork turns it into a thread update, a system message, a realtime broadcast, and (sometimes) a push notification.

```
External system (LastMile, Linear, etc.)
    → POST /integrations/<provider>/webhook (API Gateway)
    → Webhooks Lambda
        └─ Resolves webhook by token (target_type=task, provider id)
        └─ adapter.verifySignature (opt-in via webhook config secret)
        └─ adapter.normalizeEvent → NormalizedEvent
        └─ Resolves the connection via providerUserId
        └─ Resolves the per-user MCP token
        └─ adapter.refresh() → live envelope
        │      (synthetic envelope fallback if refresh fails)
        └─ Upserts the external-task thread (channel=TASK)
        └─ Writes a system message (metadata.kind="external_task_event")
        └─ Awaits [notifyNewMessage, notifyThreadUpdate, sendExternalTaskPush]
    → Mobile task card refreshes via AppSync subscription (~2s)
    → Push notification fires for assignment / status / due changes only
```

A few things are worth calling out because they are easy to get wrong:

- **The refresh has a synthetic fallback.** If `adapter.refresh()` fails — provider outage, transient MCP error, expired token that could not be refreshed — the pipeline still produces a thread update by building an envelope directly from the webhook payload. Users never see an empty task because an upstream call blipped.
- **Reassignments close the prior thread.** When a task is reassigned away from a user, their thread is closed and a new one opens for the new assignee. The thread upsert is idempotent on `(provider, externalTaskId, assigneeUserId)`.
- **The notify calls are awaited, not fire-and-forget.** All three realtime / push calls run through `Promise.allSettled` and the ingestion handler waits for them to settle before returning. This is intentional: the Lambda environment cannot safely let network work escape the handler lifetime, and a missed realtime broadcast means a stale card.

## Push notification policy

Not every webhook fires a push. The policy is narrow on purpose — the goal is to notify on things that actually change the user's plan, not on every audit event.

Push fires on:

- `task.assigned` / `task.reassigned` — you got a new task (or lost one).
- `task.status_changed` — someone moved your task to a new status.
- `task.updated` — but **only** when the updated properties include `status`, `status_id`, `due_at`, or `due_date`.

Push does **not** fire on description edits, title edits, priority changes, or updated_at-only diffs. The activity list still records everything; only push is filtered.

## MCP OAuth and token lifecycle

External task adapters authenticate to the provider through per-user OAuth tokens. The lifecycle is managed end-to-end and users do not see it after the initial connect:

- Tokens are stored in **Secrets Manager** with a pointer + `expires_at` in the `user_mcp_tokens` table (see [MCP Tool Connectors — Data model](/concepts/connectors/mcp-tools/#data-model)).
- Before any outbound call, the resolver checks `expires_at` against a **five-minute buffer**. If the token is within that window, it is refreshed before being handed to the adapter.
- Refresh exchanges the stored refresh token at the provider's token endpoint, writes the **rotated** refresh token back to Secrets Manager, updates `user_mcp_tokens.expires_at`, and only then returns the new access token to the caller.
- If refresh fails, the token row is marked expired and the user sees a "Reconnect" prompt in the mobile app on their next interaction.

Users never interact with this directly — it is automatic.

## Save-edit round-trip

Every write the user can perform goes through the adapter seam, not through a custom mutation per provider.

1. The mobile card submits an `executeExternalTaskAction` GraphQL mutation with an action type (`update_status`, `assign`, `comment`, `edit_fields`) and a `params` payload.
2. The resolver dispatches to the adapter's `executeAction`, which translates the action into whatever the provider's API needs.
3. When `executeAction` returns, the adapter immediately calls `refresh()` so the envelope reflects the post-write state.
4. The fresh envelope is written into `threads.metadata.external.latestEnvelope` and published over the same AppSync channels the webhook pipeline uses.

Two details matter for providers connected via MCP:

- **All write tools must key off the normalized task id.** Every write action the adapter issues takes the task id as its primary argument, not an internal row id. Mixing these up silently fails — the server accepts the write, the row never updates, and the user sees no error.
- **MCP tool errors are unwrapped into thrown exceptions.** Some providers return tool-level errors as a `content` array with `isError: true` instead of a transport-level error. The MCP client used by the adapter reads that flag and throws with the content text, so a failing write surfaces in the UI instead of returning silent garbage.

## Current limits

This feature is in Beta. The following are known limits today, not hypothetical concerns:

- **Status and priority form edits fail upstream for LastMile.** The edit form's status and priority selects use curated human-readable values (`todo`, `in_progress`, …), but the provider's status-update tool expects opaque workflow IDs (`status_hfcq…`). The error surfaces clearly, but users cannot actually change status from the edit form yet. Status changes through the dedicated `update_status` action still work. Planned fix: source the real workflow IDs from provider metadata so the form can present them.
- **Assignees other than the signed-in user render as raw IDs.** The provider only returns the full assignee object (name, email) for the caller's own tasks. For tasks assigned to other users, the card shows the provider's internal user id. Fixing this requires a provider user-lookup tool, which the current adapter does not have.
- **Comments are hidden by default.** The reference adapter's backing system has no comment tool, so `capabilities.commentOnTask` is `false` and the Comment action is hidden from the card. A `comment` action is defined in the block grammar for future adapters.
- **Non-canonical priority labels render lowercase.** The curated priority set is `urgent / high / normal / low`. Values outside that set (e.g. `medium`) render as-is. Cosmetic fix coming.
- **LastMile is currently the only adapter.** The adapter seam is generic, but only one reference implementation is shipped. Linear is the committed next target.

## Adding a new provider

The adapter seam is designed so a new provider is a single TypeScript file plus some configuration. The contract:

1. **Implement the eight `ExternalWorkItemAdapter` methods.** The seam is provider-neutral; everything provider-specific lives inside these methods. See `packages/api/src/integrations/external-work-items/types.ts` for the full interface and `packages/api/src/integrations/external-work-items/providers/lastmile/` for a reference implementation.
2. **Register the provider** in the adapter registry (`packages/api/src/integrations/external-work-items/index.ts`), so `getAdapter(provider)` and `hasAdapter(provider)` resolve it.
3. **Seed a `connect_providers` row** for the provider. This is the tenant-level registration that webhooks resolve against.
4. **Configure webhook delivery and a signing secret** in the provider's dashboard. Point it at `/integrations/<provider>/webhook` and store the secret on the webhook row's `config.secret`.
5. **No mobile changes.** The mobile card renders whatever `buildBlocks()` returns through the existing block grammar. This is the payoff of the bounded grammar: adding a provider does not mean shipping a mobile release.
