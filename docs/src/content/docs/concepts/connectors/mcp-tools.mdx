MCP (Model Context Protocol) is the harness's connector path for external tool systems — the tool/capability half of the harness's I/O surface, peer to [Integrations](/concepts/connectors/integrations/) which handles app and channel events. MCP lives under **Connectors**, not as a separate platform pillar, because the same trust boundary applies to both: the **Security** operating guarantee gates which MCP tools an agent's template allows it to call (capability grants), routes the call through a tenant-scoped MCP client (no shared sessions), and traces every MCP invocation in the same turn record as every other tool call.

Use MCP tools when agents need to call remote capabilities during work, such as CRM actions, ticket updates, internal API operations, search, or database queries exposed by an MCP server.

<Aside type="note">
For the broader connector model, start with [Connectors Overview](/concepts/connectors/). For app and channel integrations like Slack, GitHub, and Google Workspace, see [Integrations](/concepts/connectors/integrations/).
</Aside>

## How MCP tools fit

ThinkWork uses two broad connector paths:

1. **Integrations** bring external messages and events into threads
2. **MCP tools** give agents external capabilities they can call during a turn

Both still operate inside the same ThinkWork model:

- threads remain the unit of work
- agents decide what to do
- controls govern access and usage
- audit trails capture tool activity

## How it works

```text
Admin registers MCP server
    → Assigns it to agent template or agent
    → Agent invocation resolves auth
    → Agent container connects to MCP server
    → Tools are discovered
    → Agent calls tools during the turn
    → Calls are logged in the thread timeline
```

## Registration

MCP servers are registered at the tenant level, then assigned to agent templates or agents.

Typical config includes:

| Field | Description |
|---|---|
| Name | Display name for the MCP server |
| URL | HTTP endpoint for the server |
| Transport | Streamable HTTP or SSE |
| Auth | None, tenant API key, or per-user OAuth |

In the admin experience, this should appear under **Connectors → MCP Tools**.

### URL is read from the record on every call

The MCP endpoint URL lives in `tenant_mcp_servers.url`. The runtime resolves it on every outbound call — there are **no hardcoded defaults** and **no env-var fallbacks** for MCP base URLs. Rename the record in admin and the next request uses the new host; no code deploy required.

This matters in two concrete cases:

- The provider renames their host (e.g. `mcp-dev.example.com` → `dev-mcp.example.com`). You fix it in admin, not in code.
- A provider serves two stacks at similar-looking hostnames (e.g. a legacy build alongside the current one). Whichever URL you register is the one that's hit — the code has no opinion.

If a provider-side rename breaks a connection, update the record first and verify with a `tools/list` curl before suspecting deeper bugs.

## Authentication

MCP tool connectors commonly use one of three patterns:

### None

Useful for private-network services or systems with external network controls.

### Tenant API key

A shared secret is stored securely and resolved at invocation time for any agent allowed to use that MCP server.

### Per-user OAuth

Useful when downstream tools need user-scoped access. ThinkWork resolves the paired user's token before opening the MCP connection.

<Aside type="note">
If a per-user OAuth token is missing, ThinkWork can skip that MCP server for the invocation while still allowing the rest of the agent run to proceed.
</Aside>

## Runtime behavior

At runtime, ThinkWork discovers tools from the MCP server and adds them beside built-in tools and skill-provided tools. If an MCP server is unavailable, the agent can continue with its remaining toolset.

MCP usage is tracked in the thread timeline, including:

- tool name
- MCP server attribution
- input and output previews
- execution status

## Supported use cases

Common MCP tool connector use cases include:

- CRM lookups and updates
- internal admin APIs
- task and ticket actions
- business-system queries
- search or retrieval over external systems

## "MCP server" is a two-surface term — be precise

Internal conversations occasionally conflate two things with the same name:

- **A provider's MCP server** — the remote HTTP(S) endpoint you register here. That's what this page is about.
- **A provider's task-connector surface** (e.g. an older LastMile integration) — a different surface that routes tasks into ThinkWork threads. Not the same thing as an MCP server, even when both belong to the same provider.

When someone says "LastMile MCP," they usually mean the remote MCP server we register here. The task-connector surface is documented separately under external-task integrations. Be explicit when the two come up together.
