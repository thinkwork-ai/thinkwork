MCP (Model Context Protocol) is ThinkWork's connector path for external tool systems — the tool/capability half of the I/O surface, peer to [Integrations](/concepts/connectors/integrations/) and tracker connectors. MCP lives under **Connectors**, not as a separate platform pillar, because the same trust boundary applies: ThinkWork gates which tools a Computer or managed Agent may call, routes the call through tenant-scoped credentials, and traces every MCP invocation in the same turn record as other tool calls.

Use MCP tools when a Computer or delegated managed Agent needs to call remote capabilities during work, such as CRM actions, ticket updates, internal API operations, search, or database queries exposed by an MCP server.

## How MCP tools fit

ThinkWork uses three broad connector paths:

1. **Tracker connectors** reconcile external work items into Computer-owned work.
2. **Integrations** bring external messages and events into threads.
3. **MCP tools** give Computers and managed Agents external capabilities they can call during a turn.

Both still operate inside the same ThinkWork model:

- threads remain the visible unit of work
- Computers own continuity and may delegate to managed Agents
- controls govern access and usage
- audit trails capture tool activity

## How it works

```text
Admin registers MCP server
    → Assigns it to a template, Computer, or managed Agent
    → Invocation resolves tenant or user auth
    → Runtime connects to MCP server
    → Tools are discovered
    → Computer or managed Agent calls tools during the turn
    → Calls are logged in the thread timeline
```

## Registration

MCP servers are registered at the tenant level, then assigned to templates, Computers, or managed Agents depending on the product surface.

Typical config includes:

| Field     | Description                             |
| --------- | --------------------------------------- |
| Name      | Display name for the MCP server         |
| URL       | HTTP endpoint for the server            |
| Transport | Streamable HTTP or SSE                  |
| Auth      | None, tenant API key, or per-user OAuth |

In the admin experience, this should appear under **Connectors → MCP Tools**.

### URL is read from the record on every call

The MCP endpoint URL lives in `tenant_mcp_servers.url`. The runtime resolves it on every outbound call — there are **no hardcoded defaults** and **no env-var fallbacks** for MCP base URLs. Rename the record in admin and the next request uses the new host; no code deploy required.

This matters in two concrete cases:

- The provider renames their host (e.g. `mcp-dev.example.com` → `dev-mcp.example.com`). You fix it in admin, not in code.
- A provider serves two stacks at similar-looking hostnames (e.g. a legacy build alongside the current one). Whichever URL you register is the one that's hit — the code has no opinion.

If a provider-side rename breaks a connection, update the record first and verify with a `tools/list` curl before suspecting deeper bugs.

## Authentication

MCP tool connectors commonly use one of three patterns:

### None

Useful for private-network services or systems with external network controls.

### Tenant API key

A shared secret is stored securely and resolved at invocation time for any agent allowed to use that MCP server.

### Per-user OAuth

Useful when downstream tools need user-scoped access. ThinkWork resolves the paired user's token before opening the MCP connection.

<Aside type="note">
  If a per-user OAuth token is missing, ThinkWork can skip that MCP server for
  the invocation while still allowing the rest of the Computer or managed-Agent
  run to proceed.
</Aside>

## Runtime behavior

At runtime, ThinkWork discovers tools from the MCP server and adds them beside built-in tools and skill-provided tools. If an MCP server is unavailable, the worker can continue with its remaining toolset.

MCP usage is tracked in the thread timeline, including:

- tool name
- MCP server attribution
- input and output previews
- execution status

## Supported use cases

Common MCP tool connector use cases include:

- CRM lookups and updates
- internal admin APIs
- task and ticket actions
- business-system queries
- search or retrieval over external systems

## "MCP server" is a two-surface term — be precise

Internal conversations occasionally conflate two things with the same name:

- **A provider's MCP server** — the remote HTTP(S) endpoint you register here. That's what this page is about.
- **A provider's task-connector surface** — a different surface that routes tasks into ThinkWork threads. Not the same thing as an MCP server, even when both belong to the same provider.

When an integration has both an MCP server and a tracker connector surface, name the surface explicitly. MCP tools are called during work; tracker connectors create or update work.
