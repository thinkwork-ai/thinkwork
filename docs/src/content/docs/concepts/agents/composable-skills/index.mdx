Most of what agents do well is conversational — open-ended turns where the agent reasons, picks a tool, replies, and iterates. But a large slice of real work is the opposite: a known sequence of steps that needs to run the same way every single time, with the same inputs producing the same outputs, a hard budget ceiling, and an audit row you can point at. Prepping a sales rep for their 2pm meeting. Reconciling a customer-onboarding checklist after a deal closes. Generating the weekly renewal-risk report. Those aren't conversations. They're workflows.

A **composable skill** is ThinkWork's primitive for that second category. It's a YAML file that declares steps, each step references a smaller sub-skill, and the runtime executes them to completion — sequentially, in parallel, or both — writing a single audit row to `skill_runs` that captures inputs, outputs, status, and timing.

## The core idea

A composition is a directed graph of sub-skill invocations. Each step produces a named output that downstream steps can reference. Steps can be sequential (one after another) or parallel (fan out, wait for all, continue). Each sub-skill is itself a skill — a script that runs a deterministic Python function, a context-mode skill that calls an LLM with a prompt template, or another composition (turtles all the way down). The composition author declares *what* runs; the runtime handles *how* — dispatch, retries, budgets, audit.

A representative composition — `sales-prep`, the example used throughout these docs — looks like this:

```yaml
id: sales-prep
version: 1
execution: composition
steps:
  - id: frame
    mode: sequential
    skill: frame
    inputs:
      problem: "Prep for meeting with {customer} on {meeting_date}. Focus: {focus}."

  - id: gather
    mode: parallel
    on_branch_failure: continue_with_footer
    branches:
      - id: crm
        skill: crm_account_summary
        critical: true
      - id: ar
        skill: ar_summary
      - id: tickets
        skill: support_incidents_summary
      - id: web
        skill: web_research
      - id: wiki
        skill: wiki_search

  - id: synthesize
    skill: synthesize
    inputs:
      framed: "{framed}"
      gathered: "{gathered}"

  - id: package
    skill: package
    inputs:
      synthesis: "{synthesis}"
      format: sales_brief
    output: deliverable

budget_cap:
  tokens: 120000

delivery:
  - chat
  - email
```

Five reusable sub-skills combined into one workflow. Write it once; trigger it from chat ("prep me for Acme"), from the admin catalog ("Run now"), from a scheduled job ("weekdays at 2pm for tomorrow's meetings"), or from a webhook (CRM opportunity stage change). Same YAML, four surfaces.

## Why this exists — when compositions beat the alternatives

<Aside type="tip">
  The calibration question: "Would I want this workflow to run identically every time, even if the agent thinks it knows better?" If yes, it's a composition. If the agent's judgment is the value, it's a chat turn.
</Aside>

Three alternatives exist in the stack; each is right for something and wrong for something else.

| Approach | Right for | Wrong for |
|----------|-----------|-----------|
| **Agent chat turn** | Open-ended conversation, novel reasoning, tool selection based on unpredictable input | Repeatable multi-step work where skipping a step has real cost |
| **Monolithic skill script** | A single-function capability that doesn't branch or fan out | Anything with parallel fetches, multi-step synthesis, or reusable pieces |
| **Composable skill** | Structured multi-step work that recurs, where each step has a stable contract | True one-off scripts; open-ended reasoning where the agent should decide next action |

The wins compositions deliver that neither of the other two can:

**Deterministic shape, intelligent content.** A composition guarantees the rep gets `frame → gather → synthesize → package`, in that order, with parallel fan-out inside `gather`. The agent can't skip `gather` because it "looked unnecessary"; `synthesize` can't fire before gather completes; the whole run has a per-invocation budget ceiling. But each step is still a smart LLM call (or a deterministic script) — so the *content* of each step benefits from model reasoning while the *shape* of the workflow is fixed.

**Reusable primitives.** `frame`, `gather`, `synthesize`, `package` appear in `sales-prep`, `renewal-prep`, `account-health-review`, and `customer-onboarding-reconciler`. The second composition is 30 lines of YAML referencing skills that already exist, not 300 lines of Python re-implementing fan-out logic.

**One composition, four trigger surfaces.** The same YAML runs from chat dispatch, admin catalog "Run now", scheduled cron, and webhook ingress. Authors don't write four versions of the same workflow. Operators get consistent behavior regardless of how the run was initiated.

**Parallel fan-out that actually fans out.** `gather` pulls CRM, AR, support tickets, external web, and wiki *concurrently*. An agent loop calling five tools would do them serially. A composition explicitly models the parallelism — faster wall-clock, same model cost.

**Budget ceilings, dedup, and audit by design.** Every run has a hard `budget_cap.tokens` — exceed it and the composition fails deterministically rather than running up your Bedrock bill. Every run is deduplicated via a partial unique index on `(tenant_id, invoker_user_id, skill_id, resolved_inputs_hash) WHERE status='running'` — fire "prep me for Acme" twice in 10 seconds and the second call returns the first run's id, not a duplicate. Every run lands in `skill_runs` with inputs, outputs, timing, and a `failure_reason` if it terminated abnormally — the admin's `Analytics → Runs` surface renders all of it.

**AI-editable.** Compositions are YAML + markdown. An agent can read `sales-prep`, critique it, and propose a diff that a human reviews and merges. Try that with a Python workflow — the agent can *read* the code but can't safely modify it without running a full review cycle.

## When a composition is *not* the right tool

Compositions come with real costs. Skip them when:

- **The work is genuinely one-turn.** The user asks a question, the agent looks up one thing, the agent replies. That's a chat turn with a tool, not a composition. Don't introduce a `skill_runs` row for every Slack DM.
- **The shape changes every time.** If each invocation needs fundamentally different steps based on the user's request, you're describing an agent doing reasoning, not a workflow. Let the agent do its job.
- **You need a human approval between steps.** A composition runs to terminal status without pausing. If step 2 should wait for a human to click Approve before step 3 fires, that's a [Routine](/concepts/automations/routines-and-execution-model/), which supports approval checkpoints; compositions don't.
- **The primitives don't exist yet.** A composition is only valuable if the sub-skills it composes already exist (or will, soon). A composition with zero reusable primitives is a worse version of a monolithic script.

## Known limits

- **No human-in-the-loop checkpoints.** Compositions run to terminal status without pausing. For approval gates, use [Routines](/concepts/automations/routines-and-execution-model/).
- **Single-Lambda execution window.** A composition runs inside the agentcore Lambda's 15-minute cap. Work that genuinely takes longer needs a background-worker split.
- **Context-mode sub-skills are not yet wired in the run_skill dispatch path.** Script-mode sub-skills execute today; LLM-prompt sub-skills raise `SkillNotRegisteredError` and fail the step cleanly. This is tracked for a follow-up.
- **No built-in retry of failed steps.** If `gather`'s `crm` branch fails and it's marked `critical: true`, the composition fails. There's no exponential-backoff retry inside the runner — compositions lean on the calling client to re-trigger.

## Real scenarios this unlocks

- **Sales prep for every meeting, proactively.** A scheduled composition runs at 2pm every weekday with `meeting_date: today_plus_N: 1` and `customer: {from_tenant_config: default_customer}`. The rep gets a 60-second brief in their inbox before tomorrow's meetings. They don't remember to ask; it just arrives.
- **Customer onboarding that starts when the deal closes.** CRM fires a close-won webhook. The webhook handler dispatches `customer-onboarding-reconciler`, which creates tasks, schedules a kickoff, notifies the CSM, and writes an entry to the customer-record wiki. Nobody pressed a button.
- **Weekly renewal risk.** Every Monday, a composition scans every account ≤90 days from renewal, synthesizes signals from CRM + usage + support history, and delivers a ranked list to the account team. Same composition every week, same format, different data.
- **Same-skill, different trigger.** A customer-success manager can say "give me the health report for Northwind" in chat, or click "Run now" on the agent's skill list in the admin, or the same report runs on a monthly schedule. One composition — `account-health-review` — serves all three.

## How compositions fit in the stack

```
      ┌─ chat dispatch ──────┐
      ├─ admin catalog ──────┤     ┌────────────────────┐
      ├─ scheduled cron ─────┼────▶│  startSkillRun     │
      └─ webhook ingress ────┘     │  (service endpoint │
                                   │   or GraphQL)      │
                                   └──────────┬─────────┘
                                              │  inserts skill_runs row (status=running)
                                              ▼
                                   ┌────────────────────┐
                                   │ agentcore Lambda   │
                                   │ container          │
                                   │                    │
                                   │ loads composition  │
                                   │ YAML from S3,      │
                                   │ runs composition_  │
                                   │ runner with a      │
                                   │ SkillDispatch      │
                                   │ closure            │
                                   └──────────┬─────────┘
                                              │  dispatches each step's sub-skill
                                              ▼
                                   ┌────────────────────┐
                                   │  sub-skill executes│
                                   │  (script | context │
                                   │   | agent | nested │
                                   │   composition)     │
                                   └──────────┬─────────┘
                                              │  returns output
                                              ▼
                                   ┌────────────────────┐
                                   │ composition aggregates
                                   │ step outputs → final
                                   │ CompositionResult  │
                                   └──────────┬─────────┘
                                              │  POST /api/skills/complete
                                              ▼
                                   skill_runs.status = complete | failed
                                   + failure_reason + delivered_artifact_ref
```

Four entry points, one composition definition, one audit row. The agent runtime (`agentcore`) is the execution layer; the composition YAML is the declarative layer; `skill_runs` is the observability layer. Each is replaceable without rewriting the others.
