This page assumes you've read the [Composable Skills overview](/concepts/agents/composable-skills/) and want to write one. It walks through `sales-prep` — the canonical example — field by field, then covers the patterns for adding your own.

## The skill directory

A composition lives in the same catalog as every other skill: one directory per skill, with a `skill.yaml` defining the shape and a `SKILL.md` documenting the contract. Compositions don't have a `scripts/` directory (they delegate execution to sub-skills); they can have a `prompts/` directory if a single step needs one.

```
packages/skill-catalog/sales-prep/
├── skill.yaml        # The composition definition — this page's focus
├── SKILL.md          # Human-facing contract: what it does, who invokes it
└── README.md         # Optional — authoring notes
```

The directory name is the skill's identifier. `packages/skill-catalog/sales-prep/` produces a skill with id `sales-prep` and an S3 prefix `skills/catalog/sales-prep/`.

## Top-level metadata

```yaml
id: sales-prep
version: 1
execution: composition
mode: tool
name: "Prep for Meeting"
description: >
  Gather account context, financials, activity, and external signals for a
  sales meeting. Produces a packaged brief the rep can read in 60 seconds.
```

- **`id`** — must match the directory name. The runtime's composition loader keys on this.
- **`version`** — integer, used by the runtime and skill catalog for compatibility. Bump when the composition's contract changes (different inputs, different delivery, different step shape).
- **`execution: composition`** — the discriminator. Three legal values: `composition` (this page), `script` (a Python function), `context` (an LLM-prompt skill). Compositions always declare `execution: composition`.
- **`mode: tool`** — how the skill presents to an agent. `tool` means it's a single invocable capability. `agent` means it runs in a sub-agent context — uncommon for compositions.
- **`name`** — human-facing display name. Shown in the admin catalog.
- **`description`** — one or two sentences. Shown in the admin catalog and used by the chat dispatcher when matching user intent to a skill.

<Aside type="caution">
  The top-level schema is strict: `extra="forbid"`. Any field the composition loader doesn't recognize fails pydantic validation, and the skill won't load. If you add a custom field like `invocable_from: composition`, the composition runner will reject the YAML at load time with a single-line validation error.
</Aside>

## Typed inputs

```yaml
inputs:
  customer:
    type: string
    required: true
    resolver: resolve_customer
    on_missing_input: ask
  meeting_date:
    type: date
    required: true
    on_missing_input: ask
  focus:
    type: enum
    values: [financial, expansion, risks, general]
    default: general
```

Inputs are named parameters the composition accepts. The runtime validates values against these specs before `composition_runner` ever sees them.

- **`type`** — `string`, `date`, `enum`, or `int`. The admin's catalog UI uses this to render the right input widget.
- **`required`** — if `true` and no value is supplied, dispatch either asks the user (when the caller is chat and `on_missing_input: ask`) or fails the run with a clear reason.
- **`resolver`** — an optional resolver-skill id. For `customer`, the dispatcher may hand the raw user input ("prep me for Acme") to the `resolve_customer` skill, which turns "Acme" into a resolved customer id. Resolution happens *before* the composition runs; `composition_runner` sees the resolved value.
- **`on_missing_input`** — `ask` (prompt the user), `default` (use the declared default), or `fail` (terminate the run).
- **`values`** — for `type: enum` only; constrains legal values.
- **`default`** — used when `on_missing_input: default`. Can be overridden per-tenant (see next section).

## Tenant overrides

```yaml
tenant_overridable:
  - inputs.focus.default
  - triggers.schedule.expression
```

The `tenant_overridable` list names the YAML paths each tenant can customize without touching the canonical composition. A tenant that wants their team's default focus to be `risks` rather than `general` adds an override to `tenant_settings.features` — no new composition file, no redeploy.

<Aside type="tip">
  Pick override surfaces deliberately. Every field you mark `tenant_overridable` becomes part of the skill's contract and has to keep working across versions. Prefer overriding defaults and schedules; resist overriding step structure.
</Aside>

## Triggers

```yaml
triggers:
  chat_intent:
    examples:
      - "prep me for {customer}"
      - "brief me on {customer}"
      - "sales prep for {customer}"
    disambiguation: ask
  schedule:
    type: cron
    expression: "0 14 ? * MON-FRI *"
    bindings:
      customer:
        from_tenant_config: default_customer
      meeting_date:
        today_plus_N: 1
```

Triggers declare how the composition is invoked. A composition can be triggered from multiple sources without writing four versions of the workflow.

- **`chat_intent.examples`** — phrases the chat dispatcher should match to this composition. `{customer}` marks an input placeholder — the dispatcher pulls "Acme" out of "brief me on Acme" and passes it as `customer`.
- **`chat_intent.disambiguation`** — what to do when the user's phrase matches multiple compositions. `ask` prompts the user; `highest_confidence` picks the top match; `refuse` declines.
- **`schedule.expression`** — standard cron expression evaluated by EventBridge Scheduler. `rate()` is also supported.
- **`schedule.bindings`** — for scheduled runs, the input values are bindings rather than literals. Three binding shapes are supported:
  - `{from_tenant_config: <key>}` — read from `tenant_settings.features[<key>]`.
  - `{today_plus_N: <n>}` — `today + n days`, formatted as `YYYY-MM-DD`.
  - `{literal: <value>}` — a literal passthrough; rarely needed because plain values work too.

Scheduled runs resolve bindings in the `job-trigger` Lambda before dispatching to the composition. If a required binding can't resolve (e.g. a missing `from_tenant_config` key), the composition logs a `failed` run with `failure_reason = "invalid_binding: …"` and does not invoke `composition_runner`.

## Steps

Steps are the heart of the composition — the ordered list of sub-skill invocations.

### Sequential step

```yaml
steps:
  - id: frame
    mode: sequential
    skill: frame
    inputs:
      problem: "Prep for meeting with {customer} on {meeting_date}. Focus: {focus}."
    output: framed
```

- **`id`** — unique within the composition. Used when a later step references this step's output.
- **`mode: sequential`** — one skill, run synchronously.
- **`skill`** — the sub-skill to invoke. This is another entry in the skill catalog (`frame`, `gather`, `crm_account_summary`, etc.) — or a nested composition id. The composition runner looks up the sub-skill's `execution` mode and dispatches accordingly.
- **`inputs`** — a map of the sub-skill's input names to values. Curly braces (`{customer}`) interpolate from the composition's own resolved inputs (`customer: "Acme"`) or from an earlier step's named output (`{framed}` references the `frame` step's `output: framed`).
- **`output`** — name the step's return value so later steps can reference it. Defaults to the step's `id` if omitted.

### Parallel step

```yaml
  - id: gather
    mode: parallel
    on_branch_failure: continue_with_footer
    branches:
      - id: crm
        skill: crm_account_summary
        inputs:
          customer: "{customer}"
        critical: true
      - id: ar
        skill: ar_summary
        inputs:
          customer: "{customer}"
      - id: tickets
        skill: support_incidents_summary
        inputs:
          customer: "{customer}"
      - id: web
        skill: web_research
        inputs:
          customer: "{customer}"
          date: "{meeting_date}"
      - id: wiki
        skill: wiki_search
        inputs:
          customer: "{customer}"
    output: gathered
```

The `gather` step fans out to five branches concurrently and waits for all to complete before moving on.

- **`mode: parallel`** — every branch runs concurrently. Wall-clock is the slowest branch, not the sum.
- **`branches`** — each branch has the same fields as a sequential step (`id`, `skill`, `inputs`), plus:
- **`critical: true`** — if this branch fails, the whole step fails. Non-critical branches that fail are reported as footer notes attached to the aggregated output.
- **`on_branch_failure`** — step-level policy: `fail` (any branch failure fails the step), or `continue_with_footer` (non-critical failures are noted but don't stop the workflow).

The step's aggregated `output` is a dict keyed by each successful branch's id. Downstream steps can reference it as `{gathered.crm}`, `{gathered.ar}`, etc.

### Subsequent sequential steps

```yaml
  - id: synthesize
    mode: sequential
    skill: synthesize
    inputs:
      framed: "{framed}"
      gathered: "{gathered}"
      focus: "{focus}"
    output: synthesis

  - id: package
    mode: sequential
    skill: package
    inputs:
      synthesis: "{synthesis}"
      format: sales_brief
    output: deliverable
```

The outputs compose: `synthesize` reads `{framed}` and `{gathered}` from the earlier steps; `package` reads `{synthesis}`. The final `output: deliverable` is the composition's top-level return value — the thing delivered to the user.

## Budget caps

```yaml
budget_cap:
  tokens: 120000
```

A single number that the runtime enforces across the whole composition's LLM calls. If the sum of input + output tokens across all sub-skills exceeds the cap, the composition fails with `status=cost_bounded_error`. Deterministic script skills don't count against the cap.

<Aside type="caution">
  `budget_cap.tokens` is a ceiling, not a target. If you're regularly hitting 80%+, you're one busy day away from failures. Pad it by at least 2x expected steady-state usage.
</Aside>

## Delivery channels

```yaml
delivery:
  - chat
  - email
```

Where the final `output: deliverable` should land. Each channel has a wiring:

- **`chat`** — the deliverable is posted as a message to the invoking thread (for chat-initiated runs) or a dedicated thread (for scheduled runs).
- **`email`** — the deliverable is sent via SES to the invoking user's email.
- **`agent_owner`** (future) — delivered to the agent's owner human, regardless of who invoked the run. Used by reconciler-style compositions.

## Writing your own — the checklist

1. **Start with an existing composition as a template.** Copy `sales-prep/skill.yaml` into your new directory; strip what doesn't apply.
2. **Name and version.** Set `id` to match the directory name, `version: 1`, a clear `name` and `description`.
3. **Declare inputs.** List every parameter the composition accepts. Use `required: true` for inputs that have no sensible default.
4. **Pick your sub-skills.** Chain from the [primitives](/concepts/agents/composable-skills/primitives/) — `frame` / `gather` / `synthesize` / `package` handles most shape. If you need a new primitive, write it as a standalone skill first, validate, then reference it.
5. **Set a budget cap.** Start conservative (e.g. 60,000 tokens for a 4-step composition); raise it after observing real runs.
6. **Add triggers.** At minimum, `chat_intent.examples`. Add `schedule.expression` only if proactive runs are part of the design.
7. **Upload and validate.** `aws s3 sync packages/skill-catalog/<your-skill>/ s3://<bucket>/skills/catalog/<your-skill>/` — the container syncs from S3 on dispatch.
8. **Smoke it.** Fire `/api/skills/start` with the skill id and check `skill_runs` transitions to `status=complete` or `status=failed` with a specific reason. `scripts/smoke/complete-smoke.sh` is the template.

## Validating locally before upload

```bash
# From the repo root, in a worktree with the strands container deps installed:
cd packages/agentcore-strands/agent-container
uv run --with pydantic --with PyYAML --no-project python -c "
from skill_inputs import load_composition
comp = load_composition('../../skill-catalog/<your-skill>/skill.yaml')
print('OK:', comp.id, 'v' + str(comp.version), len(comp.steps), 'steps')
"
```

The `skill_inputs.load_composition` function is the same pydantic-validated loader the container uses at dispatch time. If this passes locally, the YAML will parse in production.
