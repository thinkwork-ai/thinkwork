Automations are the harness's scheduled-trigger surface — how ThinkWork runs agent work without a human sending each message by hand. A daily digest, a webhook-triggered triage flow, a weekly cleanup routine — they all fit the same shape: ThinkWork starts work on a schedule or in response to an event, creates an `AUTO-` thread to hold the run, invokes the configured agent, and records what happened. Automations are how the [PPAF loop](/architecture/#the-ppaf-agent-loop) starts from outside the conversation; everything downstream of that — Perception, Planning, Action, Feedback — is identical to a chat-initiated turn. Reliability lives here: AWS Scheduler + Step Functions provide fault recovery, idempotent retries, and a checkpointed run history every operator can audit.

The important design choice: an automation *run* and a chat *conversation* are the same kind of thing from the system's point of view. Both are threads. Both have the same audit surface, the same cost accounting, the same memory access. That's why automations don't need their own parallel universe of dashboards, logs, and debuggers — they reuse everything the chat path already has.

## Why this shape matters

The alternative — a separate "jobs" or "workflows" system parallel to the chat system — is how most agent platforms start. It's a trap. Every cross-cutting concern ends up with two implementations:

- **Audit** works for chat but has a different (thinner) view for jobs.
- **Cost tracking** either skips scheduled runs or double-counts them.
- **Memory** doesn't apply to jobs, or does, but through a different entry point.
- **Debugging** a scheduled run requires a different tool than debugging a chat.

ThinkWork collapses those by making the thread the canonical unit of work regardless of origin. A scheduled run gets a thread with `channel = 'AUTO'` and a prefix of `AUTO-`. It still has messages, turns, tool calls, and memory reads. You can open it in the admin app exactly the way you open a chat thread.

## The run shape

A typical automation run, end to end:

```
AWS Scheduler fires (cron) or EventBridge fires (event)
        │
        ▼
Step Functions state machine starts
        │
        ▼
ThinkWork creates AUTO- thread in Aurora
        │
        ▼
AgentCore invoked with configured prompt + agent
        │
        ▼
Agent loop runs — tools, memory, response
        │
        ▼
Step Functions records execution result
        │
        ▼
Thread marked succeeded or failed
```

Every step is observable. Step Functions has its own console for the state-machine view. AgentCore emits OTel spans that flow into turn accounting. The `AUTO-` thread has a full message timeline with every tool call and every memory read. An operator debugging a failed run has three complementary views of the same event.

## Two entry points, one execution path

ThinkWork distinguishes between **scheduled** and **event-driven** automations at the entry point:

- **Scheduled.** Something needs to run at 9am every weekday. AWS Scheduler fires. See [Scheduled and Event-driven](/concepts/automations/scheduled-and-event-driven/).
- **Event-driven.** A webhook arrives, a GitHub issue opens, a customer responds. The trigger path starts the run. Same page, second half.

Both paths converge on the same Step Functions state machine and the same AUTO thread creation. The distinction matters for configuration and for debugging (scheduled runs have predictable times; event-driven runs don't), but the downstream execution model is identical.

## Routines — multi-step automations

A single "invoke the agent with this prompt" is the simplest automation. Many real automations are multi-step:

1. Fetch yesterday's new tickets from the ticket system.
2. For each ticket, classify it.
3. For tickets classified as "urgent," draft a reply.
4. For drafted replies, request human approval via the Inbox.
5. Approved replies post back to the ticket system.

Each step is either an agent invocation, a tool call, or a wait. Routines are ThinkWork's name for these multi-step shapes, and they're backed by Step Functions state machines with branching, retries, and human-in-the-loop waits.

See [Routines and Execution Model](/concepts/automations/routines-and-execution-model/) for how steps compose, how retries work, and how failure propagates.

## Debugging a failed automation

Two complementary views:

- **The thread view.** Open the AUTO thread in the admin app. See the conversation, tool calls, memory reads, and any guardrail activations inline. This is usually enough when the failure is at the agent/turn level.
- **The execution view.** Open the Step Functions execution in the AWS console or via the admin app's Runs surface. See which state failed, how many retries were attempted, what the state-machine input and output look like at each step.

For agent-level failures, the thread view is faster. For state-machine-level failures (a step timed out, retries exhausted, a branching condition did the wrong thing), the execution view is the one to open.

<Aside type="tip">
  A common pitfall: thinking a "failed automation" means the agent did the wrong thing. Often it means the state machine couldn't *reach* the agent — a step that fetches input data failed, a trigger misfired, or a scheduler configuration was wrong. Always check the execution view first before spelunking the thread.
</Aside>
