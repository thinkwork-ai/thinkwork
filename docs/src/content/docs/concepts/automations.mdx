Automations are ThinkWork's prompt-first product surface for recurring or
manually rerunnable agent work. An Automation owns the prompt, execution Space,
trigger, run history, and evidence operators inspect.

Internally, Automations are backed by the AgentLoop runtime. AgentLoop remains
the durable contract for trigger, goal, worker, judge, policy, run ledger, and
evidence. Users should not need to learn that name to create useful recurring
work.

## Product Taxonomy

<CardGrid>
  <Card title="Automations" icon="time">
    Prompt-first autonomous work. Use them when work repeats or can be rerun, a
    selected Space should receive the execution thread, and the operator needs
    evidence explaining what happened.
  </Card>
  <Card title="Workflows" icon="gear">
    Explicit multi-step orchestration. Use them for known sequences with
    branching, retries, human-in-the-loop waits, and app or data operations that
    should render as a graph.
  </Card>
  <Card title="Evaluations" icon="approve-check">
    Test and scoring suites. Use them to measure agent behavior against datasets
    or rubrics. They may share judgment primitives with Automations, but they
    are not the recurring-work authoring surface.
  </Card>
</CardGrid>

## Creation Builder

New Automations start in a single block builder. The default page shows the
Automation name, triggers, instructions, MCPs/connectors, Advanced, and the
create action in one flow.

The common path is:

1. name the Automation;
2. choose a supported trigger, currently manual or schedule;
3. write a **Start session** instruction;
4. keep the default Space or choose where the Automation runs;
5. create it.

ThinkWork derives the internal goal, self-check judge, worker, and policy
defaults from that builder state. Advanced expands in place for explicit
criteria, judge settings, policy limits, evidence retention, and raw runtime
metadata. Chat help remains available as an optional setup-history helper that
can prefill the same builder; it is not the default creation mode.

## Spaces

Every Automation runs in a Space. New Automations default to the Space selected
in Agent settings, and the selected Space is persisted on the Automation.

When an Automation runs, ThinkWork creates the execution thread in that Space so
the result appears in the same conversation surface as ordinary agent work. For
older Automation records that do not yet have a saved Space, the runtime falls
back to the Agent default Space.

## Builder Threads

Optional chat help uses a real setup thread, but that thread is hidden from
normal navigation and search. ThinkWork creates or reuses a system-managed
Automation Builder Space with hidden visibility and stores setup conversation
history there.

The builder thread is not the execution thread. It is setup history. The
Automation detail page can link back to it when operators need to understand how
the draft was produced.

## Automation Loop Designer Skill

The optional chat helper is seeded with an `automation-loop-designer` workspace
skill. It helps turn a fuzzy instruction into a reviewable `AutomationDraft`:
prompt, Space, trigger, goal, judge, policy, and source metadata.

The skill is adapted from Kevin Simback's
[Looper](https://github.com/ksimback/looper) project under the MIT license. The
ThinkWork version keeps Looper's design-coaching ideas, including goal
clarification, verification rubrics, stop guards, observability, and privacy
boundaries, but emits a ThinkWork-native draft instead of an external runner or
slash command.

## Run Shape

```
Manual Run Now or AWS Scheduler fires
        |
        v
Automation dispatcher
        |
        v
AgentLoopRun + first iteration row
        |
        v
Execution thread in the selected Space
        |
        v
Agent wakeup -> AgentCore Pi goal-mode runtime
        |
        v
Worker reply + goal_run evidence
        |
        v
Judgment and evidence projection
        |
        v
Complete, continue, fail, budget-stop, or escalate
```

Manual and scheduled runs share the same dispatcher. That keeps local "Run now"
validation aligned with recurring schedule behavior.

Phase 1 executable judge modes are self-check and human-approval escalation.
Model judges and reviewer-agent judges are reserved in the shared contract for
later verification loops, but are not selectable as working Phase 1 behavior.

## Loop Suitability Gate

Before enabling a recurring Automation, check that the work is actually
loop-worthy:

- The task repeats often enough to justify automation, usually weekly or more.
- Bad output can be rejected by a test, hard rule, measurable condition,
  self-check criteria, or human-escalation gate.
- The worker can complete the job end to end with available tools and
  connectors.
- "Done" is objective enough to inspect, even when ThinkWork initially stores a
  runtime-inferred fallback criterion.

If a task fails that gate, keep it as a manual run, an ordinary prompt, or a
Workflow. Recurrence without judgment is only a schedule.

## Debugging

- **Automation behavior.** Open Settings -> Automations -> recent run. Prefer
  the linked execution thread for the user-visible outcome, then inspect the run
  evidence and terminal reason when needed.
- **Setup questions.** Open the linked builder thread from the Automation detail
  page when the saved draft needs context.
- **Workflow structure.** Open Settings -> Workflows -> run detail. Use the
  graph and step-detail panel.
- **Underlying schedule plumbing.** Check `scheduled_jobs`, the
  job-schedule-manager, and AWS Scheduler only when no Automation run exists.

<Aside type="tip">
  A common pitfall is debugging a scheduled Automation as if the schedule were
  the product. First ask whether an Automation run exists. If it does, debug the
  run thread and evidence. If it does not, debug the scheduler plumbing.
</Aside>
