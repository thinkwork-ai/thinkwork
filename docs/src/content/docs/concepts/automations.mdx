AgentLoops are ThinkWork's v1 automation primitive. An AgentLoop is a durable
contract for autonomous work: trigger, goal, worker, judge, loop policy, run
state, and evidence. A weekly account check-in, a recurring quality review, or a
manual "run this loop now" action should be modeled as an AgentLoop, not as a
bare schedule or prompt.

The old "Automations" noun now describes an implementation family, not the
product object. `scheduled_jobs`, AWS Scheduler, and EventBridge are plumbing.
The product surfaces operators inspect are AgentLoop runs, Workflow runs,
threads, and evaluation runs.

## Product Taxonomy

<CardGrid>
  <Card title="AgentLoops" icon="time">
    Autonomous goal/policy loops. Use them when work repeats or can be rerun,
    completion can be judged, and the operator needs run evidence explaining why
    the loop completed, continued, failed, or escalated.
  </Card>
  <Card title="Workflows" icon="gear">
    Explicit multi-step orchestration. Use them for known sequences with
    branching, retries, human-in-the-loop waits, and app or data operations that
    should render as a graph.
  </Card>
  <Card title="Evaluations" icon="approve-check">
    Test and scoring suites. Use them to measure agent behavior against datasets
    or rubrics. They may share judgment primitives with AgentLoops, but they are
    not how operators create recurring automation.
  </Card>
</CardGrid>

## Loop Suitability Gate

Before enabling a recurring AgentLoop, check that the work is actually
loop-worthy:

- The task repeats often enough to justify automation, usually weekly or more.
- Bad output can be rejected by a test, hard rule, measurable condition,
  self-check criteria, or human-escalation gate.
- The worker can complete the job end to end with available tools and
  connectors.
- "Done" is objective enough to express as completion criteria, not only taste.

If a task fails that gate, keep it as a manual run, an ordinary prompt, or a
Workflow. Recurrence without a judge is only a schedule.

## Manual Run First

New scheduled loops should prove themselves with a manual run before the
operator trusts the cadence. The expected first-use path is:

1. Open Settings -> AgentLoops.
2. Start from the `Weekly Agent Check-In` preset.
3. Choose the worker agent or profile.
4. Write goal intent and explicit completion criteria.
5. Save the loop, run it now, and inspect the AgentLoopRun evidence.
6. Enable or keep the schedule only after the manual run explains its outcome.

The run inspector is the durable debugger. It shows trigger source, goal/policy
snapshot, iterations, judgments, evidence pointers, cost, and terminal reason.

## How Schedules Fit

AWS Scheduler still wakes scheduled loops, but it is not the source of truth.
ThinkWork stores schedule plumbing in `scheduled_jobs`; when a scheduled
AgentLoop fires, the trigger path creates an AgentLoopRun and iteration, then
dispatches the worker through the same goal-mode path as a manual run.

That separation matters:

- Operators query and audit AgentLoop state through GraphQL and Settings.
- EventBridge/Scheduler rows can be reconciled without changing run history.
- Manual and scheduled runs share one dispatcher and one evidence model.
- Future event-driven or n8n-authored loops can trigger AgentLoops without
  owning AgentLoop identity, judgment, or state.

## AgentLoop Run Shape

```
Manual Run Now or AWS Scheduler fires
        |
        v
AgentLoop dispatcher
        |
        v
AgentLoopRun + first iteration row
        |
        v
Agent wakeup -> AgentCore Pi goal-mode runtime
        |
        v
Worker result + goal_run evidence
        |
        v
AgentLoop judgment projection
        |
        v
Complete, continue, fail, budget-stop, or escalate
```

Phase 1 executable judge modes are self-check and human-approval escalation.
Model judges and reviewer-agent judges are reserved in the shared contract for
later verification loops, but are not selectable as working Phase 1 behavior.

## Workflows Still Matter

A scheduled "every overnight, fetch new emails, classify each, draft urgent
replies, wait for approval, then send" is better as a Workflow: it is a known
state machine with visible steps. The Workflow run-detail graph shows step
status, input/output, cost, and Step Functions adapter evidence.

An AgentLoop can later trigger, contain, or be triggered by a Workflow, but the
two should not collapse into one table or one UI. AgentLoops answer "is this
autonomous goal making progress under policy?" Workflows answer "which explicit
step of this process ran or failed?"

## Debugging

- **AgentLoop behavior.** Open Settings -> AgentLoops -> run detail. Read the
  judgment, evidence, policy snapshot, and terminal reason.
- **Workflow structure.** Open Settings -> Workflows -> run detail. Use the graph
  and step-detail panel.
- **Underlying schedule plumbing.** Check `scheduled_jobs`, the
  job-schedule-manager, and AWS Scheduler only when the loop never fired.
- **Agent-level execution.** Follow the linked thread turn or Activity evidence
  from the AgentLoopRun when the worker ran but produced the wrong outcome.

<Aside type="tip">
  A common pitfall is debugging a scheduled AgentLoop as if the schedule were
  the product. First ask whether an AgentLoopRun exists. If it does, debug the
  run evidence. If it does not, debug the scheduler plumbing.
</Aside>
