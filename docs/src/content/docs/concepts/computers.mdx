A **ThinkWork Computer** is the user's durable AI work environment. It has an owner, a persistent EFS-backed workspace, runtime state, budget state, and an orchestration loop that can call tools directly or delegate focused work to managed Agents.

This reframes the product model:

- **Computer** — one per user, long-lived, stateful, workspace-owning, responsible for orchestration and continuity.
- **Agent** — a reusable delegated worker with a defined capability set, runtime, model, tools, and guardrails.
- **Template** — the reusable configuration base for either a Computer or an Agent.

The Computer is what makes the product feel less like "another agent list" and more like an AWS-native cloud computer for AI work. It can keep a live filesystem, hold installed CLIs, maintain local caches, run a real browser later, and route expensive or specialized reasoning to AgentCore workers when that is the better substrate.

<Aside type="note">
Phase 3 activates the runtime skeleton with image rollout tooling, CLI lifecycle controls, task enqueueing, EFS workspace file writes, Google CLI smoke checks, and a scheduled runtime reconciler. It does not yet promise full unattended browser automation, OAuth-backed Google Workspace actions, or production-grade per-tenant hard isolation beyond the shared Fargate/EFS foundation.
</Aside>

## Runtime model

Each Computer is backed by:

| Layer | Purpose |
|---|---|
| `computers` row | Owner, status, desired runtime state, budget fields, migration provenance |
| EFS access point | Persistent workspace mounted at `/workspace` inside the runtime task |
| ECS service | Always-on-ish lightweight runtime task, usually one task per active Computer |
| Runtime callback API | Service-auth endpoints for config, heartbeat, task claim, events, complete/fail |
| Computer tasks/events | Durable queue and audit stream for runtime work |
| Managed Agents | Delegated AgentCore workers for focused jobs that should not live in the Computer loop |

The default runtime image is intentionally small. It boots, verifies the workspace, heartbeats to the API, checks available CLIs, and polls for tasks. The platform can evolve the image with Google Workspace CLI tooling, browser-use tools, repo CLIs, and other local capabilities without changing the core product model.

Phase 3 task types are deliberately bounded:

| Task | What it proves |
|---|---|
| `health_check` | the runtime can claim work and write a health marker into EFS |
| `workspace_file_write` | the Computer owns a real live filesystem workspace |
| `google_cli_smoke` | the runtime image can carry Google Workspace CLI tooling without exposing user OAuth tokens |

## Desired state

Computers separate operator intent from observed runtime state:

| Field | Meaning |
|---|---|
| `desired_runtime_status` | What the platform wants: `running` or `stopped` |
| `runtime_status` | What the runtime or ECS manager last observed: `pending`, `starting`, `running`, `stopped`, `failed`, or `unknown` |
| `last_heartbeat_at` | Last callback from the running container |
| `ecs_service_name` | ECS service reconciled by the manager |
| `efs_access_point_id` | Workspace mount boundary |

Admin mutations can update metadata and desired state. Runtime provisioning and ECS desired-count changes flow through the service-auth Computer manager endpoint so GraphQL stays a product API rather than a direct AWS control plane.

A scheduled reconciler watches active Computers and moves ECS services toward the desired state in bounded batches. Operator CLI actions are still useful for immediate smoke tests and break-glass control, but the always-on product posture comes from the database intent plus reconciliation loop.

## Why not just AgentCore?

AgentCore remains the right home for managed Agents: fast delegated work, Bedrock-native runtime integration, and low operational surface area.

Computers add value where a long-lived environment matters:

- persistent filesystem without S3 sync choreography
- warm local state for CLIs, repos, credentials, browser profiles, caches, and scratch artifacts
- an orchestration process that can stream quickly and decide when to delegate
- future browser/computer-use capability in a real Linux environment
- clearer product language: the user owns a Computer; the Computer hires Agents

The cost target is deliberately modest: a tiny Fargate task plus shared EFS substrate is acceptable for the primary product surface, and heavier reasoning can still be outsourced to AgentCore.

## Migration model

Phase 2 migrates user-paired Agents into Computers only when the source shape is unambiguous:

- one active user-paired Agent for a human
- source template is typed as a Computer Template
- no conflicting existing Computer for that human

Blocked cases remain Agents until an operator resolves them. The migration report classifies each owner group with severity, recommended action, and apply disposition so operators can dry-run before creating Computers.
