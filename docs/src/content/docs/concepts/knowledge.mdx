Memory is how ThinkWork carries useful context forward without treating any one backend as the product. It's the layer that decides — on every turn — what an agent should see *beyond* the latest user message: which documents to pull, which prior facts to recall, which entity pages from the agent's own wiki to surface, and how to weave all of that into the model's context window.

The clean model is:

- **Threads** keep the full record of work.
- **Memory** decides what context should carry forward into the current turn.
- **Agents** act on that assembled context.

That means Memory is not just document retrieval (RAG), and not just long-term recall. It's the harness-owned context layer above whatever retrieval and memory engines we happen to use.

## Why we frame it this way

Most agent systems conflate three different things — document retrieval, long-term memory, and turn assembly — and pick one backend as "the memory product." That's convenient in the short term and painful in the long term:

- When the backend changes, every agent's behavior changes.
- When a new context source appears (a wiki, a graph, a new embedding model), the system has no seam to plug it in without rewriting.
- When compliance asks "what did this agent know at the moment it made that decision," the answer depends on which backend is "memory" today.

ThinkWork picks the other design. There's a stable memory contract owned by the harness. Underneath, engines come and go — Hindsight, AgentCore Memory, Bedrock Knowledge Bases, the compounding-memory wiki. Above, agents and tools see one interface.

## The three layers

Memory in ThinkWork spans three concerns. They are distinct; they don't collapse into "memory == KB" or "memory == chat history."

### How the layers relate

A single agent turn, at a glance:

```
User message arrives in a thread
        │
        ▼
  Context assembly (the Memory layer)
        │
        ├─ Recent thread history (windowed)
        ├─ Long-term memory recall (Hindsight / AgentCore Memory)
        ├─ Document retrieval (Bedrock KB)
        ├─ Compounding-memory wiki pages (when the agent has one)
        │
        ▼
  Assembled context → Bedrock model
        │
        ▼
  Response + tool calls → back into the thread
```

Every layer is optional per agent. A focused chatbot may use only document knowledge. A long-running research agent adds long-term memory. An agent that lives in a rich workspace (a user's inbox, a product's Slack) uses all four. The harness handles the assembly without each agent's author wiring the pieces by hand.

## The adapter story

For long-term memory specifically, ThinkWork supports two adapters, and exactly one is canonical per deployment:

- **[Hindsight](https://hindsight.vectorize.io/)** — the default for hosted ThinkWork. Runs as an ECS service in your VPC, offers semantic search, entity extraction, and reflection primitives. Richer, slightly heavier.
- **[AgentCore Memory](https://docs.aws.amazon.com/bedrock-agentcore/latest/devguide/memory.html)** — the default for serverless-friendly deployments. Four built-in strategies (semantic, summarization, user-preference, episodic) with per-turn auto-retention. Thinner, no persistent service to run.

Both adapters implement the same memory contract the harness exposes. Swapping adapters swaps engines; it does not swap APIs for tools, agents, or the compounding-memory pipeline built on top.

<Aside type="note">
  Short-term context still comes from thread history. Long-term memory should never masquerade as the thread record — the thread is always the canonical history of what happened.
</Aside>

## Where compounding memory fits

The compounding-memory layer is the newest and most distinctive piece. It sits *above* the long-term memory adapter: raw memory records flow from Hindsight (or AgentCore Memory, when that path matures) into a compile pipeline that produces a continuously-updated wiki of entity and topic pages.

That wiki is not an index of memory; it's a derived, structured view of what the agent has learned. A memory record about "dinner at Franklin Barbecue" contributes to a Franklin Barbecue page (entity) and an Austin Restaurants page (topic hub). Agents read the wiki at recall time — often faster and more targeted than raw memory search — and operators browse it in the admin app to see what an agent actually knows.

This is the main reason memory in ThinkWork is framed as its own layer, not an implementation detail of an engine. The wiki works on top of Hindsight today; it will work on top of any adapter that implements the same recall contract.
