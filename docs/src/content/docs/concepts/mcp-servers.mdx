MCP servers expose tools that agents can call during conversations. By registering an MCP server, you give agents access to external capabilities — CRM lookups, task management, database queries — without writing custom skill code.

Thinkwork connects to MCP servers using HTTP streaming (Streamable HTTP transport). The agent container creates an `MCPClient` via the Strands Agent SDK, which handles the full connection lifecycle: initialization, tool discovery, tool calls, and cleanup.

## How it works

```
Admin registers MCP server (URL, auth type)
    → Assigns to agent template (Skills tab)
    → Template syncs to agents
    → User connects OAuth from mobile app (if required)
    → At invocation time:
        1. Wakeup processor resolves auth token
        2. MCP config passed to agent container
        3. Strands Agent creates MCPClient, discovers tools
        4. Agent calls tools as needed during conversation
        5. MCPClient cleaned up after invocation
```

## Registration

MCP servers are registered at the **tenant level** in the Admin UI under **MCP Servers** in the sidebar. Click **Register Server** and provide:

| Field | Description |
|-------|-------------|
| Name | Display name (e.g., "Lastmile CRM") |
| URL | MCP endpoint URL (e.g., `https://mcp.example.com/crm`) |
| Transport | `streamable-http` (recommended) or `sse` |
| Authentication | None, Tenant API Key, or OAuth (server-managed) |

After registering, use **Test Connection** to verify connectivity and cache the server's tool list.

### CLI

```bash
thinkwork mcp add "my-server" \
  --url https://mcp.example.com/crm \
  --auth-type tenant_api_key \
  --api-key sk-xxx \
  --tenant my-tenant -s dev

thinkwork mcp list --tenant my-tenant -s dev
thinkwork mcp test <server-id> --tenant my-tenant -s dev
```

## Authentication

### None

No authentication headers sent. For MCP servers on private networks or servers that handle auth separately.

### Tenant API Key

A shared API key stored in AWS Secrets Manager. All agents in the tenant use the same key, sent as a `Bearer` token in the `Authorization` header.

### OAuth (server-managed)

For MCP servers that need per-user identity (e.g., a CRM scoping data by user). The MCP server manages its own OAuth flow per RFC 9728.

The flow:
1. Admin registers the MCP server with auth type "OAuth"
2. Admin assigns MCP server to an agent template
3. Template syncs to agents
4. **User** opens the mobile app → **MCP Servers** screen → taps **Connect**
5. The Thinkwork API discovers the MCP server's OAuth metadata via `/.well-known/oauth-protected-resource`
6. User is redirected to the MCP server's authorization endpoint (PKCE flow)
7. After sign-in, the access token + refresh token are stored in Secrets Manager
8. On subsequent agent invocations, the token is resolved and sent to the MCP server

<Aside type="note">
If a user has not connected their OAuth for an MCP server, that server is skipped for their agent's invocations. The agent continues with its other tools. The mobile app shows "Needs Connection" for servers awaiting OAuth.
</Aside>

#### Token refresh

When a token is within 5 minutes of expiry and a refresh token exists, the wakeup processor automatically refreshes it:
1. Discovers the MCP server's token endpoint via RFC 9728
2. Exchanges the refresh token for a new access token
3. Updates Secrets Manager and the `user_mcp_tokens` table
4. If refresh fails, marks the token as expired (mobile shows "Reconnect")

## Assigning to agents

MCP servers are assigned on the **agent template Skills tab**. The MCP Servers section shows registered servers and lets you add/remove them. When you sync the template to agents, MCP assignments are copied to `agent_mcp_servers`.

Assignments are stored in the `agent_template_mcp_servers` join table (not JSONB).

### Via CLI

```bash
thinkwork mcp assign <mcp-server-id> --agent <agent-id> -s dev
thinkwork mcp unassign <mcp-server-id> --agent <agent-id> -s dev
```

## Runtime behavior

During an agent invocation:

1. The wakeup processor queries `agent_mcp_servers` joined with `tenant_mcp_servers`
2. For each enabled server, auth is resolved:
   - **Tenant API key**: read from `auth_config`
   - **OAuth**: read user's token from Secrets Manager via `user_mcp_tokens.secret_ref`
   - Servers with unresolved auth are **skipped** (not injected)
3. MCP configs are passed to the agent container in the invocation payload
4. The container creates `MCPClient` instances with `streamablehttp_client`
5. The `MCPClient` is passed to the Strands `Agent` as a tool provider — the Agent manages the connection lifecycle (start, tool discovery, tool calls, stop)
6. After the agent completes, MCP clients are cleaned up

<Aside type="caution">
Do not call `MCPClient.start()` manually before passing it to the Agent. The Agent's tool loading calls `start()` internally. Calling it twice causes a "client session is currently running" error.
</Aside>

If an MCP server is unreachable, it's logged as a warning and skipped — the agent continues with its other tools.

## Tool invocation tracking

MCP tool calls appear in the thread timeline with:
- Tool name
- MCP server name (for attribution)
- Input/output previews
- Execution status

## Data model

| Table | Scope | Purpose |
|-------|-------|---------|
| `tenant_mcp_servers` | Tenant | Registry of available MCP servers |
| `agent_template_mcp_servers` | Template | Template-level MCP assignment (join table) |
| `agent_mcp_servers` | Agent | Per-agent enablement (synced from template) |
| `user_mcp_tokens` | User | Per-user OAuth token metadata + Secrets Manager ref |

Token storage: `user_mcp_tokens.secret_ref` points to a Secrets Manager secret at `thinkwork/{stage}/mcp-tokens/{userId}/{mcpServerId}` containing `{ access_token, refresh_token, token_type, obtained_at }`.

## Supported transports

| Transport | Description |
|-----------|-------------|
| Streamable HTTP | Modern HTTP-based MCP transport (recommended) |
| SSE | Server-Sent Events transport (legacy, still widely supported) |

<Aside type="note">
Stdio-based MCP servers are not currently supported. HTTP streaming covers most remote MCP servers.
</Aside>
