External tasks — work items owned by an external system like LastMile — render natively in the mobile app, not as markdown and not as iframes. Each task becomes a thread with a **pinned task card** above the chat timeline; updates land within ~2s via AppSync subscriptions; status and field edits round-trip back to the provider.

For the architecture (adapter seam, webhook pipeline, envelope shape, block grammar), see the [External Task Connectors](/concepts/connectors/external-tasks/) concept page. This page covers what the user actually sees on their phone.

## When the Tasks tab appears

The Tasks tab is **conditional**. The app bootstraps with two tabs (Threads + Settings). Once the user has at least one connected task integration from [Connectors](/applications/mobile/integrations-and-mcp-connect/) (currently LastMile; future: Linear), a third **Tasks** tab appears with a count badge of assigned tasks.

If no task connector is active, the Tasks tab stays hidden.

## The Tasks tab

Tapping **Tasks** lands on a filtered view of the user's assigned external tasks, newest first. Each row carries a task identifier (`TASK-N`), title, description snippet, and a relative timestamp.

![Tasks tab list view showing assigned LastMile tasks](/images/external-tasks/tasks-tab-list.png)

The same thread subscription that drives the Threads list (`useThreadUpdatedSubscription`) keeps this view live — new tasks land without a pull-to-refresh, and the tab badge updates as tasks are assigned or resolved.

A **filter icon** in the header lets the user narrow to active / done / all. A **task-compose footer** at the bottom of the screen (where the Threads tab has "Start a new thread…") reads "Create a new task…" on this tab — a future hook for authoring tasks directly from the mobile app.

## The task detail

Tapping a task row opens the thread detail screen with a **pinned task card** at the top above the chat timeline:

![Task detail: pinned LastMile task card above the chat timeline](/images/external-tasks/task-card-detail.png)

The card is fully rendered from the task envelope's block list — there is no per-provider UI. Reading top-to-bottom, the card carries:

- **`task_header`** — title, description, "LastMile Task" provider label in the back navigation bar
- **`field_list`** — labelled definition list: STATUS, PRIORITY, ASSIGNEE, DUE
- **`action_bar`** — row of action buttons (Change status, Assign, Comment) plus a primary **Edit** button for opening the full form
- **`activity_list`** — rendered inline when there are webhook-driven audit rows; hidden when empty (as in this screenshot)

Below the card, the **chat timeline** runs normally: user messages on the right, agent messages on the left, streaming render, tool-call inline expansion. Chat and task UI share a screen but stay separate — the activity on the card and the messages below are two different feeds (see [Activity vs chat](#activity-vs-chat)).

## Block grammar from the user's perspective

Every external task card is made from the same bounded set of blocks. The same grammar drives LastMile today and will drive future providers unchanged.

| Block | What the user sees |
|---|---|
| `task_header` | Title (bold), description (muted), optional provider badge and updated timestamp |
| `field_list` | Two-column label + value definition list; dates localized, select values resolved to labels, assignee normalized to name/email when possible |
| `badge_row` | Inline pill cluster for tag-like fields |
| `activity_list` | Webhook-driven audit rows with history icon + summary + relative time; newest first, capped at ~10 |
| `action_bar` | Row of primary action buttons that open bottom-sheet modals |
| `form` | Bottom-sheet edit form rendered from the task's edit form schema |
| `section` | A titled group of nested blocks (used for longer forms) |
| `empty_state` | Fallback when a block has no data |

## The edit modal

Tapping **Edit** opens a 75% bottom-sheet with the full edit form:

![Edit task bottom sheet: status pills, priority pills, assignee, due date, description textarea, Save changes button](/images/external-tasks/edit-modal.png)

Field types map to controls:

- **Status** and **priority** — pill buttons with the current selection highlighted
- **Assignee** — free-text input
- **Due date** — text input (ISO-8601 format)
- **Description** — multi-line textarea
- **Save changes** — right-aligned outlined button at the bottom

On submit, the mobile card fires the `executeExternalTaskAction` GraphQL mutation with the action type (`edit_fields`) and the form values. The resolver dispatches to the adapter's `executeAction`, which translates into provider API calls, and then immediately calls `refresh()` so the envelope reflects the post-write state before returning. The card re-renders with the new values.

<Aside type="caution">
**Status and priority form edits currently fail upstream for LastMile.** The form's select options use curated value strings (`"todo"`, `"in_progress"`) but LastMile's status-update tool expects opaque workflow IDs. Status changes through the dedicated **Change status** button still work; the form-field path does not yet. Fix is in flight — see [Current limits](#current-limits).
</Aside>

## Activity vs chat

When webhook events land for a task (status changed, assigned, updated), each one writes a system row to the thread's message table with `metadata.kind = "external_task_event"`. The mobile app derives two feeds from those rows:

- **Activity list** on the task card — filters to `role = system` + `metadata.kind = "external_task_event"`
- **Chat timeline** below the card — filters to `role ∈ {user, assistant}`

The split is deliberate: audit events belong on the card, conversation belongs in chat. If you need to render something durable in the conversation from an external source, write a system message with a different `metadata.kind` so the activity filter ignores it.

Because this task detail had no webhook events yet, the activity block wasn't rendered in the screenshot above. Once LastMile starts firing events, a history-icon row appears inside the card between the field list and the action bar, newest first.

## Realtime and push

Everything on the task card and the chat timeline is push-driven, not polled:

- `useThreadUpdatedSubscription` re-executes the thread query on any thread-level change (including new system messages from webhook events)
- `useNewMessageSubscription` delivers streaming chunks for the chat timeline

Push notifications fire on top of those subscriptions for a narrow set of events only — assignment, reassignment, status change, and `status`/`status_id`/`due_at`/`due_date` updates. Description edits, title edits, priority changes, and updated-at-only diffs never fire push. See [Push Notifications → Narrow push policy for external tasks](/applications/mobile/push-notifications/#narrow-push-policy-for-external-tasks).

## Current limits

A few mobile-visible rough edges worth knowing about. Fuller detail lives on the [concept page](/concepts/connectors/external-tasks/#current-limits):

- **Status / priority form edits fail upstream for LastMile** (see callout above). Use the dedicated **Change status** button for now.
- **Assignees other than the signed-in user render as raw IDs.** The edit form above shows `user_e2e_test_synthetic` because the underlying `tasks_get` call only hydrates the full assignee object for the task's own assignee. The mobile app has no user-lookup fallback yet.
- **Non-canonical priority labels render as-is.** Curated pills are `Urgent / High / Normal / Low`; a provider value like `medium` bypasses the label lookup and renders as the raw string.
- **LastMile is the only adapter today.** Linear is the committed next target. The mobile card renders whatever `buildBlocks()` returns, so adding a provider does not require a mobile release.
