The desktop app is the macOS client for end users who want ThinkWork Spaces as a daily-driver application instead of a browser tab. It packages the same Spaces experience served at `app.thinkwork.ai`: threads, Spaces navigation, agent chat, generated artifacts, side panels, and the command composer.

The app is built with Electron. Most product UI lives in the shared `apps/spaces` React app, while the desktop shell adds native-window behavior, app branding, protocol callbacks, secure token storage, and packaging.

<Aside type="note">
  The desktop app is not a separate product surface. It is the Spaces web app
  running inside a native shell. When shared Spaces UI changes are released into
  a new desktop build, the same thread, sidebar, composer, and artifact changes
  appear in the desktop app.
</Aside>

## Who uses it

End users inside a ThinkWork tenant. The desktop app is for people who spend most of their day in Spaces and want:

- A dedicated macOS app with its own Dock icon
- A native window that can sit beside browsers, terminals, spreadsheets, and communication tools
- Persistent sign-in backed by the macOS keychain
- The same Spaces, threads, and artifacts they use on the web

Operators still configure tenant-level settings in the [Admin Web App](/applications/admin/). Personal mobile workflows, push notifications, and user-scoped integration setup still live in the [Mobile App](/applications/mobile/).

## Sign in

When the app starts without a saved session, it shows a minimal sign-in screen with a **Log in** button. Clicking the button opens the tenant's Cognito hosted sign-in flow in the user's browser.

After the user completes Google/Cognito sign-in, the browser asks to open **ThinkWork Spaces**. Approving that prompt sends the OAuth callback back to the desktop app through the ThinkWork desktop deep-link scheme. The app stores the resulting Cognito tokens in encrypted macOS safe storage and opens the Spaces shell.

On a trusted Mac, the user can select the browser's "always allow" option for the ThinkWork callback prompt to make future sign-ins smoother.

<Aside type="tip">
  macOS may ask for permission to use confidential information in the "ThinkWork
  Spaces Safe Storage" keychain item. Allowing this lets the desktop app restore
  the saved session instead of asking the user to sign in again.
</Aside>

If Cognito rejects a login attempt, the desktop app returns to the sign-in screen and shows the OAuth error message so the user can retry with a fresh login URL.

## Header and navigation

The desktop shell uses native-style app chrome:

| Area               | Behavior                                                                                                                                               |
| ------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Sidebar header** | Contains the macOS window controls, sidebar toggle, back button, and forward button. Empty space in this header can be dragged to move the app window. |
| **Main header**    | Shows the active page or thread title, plus page actions such as overflow, info, fullscreen, and side-panel controls.                                  |
| **Sidebar**        | Runs all the way to the top of the window so the desktop app keeps the same visual frame as the web Spaces sidebar.                                    |

The desktop app intentionally removes the post-login product title from the header. Once signed in, the header should focus on the user's current thread or page.

## Web and desktop updates

Most user-facing work should be built in the shared Spaces app first. Updates to thread views, composer behavior, generated artifacts, side panels, sidebar navigation, and shared styling flow into both:

- **Web** when `app.thinkwork.ai` is deployed
- **Desktop** when a new Electron build packages the updated Spaces bundle

The desktop app does not load the production website live at runtime. It packages a built Spaces bundle into the app, so web changes flow into desktop at build/release time.

Desktop-specific work is still needed for native concerns:

- OAuth callback and deep-link handling
- Secure token storage
- App name, icon, signing, notarization, and update channels
- Native window chrome and draggable regions
- Differences between browser APIs and Electron APIs
