Automations are the admin app's v1 surface for recurring or manually
rerunnable autonomous work. They own the prompt, Space, trigger, run history,
and evidence that operators inspect. Internally, the AgentLoop runtime keeps
the durable trigger, goal, worker, judge, policy, run ledger, and evidence
contract. Schedule plumbing such as `scheduled_jobs`, AWS Scheduler,
EventBridge, and webhook ingress remains behind the product surface.

Active multi-step work appears in **Settings -> Workflows**. Step
Functions-backed workflows still use the Routine adapter internally, so old
Routine deep links continue to resolve during the transition, but Routines are
not presented as a separate top-level product surface.

## Where it lives in the admin sidebar

The current operator placement is:

```
Settings
├── Automations            ← /settings/automations
├── Workflows              ← /settings/workflows
└── Webhooks               ← webhook endpoints
```

The previous Settings AgentLoop list redirects to `/settings/automations`.
Old `/automations` shell routes redirect into the Automations list or the
scheduled-job compatibility detail. Old Routine detail and execution URLs
remain compatibility routes that either redirect to the matching Workflow
detail/run or render the Step Functions adapter detail as a fallback.

## Automations

**Inventory route:** `/settings/automations`
**Detail route:** `/settings/agent-loops/:agentLoopId`
**Run-detail route:** `/settings/agent-loops/:agentLoopId/runs/:runId`

An Automation should be used when:

- the work repeats or needs a durable "run now" button;
- completion can be judged by criteria, a self-check, or a human escalation;
- the worker can run with configured tools and connectors;
- operators need evidence explaining completion, continuation, failure, budget
  stop, or escalation.

Phase 1 Automations support manual and scheduled triggers, worker goal-mode
dispatch, self-check/human-approval judgment policy, and a run inspector backed
by AgentLoop internals. Model/reviewer judges and event-driven triggers remain
later phases.

## Workflows

**Inventory route:** `/settings/workflows`
**Detail route:** `/settings/workflows/:workflowId`
**Run-detail route:** `/settings/workflows/:workflowId/runs/:runId`

A Step Functions-backed workflow is an Amazon States Language state machine the chat builder agent emits, validated and provisioned by the Routine adapter's `publishRoutineVersion` GraphQL mutation. Each one has:

- **An ASL state machine** versioned via Step Functions `PublishStateMachineVersion`. The adapter records the version ARN so Workflow runs can show exactly which definition executed.
- **An agent-authored markdown summary** (`documentationMd`), regenerated on every publish, that names the steps the operator will see in the run-detail graph plus any HITL approval points.
- **A step manifest** (`step_manifest_json` on `routine_asl_versions`) that maps ASL state names to v0 recipe ids — the run-detail graph derives nodes from this manifest so brand-new executions render their shape immediately.
- **A visibility model** — `visibility = 'agent_private'` (only the owning agent can invoke) or `visibility = 'tenant_shared'` (any agent in the tenant). Splits the conflated `agentId` field; `owning_agent_id` is the authoring/stewardship agent for private Step Functions-backed workflows.

### The workflow inventory

The inventory aggregates Step Functions-backed workflows, n8n-connected workflows, CRM lifecycle workflows, and future connected-app/native sources. Columns show workflow identity, readiness, source, trigger family, lifecycle status, version, and last run. Blocked workflows remain visible with readiness reasons so operators can see missing OAuth, MCP, plugin, or engine prerequisites.

Step Functions-backed rows expose a Step Functions tab in detail, reusing the existing graph and definition panel. Source-specific setup remains on source pages such as `/settings/plugins/n8n`.

### The workflow detail page

Workflow detail has source-appropriate tabs:

- **Overview** — lifecycle, readiness, trigger family, source binding, and readiness reasons.
- **Step Functions** — only for `step_functions_routine` bindings; embeds the Routine adapter definition panel.
- **Runs** — recent Workflow runs with canonical status, trigger source, start time, and duration.
- **Definition** — version, source kind, capability snapshot, and definition snapshot.

### The workflow run-detail page

The customer-visible "watch the workflow execute" surface combines canonical Workflow ledger data with source-specific evidence:

- **`ExecutionGraph`** — vertical-stepper visualization of the state machine. Nodes derive from the routine_asl_versions step manifest backing this specific execution (resolved via `RoutineExecution.aslVersion`); fall back to deriving from `routine_step_events` when the manifest isn't fetchable. Latest-event-per-node selection collapses retry chatter into a single graph dot. Status icons: spinning loader for running, green check for succeeded, red X for failed, purple pause for awaiting_approval, amber clock for timed_out.
- **`StepDetailPanel`** — click a step in the graph; this panel shows input/output JSON, recipe type, retry count, LLM cost, duration. For `python` recipe steps with S3-offloaded stdout/stderr, the inline 4KB preview surfaces with a "(truncated)" marker plus the S3 URI for full output.
- **Workflow evidence** — backend/app proof such as Step Functions executions, n8n execution IDs, CRM event references, and redacted payload summaries.

The page polls every 5s while non-terminal. Subscription wiring (AppSync `OnRoutineExecutionUpdated`) is a Phase E follow-up — current polling is fine at 4-tenant scale.

## Scheduled-Job Compatibility

**Compatibility detail:** `/settings/automations/:scheduledJobId`
**Retired shell:** `/automations` redirects to `/settings/automations`
**Retired shell detail:** `/automations/:scheduledJobId` redirects to the
Settings compatibility detail

The `scheduled_jobs` substrate behavior is unchanged. It still carries internal
scheduled work for evals, skill runs, idle-memory learning, AgentLoop schedules,
and Workflow triggers. The EventBridge -> `job-schedule-manager` ->
`job-trigger` provisioning chain remains the operational wake path.

Operators should start by inspecting the owning product run:

- Automation schedule fired? Open the Automation run inspector.
- Workflow schedule fired? Open the Workflow run detail.
- No product run exists? Then inspect `scheduled_jobs`, AWS Scheduler, and
  `job-trigger` logs.

## Webhooks

**Route:** `/settings/webhooks`
**Detail:** `/settings/webhooks/:webhookId`

The webhook surface behavior is unchanged: webhook tokens, target_type (agent |
workflow, with `routine` as the compatibility storage value for Step
Functions-backed workflow targets), enabled toggle, invocation history.
Workflow-target webhooks create Workflow run evidence when they invoke
multi-step work. Event-driven Automations are a later phase; webhooks may
trigger them later, but ThinkWork should still own AgentLoop state, judgment,
and evidence behind the product surface.

## What does _not_ live here

- **Workspace HITL reviews** — system-agent workspace reviews surface in [Inbox](/applications/admin/inbox/), not Automations or schedule plumbing. Workflow-driven HITL approvals (the `inbox_approval` recipe) also surface in Inbox; deciding an inbox item with `type = 'routine_approval'` resumes the paused Step Functions execution.
- **Per-thread agent invocations** — those live in [Threads](/applications/admin/threads/). Multi-step workflows are distinct: a Workflow run has its own ledger and, for Step Functions-backed workflows, adapter evidence in `routine_executions` rather than a `thread_turns` row.
- **Per-user OAuth or integration credentials** — those live in the mobile app. Admin owns tenant-wide infra and config; users own their own integration auth on mobile.
- **Legacy Python routines** — `engine = 'legacy_python'` rows are archived during the compatibility migration and excluded from active Workflow inventory. They're not deleted (status flips to `archived`); they stop being actionable from this UI.

## Rollout Notes

- Apply the AgentLoop database migration before deploying API/web code that
  expects the Automation backing tables.
- Keep `/api/scheduled-jobs` and `/api/job-schedules` routes deployed; internal
  jobs and compatibility details still use them.
- Do not introduce new REST routes for Automations unless GraphQL is
  insufficient.
- If a tenant has user-created scheduled jobs from the retired Automations shell,
  preserve their detail URLs until they are migrated to AgentLoop definitions or
  explicitly retired with operator evidence.

## Under the hood

- Workflows: first-class `workflows`, `workflow_versions`, `workflow_triggers`, `workflow_engine_bindings`, `workflow_runs`, `workflow_run_events`, and `workflow_evidence` rows form the product control plane and run ledger.
- Step Functions adapter: `routines` stores the adapter record; `routine_asl_versions` is the ASL version history; `routine_executions` records each SFN execution; `routine_step_events` is the per-step event log. EventBridge bridges SFN execution-state-change events into `routine-execution-callback` so the adapter execution row tracks lifecycle even for `agent_invoke`-only workflows that have no wrapper Lambda.
- Automations: `agent_loops`, `agent_loop_versions`, `agent_loop_runs`,
  iterations, judgments, and evidence tables own autonomous loop identity.
- Schedules: `scheduled_jobs` table; `job-schedule-manager` reconciles to AWS
  Scheduler; `job-trigger` fires the owning AgentLoop, Workflow, eval, skill,
  idle-memory, or other internal target.
- Webhooks: `webhooks` table; `webhooks` handler authenticates the public endpoint and routes the payload.

`rate()` schedules treat the cadence as creation-time + interval, not wall-clock-aligned — a `rate(1 hour)` job created at 09:23 fires at 10:23, 11:23, … rather than on the hour. Use `cron()` for wall-clock alignment. The Schedule create form steers operators toward `cron()` for this reason.

User-initiated create and update Lambda invokes use `RequestResponse` so errors surface synchronously — no fire-and-forget paths.
