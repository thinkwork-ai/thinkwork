Symphony is the admin surface for ThinkWork's first production tracker connector experience. It lets operators configure the Linear connector, bind it to a target Computer, verify GitHub PR-producing setup, and inspect each connector run without reaching for SQL.

Today Symphony is Linear-only. It proves the connector lifecycle: Linear issue with the `symphony` label -> Computer-owned thread -> managed-Agent work -> draft PR -> Linear `In Review` writeback.

## Where it lives

**Route:** `/symphony`  
**File:** `apps/admin/src/routes/_authed/_tenant/symphony.tsx`

The page has two tabs:

| Tab        | Purpose                                                                        |
| ---------- | ------------------------------------------------------------------------------ |
| Connectors | Configure active, paused, unhealthy, or archived connector rows                |
| Runs       | Inspect connector executions and their Computer/delegation/writeback lifecycle |

The page lives under **Work** in the admin sidebar because operators use it to connect ThinkWork to existing work systems, not to author managed Agents.

## Connectors tab

The Connectors tab shows the tenant's connector rows. Each row stays single-line and surfaces:

- connector status;
- connector name and type;
- target type and target id;
- enabled state;
- last updated time;
- run/edit/archive controls.

For the Linear checkpoint, the expected connector type is `linear_tracker` and the expected target type is **Computer**. Direct managed-Agent, routine, and hybrid-routine targets are advanced paths; the normal Symphony proof uses a Computer target.

## Linear setup form

Creating or editing the Linear connector uses structured fields first. Advanced JSON remains available for inspection and precise edits, but it is not the primary operator path.

Key fields:

| Field                    | Expected checkpoint value                          |
| ------------------------ | -------------------------------------------------- |
| Linear team key          | The Linear team key, such as `TECH`                |
| Label                    | `symphony`                                         |
| Linear credential        | Usually `linear`                                   |
| Target Computer          | The Computer that should own connector-origin work |
| Dispatch writeback state | Usually `In Progress`                              |
| PR writeback state       | Usually `In Review`                                |
| GitHub credential        | Usually `github`                                   |
| GitHub owner             | `thinkwork-ai` for the checkpoint repo             |
| GitHub repo              | `thinkwork` for the checkpoint repo                |
| Base branch              | `main`                                             |
| Checkpoint file path     | Usually `README.md`                                |

<Aside type="caution">
  The checkpoint label is `symphony`, not `symphony-eligible`. Using the old
  label can route the issue through an older Symphony project or create
  confusing duplicate pickup behavior.
</Aside>

### GitHub setup required

If the connector row or dialog shows **GitHub setup required**, the connector cannot safely produce draft PRs yet. The usual causes are:

- the configured GitHub credential slug does not exist for the tenant;
- the credential exists but is inactive;
- the credential does not have write access to the configured repo.

Create or reactivate the tenant GitHub credential, return to Symphony, and save the connector through the structured form. Do not hide the problem by editing Advanced JSON directly.

## Runs tab

The Runs tab is the operator view of connector lifecycle. It shows recent connector executions and hides cancelled historical rows by default.

A successful Linear PR-producing run should show:

| Column       | Expected signal                                      |
| ------------ | ---------------------------------------------------- |
| State        | `Terminal`                                           |
| External ref | Linear issue key, such as `TECH-70`                  |
| Connector    | The Linear connector name                            |
| Lifecycle    | Completed task, completed delegation, succeeded turn |
| Writeback    | `Linear: In Review`                                  |
| PR           | Link to the draft pull request                       |
| Open         | Button to open the Computer-owned thread             |
| Age          | Recent relative timestamp                            |

The Runs tab is the first checkpoint surface. Use SQL only when the UI is unclear or when diagnosing historical stale rows.

## Manual run

The run action is useful for debugging an individual connector, but the production checkpoint should rely on the scheduled poller. A real unattended pass means:

1. Create one fresh Linear issue with only the `symphony` label.
2. Do not manually invoke the poller Lambda.
3. Wait for the scheduled poller window.
4. Verify the Linear issue moves to `In Progress`, then `In Review` after the draft PR exists.
5. Verify the Runs row shows the full lifecycle and PR link.

## Known limits

- **Linear-only.** Symphony currently documents and operates the Linear tracker proof.
- **One explicit target Computer.** Automatic external actor routing and team queues are future work.
- **Advanced JSON exists for escape hatches.** Operators should prefer the structured Linear and GitHub fields unless debugging.
- **Historical stale rows may exist.** The Runs tab hides cancelled rows by default; the runbook covers cleanup for old pre-fix rows.

## Under the hood

The admin route lives in `apps/admin/src/routes/_authed/_tenant/symphony.tsx`. Connector form defaults, validation helpers, lifecycle display helpers, writeback formatting, and PR link extraction live in `apps/admin/src/lib/connector-admin.ts`.

GraphQL operations for connectors and connector executions are defined in `apps/admin/src/lib/graphql-queries.ts` and generated into `apps/admin/src/gql/`. The backend connector resolvers live under `packages/api/src/graphql/resolvers/connectors/`.
