The Computers surface is where operators manage the durable per-user ThinkWork Computer. A Computer owns the user's live workspace and orchestration runtime; managed Agents remain available as delegated workers.

Computers are also valid connector-facing actors. The current Linear/Symphony checkpoint routes a `symphony`-labeled Linear issue to a Computer as a `connector_work` task, records a `connector_work_received` event, and creates a Computer-owned connector thread before any deeper runtime delegation occurs.

## Computer list

**Route:** `/computers`  
**File:** `apps/admin/src/routes/_authed/_tenant/computers/index.tsx`

The list is driven by `ComputersListQuery` and shows:

| Column    | Notes                                             |
| --------- | ------------------------------------------------- |
| Computer  | Name, slug, and detail link                       |
| Owner     | Human owner for the one-user Computer             |
| Status    | Product lifecycle status                          |
| Runtime   | Desired and observed runtime state                |
| Template  | Linked Computer Template                          |
| Budget    | Monthly cap and spend                             |
| Heartbeat | Last runtime callback                             |
| Migration | Source Agent provenance when created by migration |

The primary sidebar includes **Computers** in the Work group with a live count badge.

## Computer detail

**Route:** `/computers/:computerId`  
**File:** `apps/admin/src/routes/_authed/_tenant/computers/$computerId.tsx`

The detail page shows three operational panels:

- **Status** — desired runtime status, observed runtime status, lifecycle status, budget pause state, heartbeat, and last active time.
- **Runtime** — ECS service name, EFS access point, live workspace root, and runtime config JSON.
- **Migration** — source Agent, migration metadata, and creation provenance.

The status panel can update `desired_runtime_status`. The scheduled Computer runtime reconciler treats that as intent and uses the service-auth manager path to reconcile ECS desired count and provisioning.

<Aside type="note">
  GraphQL remains the user/operator product API. Provisioning ECS services, EFS
  access points, task definitions, and runtime desired count belongs to the
  internal Computer manager endpoint.
</Aside>

## Migration operations

Migration is exposed through the CLI:

```bash
thinkwork computer migration dry-run --tenant-id <tenant-id>
thinkwork computer migration apply --tenant-id <tenant-id> --confirm
```

Dry-run is the default operating mode. Apply refuses blocker groups such as multiple candidate Agents for one human, an existing conflicting Computer, or a source Agent backed by an Agent Template instead of a Computer Template.

## Runtime operations

The CLI is the first supported operator surface:

```bash
thinkwork computer runtime provision --tenant-id <tenant-id> --computer-id <computer-id>
thinkwork computer runtime status --tenant-id <tenant-id> --computer-id <computer-id>
thinkwork computer runtime restart --tenant-id <tenant-id> --computer-id <computer-id>
```

Operators can also enqueue Phase 3 smoke tasks:

```bash
thinkwork computer task enqueue \
  --tenant-id <tenant-id> \
  --computer-id <computer-id> \
  --type workspace_file_write \
  --path smoke/admin.txt \
  --content "admin smoke"
```

The Computer manager API remains internal service-auth:

```http
POST /api/computers/manager
Authorization: Bearer <API_AUTH_SECRET>

{
  "tenantId": "<tenant uuid>",
  "computerId": "<computer uuid>",
  "action": "provision"
}
```

Supported actions are `provision`, `start`, `stop`, `restart`, and `status`.

The task surface currently supports `health_check`, `workspace_file_write`, `google_cli_smoke`, and `connector_work`. `connector_work` is the connector handoff task used by the Linear checkpoint. For the Symphony proof, that task now delegates repo-backed coding work to a managed Agent and links the result back to the Computer-owned thread. Google Workspace OAuth-backed actions are still deferred; the smoke only verifies image tooling.

## Connector checkpoint

The first Computer-facing connector proof is Linear-only:

1. A Linear issue has the `symphony` label.
2. The Symphony connector is enabled and targets a Computer.
3. The connector runtime records a terminal connector execution.
4. The target Computer receives a `connector_work` task and `connector_work_received` event.
5. ThinkWork creates a connector thread assigned to that Computer.
6. The Computer delegates the coding turn, receives PR metadata, and the Symphony Runs tab shows the lifecycle.

Use [Run the Symphony Linear Checkpoint](/guides/symphony-linear-checkpoint/) for the normal operator walkthrough and `docs/runbooks/computer-first-linear-connector-checkpoint.md` for lower-level deploy verification and duplicate-check expectations.
