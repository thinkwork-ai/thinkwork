Webhooks are custom HTTP endpoints that external systems can POST to in order to trigger an agent turn or a workflow. They're blank slates — any system that can make an authenticated HTTP POST can become a ThinkWork trigger through a webhook.

## The list

**Route:** `/webhooks`  
**File:** `apps/web/src/routes/_authed/_tenant/webhooks/index.tsx`

A searchable `DataTable` with:

| Column           | Notes                                                                  |
| ---------------- | ---------------------------------------------------------------------- |
| Name             | Display name                                                           |
| Type             | Badge: **Agent** or **Workflow** depending on what the webhook targets |
| Target           | Agent or workflow name                                                 |
| Enabled          | Enabled / Disabled badge                                               |
| Last Invoked     | Relative timestamp                                                     |
| Invocation count | Lifetime count                                                         |
| Created          | Timestamp                                                              |

A **New Webhook** button opens the creation dialog.

## The detail page

**Route:** `/webhooks/:webhookId`  
**File:** `apps/web/src/routes/_authed/_tenant/webhooks/$webhookId.tsx`

### Header

- Webhook name
- Enable / disable toggle
- Delete with confirmation

### Token card

- The **bearer token** (masked by default with a show / copy button)
- The full invocation URL: `https://<api-endpoint>/webhooks/<webhookId>`

External systems call the URL with `Authorization: Bearer <token>` and a JSON body. The token is a random 32-character string generated at webhook creation and never rotated automatically.

### Configuration

Read-only display of:

- Target type (agent / workflow)
- Target name
- Rate limit (requests per minute)
- Prompt override (for agent targets — the string the agent receives when the webhook fires)

### Invocation log

An expandable table of run cards, one per invocation:

- Status — succeeded / failed / accepted with warning
- Started at / finished at
- Error or warning message
- Context snapshot — the JSON payload the webhook received, used for replay and debugging

## External usage

Once created, external systems trigger the webhook with a simple POST:

```bash
curl -X POST "https://<api-endpoint>/webhooks/<webhookId>" \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/json" \
  -d '{"message": "hello from external system", "context": {"foo": "bar"}}'
```

The backend:

1. Validates the token
2. Checks the rate limit (returns 429 if exceeded)
3. Creates or queues the target work
4. Writes a delivery row with the payload preview, response code, thread id when applicable, and any warning or error message
5. Returns a response once the target starts (not once it finishes)

The webhook is fire-and-forget from the caller's perspective. If the caller wants to know the result, they should read it from the created thread or poll the run status.

## Space webhooks

When a webhook is attached to a Space and targets an agent, each accepted POST creates a fresh Space thread. ThinkWork seeds that thread with a readable system trigger message before waking the tenant platform agent, and the agent receives the full structured webhook payload in turn context.

If the Space has an automatic thread-start workflow configured, the webhook-created thread starts that workflow just like a manually created thread. For example, a Customer Space configured for Customer Onboarding gets the onboarding goal, kickoff details, and task checklist during thread creation.

If thread creation succeeds but workflow initialization fails, the endpoint still returns an accepted response, usually HTTP 202, so external systems do not retry into duplicate threads. The delivery history shows this as **Accepted with warning** and stores the warning in the delivery detail.

## Data model

- **`webhooks`** — Aurora table with id, tenant_id, name, description, token, target_type (`agent` / `routine` compatibility value for Step Functions-backed workflows), agent_id / routine_id, prompt, config, enabled, rate_limit, last_invoked_at, invocation_count, created_at, updated_at
- **Runs** — `thread_turns` rows with `webhook_id` FK and a `context_snapshot` field storing the inbound JSON payload
- **Rate limit state** — in-memory per webhook (per instance, not distributed); resets on Lambda cold start

## Rate limiting

Each webhook has a per-minute rate limit (default 10). When the limit is exceeded, the backend returns HTTP 429 without invoking the target and without writing to the run log. The rate limit is a hard cap — there's no queue or back-pressure.

Because rate limit state is per-Lambda-instance, distributed invocations can collectively exceed the declared limit by the number of warm Lambda instances. For workflows that genuinely need hard caps, use a downstream queue rather than relying on the webhook's rate limit.

## Prompt overrides

For agent-targeted webhooks, the **prompt** field lets operators specify a canonical prompt that the agent receives when the webhook fires. The inbound JSON body is still available in `context_snapshot` and can be referenced by the agent's tools, but the prompt override controls what the agent "hears" when the turn starts.

This is how operators ship "when something happens, do X" workflows: the prompt override says "do X" and the inbound payload carries the context.

## Workflows

### Set up a custom Zapier trigger

1. Click **New Webhook**
2. Name it "Zapier → daily-summary agent"
3. Target type: agent
4. Target: the `daily-summary` agent
5. Rate limit: 10/min
6. Prompt override: "A new Zapier trigger fired. Review the payload and produce a summary."
7. Save
8. Copy the token and the URL
9. Paste into Zapier's webhook action
10. Test from Zapier — verify the invocation appears in the log

### Debug a failing webhook

1. Open the webhook detail page
2. Scroll to the invocation log
3. Expand the most recent failed row
4. Check the error message
5. Check `context_snapshot` to see what was actually received
6. Common causes: token mismatch (401), rate limit (429), target agent not found or disabled (500), payload parse error (400)

### Rotate a leaked token

<Aside type="caution">
The admin UI does not currently expose an in-place token rotation. If a token leaks, the remediation today is:
1. Disable the webhook
2. Delete it
3. Create a new webhook with the same target and prompt
4. Update the external system with the new token and URL

The webhook id changes, so the URL also changes. Plan for that in the external system's configuration.

</Aside>

## Known limits

- **No in-place token rotation.** Rotation is delete + recreate.
- **Per-instance rate limiting.** The rate limit is enforced per warm Lambda instance, not distributed. Distributed usage can collectively exceed the declared limit.
- **No queue.** Rate-limited requests are rejected, not queued.
- **Fire-and-forget.** The caller gets an immediate ack; the actual result lands on the thread created by the invocation. For Space webhooks, a 202 accepted-with-warning response means a thread exists but workflow initialization degraded; operators should inspect the delivery warning and thread rather than blindly retrying.
- **REST-only.** Webhooks don't flow through urql; the list refetches after each mutation.
