Routines are reusable multi-step workflows — sequences of agent invocations, tool calls, and conditional branches that run as a single coordinated unit. Where a scheduled job triggers a single agent turn, a routine triggers a whole composed workflow. Routines are backed by AWS Step Functions state machines so they can fan out, retry, branch, and survive long-running work.

## The list

**Route:** `/routines`  
**File:** `apps/admin/src/routes/_authed/_tenant/routines/index.tsx`

A searchable `DataTable` with columns:

| Column | Notes |
|---|---|
| Status | Active / Disabled / Archived |
| Name | Display name |
| Type | `workflow`, `agent_chain`, or `conditional` |
| Schedule | With a clock icon if the routine has any scheduled triggers |
| Agent | With a bot icon — the primary agent the routine hands work to |
| Last run | Relative timestamp |
| Next run | Estimated from attached scheduled jobs |

A **New Routine** button opens a creation dialog.

## The detail page

**Route:** `/routines/:routineId`  
**File:** `apps/admin/src/routes/_authed/_tenant/routines/$routineId.tsx`

The detail page has:

### Header

- Routine name and description
- Status badge
- Edit, delete, and enable/disable actions

### Sidebar

Routine metadata:

- Type
- Schedule (aggregated from attached scheduled jobs)
- Primary agent / team
- Last run, next run
- Created at

### Tabs

Two tabs:

- **Runs** — historical execution list with run id, start time, status, and a drill-in to see the step-by-step execution trace
- **Scheduled Jobs** — the scheduled-job triggers that fire this routine. Each row links back to the [Scheduled Jobs](/applications/admin/scheduled-jobs/) detail page.

## Routine types

| Type | Meaning |
|---|---|
| **`workflow`** | DAG-style step execution — each step runs in order, possibly with parallel branches, and the routine completes when all steps have terminal status |
| **`agent_chain`** | Chained agent invocations where the output of each agent becomes the input to the next; used for refine-and-handoff patterns |
| **`conditional`** | Steps gated on conditions (e.g. "if step 1 returned empty, run step 2") |

The type dictates the Step Functions state machine shape. A `workflow` routine maps to Parallel / Sequence states, `agent_chain` to sequential Task states, and `conditional` to Choice states.

## The authoring surface

<Aside type="note">
The routine builder — where the actual step definitions are authored — is **not fully visible in the current admin app routes**. Routines today are defined at the Terraform / CLI level as JSON blobs in the routine's `config` field, and the admin UI surfaces the metadata and run history but not the step editor. A visual routine builder is on the roadmap.
</Aside>

For now, the authoring workflow is:

1. Define the routine in code or Terraform with the intended step structure
2. Deploy via CLI or Terraform
3. Use the admin UI for:
   - Monitoring runs and triage
   - Enabling / disabling
   - Attaching scheduled jobs
   - Browsing run history and errors

## Run history

The Runs tab shows each historical execution of the routine with a status (`running`, `succeeded`, `failed`). Clicking a run expands to show:

- Per-step status and timing
- The Step Functions execution id (useful for debugging in CloudWatch)
- Error messages for any failed step
- The final output of the routine (where applicable)

Each run is also reflected as a row in `thread_turns` with a `routine_id` foreign key, so filtering threads by routine produces the same history.

## Data model

- **`routines`** — Aurora table with id, tenant_id, name, description, type, status (active / disabled / archived), schedule, config (JSON), agent_id, team_id, last_run_at, next_run_at, created_at
- **Run history** — `thread_turns` rows with `routine_id` FK; status progresses `queued → running → succeeded | failed`
- **Scheduled triggers** — `scheduled_jobs` rows with `routine_id` FK and `trigger_type in (routine_schedule, routine_one_time)` (many-to-one — a single routine can have many schedules)
- **Step Functions state machines** — one per routine, named by routine slug, provisioned through the Terraform module

## Scheduled jobs vs routines — which to use

The short version:

- Single agent call on a timer → **scheduled job** with type `agent_scheduled`
- Multi-step workflow → **routine** (possibly also attached to a scheduled job)
- Event-driven multi-step workflow → **routine** attached to a **webhook** (see [Webhooks](/applications/admin/webhooks/))

One routine can have multiple triggers. A scheduled job triggers exactly one target (agent or routine). Keep them separated in your mental model.

## Workflows

### Set up a morning triage workflow

1. Define a routine with type `workflow` in Terraform / CLI: step 1 = "fetch overnight mentions", step 2 = "categorize", step 3 = "post digest"
2. Deploy the routine
3. Open `/routines`, click into the new routine
4. Go to the Scheduled Jobs tab
5. Attach a `routine_schedule` scheduled job firing at 08:00 weekdays
6. Enable the scheduled job
7. Watch the Runs tab for the next morning's execution

### Debug a failing routine

1. Open the routine detail page
2. Go to the Runs tab
3. Expand the failed run
4. Read the per-step status to find which step failed
5. Read the error message for that step
6. Cross-reference with the Step Functions execution id in CloudWatch for the full state machine trace
7. Fix the routine code or the underlying agent, redeploy, and rerun

### Manually trigger a routine

1. Find or create a scheduled job of type `routine_one_time` targeting the routine
2. Use **Run Now** on that scheduled job
3. Or fire a manual invocation via webhook if the routine is attached to one

## Known limits

- **No in-UI routine editor.** Routine step definitions live in code / Terraform today; the admin UI displays metadata and runs but not the DAG.
- **No per-step retry from the UI.** If a step failed, the retry is at the whole-routine level. Step-level retry requires retriggering the whole routine.
- **Run history is immutable.** Past runs can be browsed but not edited or deleted from the UI.
- **Step Functions execution ids are the debugging primary key.** For deep traces, the admin UI surfaces the execution id but the operator has to open CloudWatch for the full step-by-step view.
