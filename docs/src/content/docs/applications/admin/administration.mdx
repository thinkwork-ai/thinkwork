The Administration surface covers the housekeeping corners of the admin app: tenant-level users, tenant configuration, the underlying deployment status, automations, routines, and the org chart.

## Humans (tenant users)

**Route:** `/humans`

The Humans page lists every user in the tenant with their role, status (invited / active / disabled), and last-active timestamp. Operators can:

- **Invite** a new user by email — the invite flow sends a Cognito invite and records the pending user in the tenant.
- **Promote or demote** a user — change their role between operator, author, viewer, or custom roles defined for the tenant.
- **Deactivate** a user — revoke their access without deleting the record (so audit history stays attributable).

Invited users complete sign-up through Cognito and land in the admin app's `/sign-in` flow on first login.

<Aside type="note">
The Humans page is tenant-scoped. Cross-tenant user administration (a single user with access to multiple tenants) is handled by switching tenants in the top bar, not by a global user table.
</Aside>

## Settings (tenant configuration)

**Route:** `/settings`

The Settings page is the canonical read-out of the tenant's configuration and the underlying deployment. Most of it is read-only — configuration lives in Terraform and the admin app reflects what's actually deployed.

Sections:

**Tenant identity** — name, slug, plan, and the issue counter (a monotonic counter used for tenant-scoped sequential IDs like thread numbers).

**Deployment status** — stage (dev / staging / prod), region, whether managed memory is enabled, whether the Hindsight add-on is enabled, CLI version, and Terraform module version.

**Resource URLs** — the public-ish URLs the tenant uses day-to-day: S3 bucket names, Aurora endpoint, ECR registry, API Gateway URL, AppSync URL, admin app URL, docs URL. These are convenient links for operators diagnosing something directly in AWS.

**Features and flags** — tenant-level feature flags the admin app respects (for example, beta features being tried out on a single tenant before rolling out).

To change anything under Settings that isn't a simple toggle, edit the Terraform configuration for the tenant and re-deploy with `thinkwork deploy` (see [CLI](/applications/cli/)). The admin app re-reads state on next load.

## Automations and routines

Two related but distinct surfaces:

### Automations

**Route:** `/scheduled-jobs`

Tenant-level scheduled automations. Each automation has:

- A **trigger** — cron schedule, event filter, or webhook
- An **agent or routine** to invoke when the trigger fires
- **Enable / disable** state
- **Invocation history** with success / failure counts

These are distinct from per-agent scheduled jobs (which live under `/agents/:agentId/scheduled-jobs`) — automations here are tenant-level workflows that can span multiple agents or don't belong to any single agent.

### Routines

**Route:** `/routines`

Routines are scheduled, multi-step jobs that compose several agent invocations into a single recurring workflow. Think: "every morning at 9am, run the sales-summary agent, then hand the output to the briefing-agent, then post to Slack."

Routines are implemented on top of Step Functions and have richer state (per-step success / failure, retry policy) than a single scheduled-jobs row. They're the right tool when a single cron entry can't capture the workflow.

See [Automations (concept)](/concepts/automations/) and [Scheduled and Event-driven](/concepts/automations/scheduled-and-event-driven/) for the underlying model, and [Routines and Execution Model](/concepts/automations/routines-and-execution-model/) for the routine pattern specifically.

## Org chart

**Route:** `/org`

The Org Chart is a read-only view of the tenant's agents and teams. It visualizes:

- Which agents are in the tenant
- Which teams each agent belongs to
- The delegation relationships between parent agents and sub-agents

This is deliberately read-only in v1 — team creation and assignment happen through agent templates and the tenant configuration, not through a direct org-chart UI. The Org Chart page is the "at a glance" view for operators who want to see the whole shape of the tenant without clicking through every agent.

## Artifacts

**Route:** `/artifacts`

Artifacts are structured, thread-attached outputs that agents produce during a turn: reports, notes, plans, drafts, and digests. The Artifacts page gives operators a tenant-wide view of those outputs, filterable by type, agent, and status.

Unlike messages — which are conversational — artifacts are durable, named outputs meant to outlive the thread that produced them. Clicking an artifact opens a detail modal with the full content and a link back to the thread that produced it.

## Agent invites

**Route:** `/agents/invites`

Agent invites are a mechanism for inviting an external agent (or external user acting as an agent) into the tenant. An invite generates a public token; the invitee hits a specific `/invite/:token` URL, walks through adapter selection and API key claim, and joins the tenant as a registered agent.

This is how a BYO agent runtime outside AWS registers itself with a ThinkWork tenant without an operator having to provision it directly.
