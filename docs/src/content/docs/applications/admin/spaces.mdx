Spaces are the primary way operators shape how the tenant platform agent behaves in a particular context. A Space can represent a team, customer, project, workflow, inbox, or channel-bound workroom.

**List route:** `/spaces`  
**Detail routes:** `/spaces/:spaceId/configuration`, `/spaces/:spaceId/workspace`, `/spaces/:spaceId/tools`, `/spaces/:spaceId/memory`, `/spaces/:spaceId/automations`

The Spaces list shows each Space's name, access mode, status, and last update time. The header includes:

- **Configure agent** — opens [Tenant Agent](/applications/admin/agents/) for tenant-wide defaults.
- **New Space** — creates a Space with name, description, and access mode.

## Space configuration

The Configuration tab controls the Space's identity and runtime behavior.

| Field          | Meaning                                                                           |
| -------------- | --------------------------------------------------------------------------------- |
| Name           | Human-readable Space name                                                         |
| Description    | Optional context shown to operators and used to orient the workroom               |
| Access mode    | `Public` or `Private`                                                             |
| Email triggers | Whether registered users can start new cold-contact threads by emailing the Space |

Public Spaces can be used by registered tenant users. Private Spaces restrict access to Space members.

### Runtime overrides

The Advanced runtime section lets a Space override the tenant agent baseline for work that happens in that Space.

| Override             | Behavior when empty                          |
| -------------------- | -------------------------------------------- |
| Model                | Inherits the tenant agent model              |
| Guardrail id         | Inherits the tenant agent guardrail behavior |
| Monthly budget cents | Inherits the tenant agent monthly budget     |
| Budget paused        | Uses the tenant agent budget state           |
| Sandbox              | Inherits the tenant agent sandbox setting    |

Use Space overrides when a workroom needs a different risk or cost profile. Examples: a finance Space can use a stricter guardrail, an experimentation Space can use a cheaper model, or an internal compute Space can disable public-network sandbox access.

<Aside type="tip">
  Keep tenant-wide defaults broad and conservative. Use Space overrides for
  local policy, cost, and model differences so the rest of the tenant remains
  easy to reason about.
</Aside>

## Workspace

The Workspace tab holds files that apply only inside this Space. Use it for procedures, customer context, team norms, runbooks, and domain notes that should not leak into other Spaces.

The tenant agent reads its own baseline workspace first, then Space context is rendered for the current turn. This gives you a shared platform-agent identity with local Space behavior.

## Tools

The Tools tab manages which built-in tools and MCP servers are available in the Space. Tenant-level registrations still live in [Tenant MCP Servers](/applications/admin/mcp-servers/) and [Built-in Tools](/applications/admin/builtin-tools/); the Space controls which of those capabilities are available for that workroom.

## Memory

The Memory tab shows memory scoped to the Space. Use it to inspect what the agent has learned or retained for that workroom without mixing unrelated tenant context.

## Automations

The Automations tab lists recurring or event-driven work tied to the Space. Use it when a workflow should wake the tenant agent in a specific context rather than at a tenant-wide default.

## Email triggers

Every Space has an address:

```text
<tenant-slug>.<space-slug>@agents.thinkwork.ai
```

Cold-contact inbound email follows these rules:

- Email triggers must be enabled on the Space.
- Archived Spaces cannot receive new cold-contact email.
- Public Spaces accept cold-contact email from registered users in the tenant.
- Private Spaces accept cold-contact email only from Space members.
- Token-bearing replies to agent-initiated emails work even when cold-contact triggers are disabled.

Legacy per-agent addresses such as `<agent-slug>@agents.thinkwork.ai` are retired. Send new inbound work to a Space address.

## Common workflows

### Create a Space

1. Open `/spaces`.
2. Click **New Space**.
3. Enter a name and optional description.
4. Choose Public or Private.
5. Save, then configure workspace files, tools, memory, automations, and runtime overrides as needed.

### Give a Space a stricter runtime profile

1. Open the Space's Configuration tab.
2. Expand Advanced runtime.
3. Set the model, guardrail id, budget, paused state, or sandbox value that should differ from the tenant agent.
4. Save overrides.
5. Leave fields empty when the Space should inherit the tenant agent baseline.

### Start work by email

1. Open the Space's Configuration tab.
2. Enable Email Triggers.
3. Copy the Space email address.
4. Share it with registered tenant users, or with Space members for a Private Space.
