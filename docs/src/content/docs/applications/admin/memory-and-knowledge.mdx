The Memory and Knowledge surface is where operators and agent authors manage the two context layers that surround an agent: **long-term memory** (what the agent remembers from prior turns) and **knowledge bases** (what the agent can retrieve from documents).

## Memory

**Route:** `/memory`

The Memory page is a browser and editor for memory records stored in Aurora. It covers records produced by the default AgentCore managed memory strategies (semantic, preferences, summaries, episodes) and, when enabled, records produced by the Hindsight add-on.

The page has two main modes:

**Graph view** — a 3D force-graph visualization of memory records, grouped by agent and topic. Useful for spotting clusters, dense topics, and orphaned records. Pan, zoom, and click a node to open its record.

**List / search view** — a flat, searchable list. Queries run against `MemoryRecordsQuery` and `MemorySearchQuery` with real-time subscriptions for live updates.

### Editing memory records

Memory records use a lightweight XML-topic format so records can carry multiple topics without exploding the schema:

```xml
<topic name="preferences">User prefers short, bulleted responses.</topic>
<topic name="identity">User's name is Alice; she works in the risk team.</topic>
```

The admin parser reads this format and displays each topic as its own row. Operators can:

- **Edit** a record's content or topic tags
- **Delete** a record (soft-delete with retention policy)
- **Add** a new record manually (useful for seeding an agent with a canonical "about the user" entry)
- **Search** across records by content or topic

<Aside type="note">
Memory records are per-agent by default. Editing or deleting a record affects only that agent's future turns. The underlying AgentCore memory strategies and Hindsight pipeline continue to emit new records from live agent activity — manual edits coexist with automatic ones.
</Aside>

### System defaults

A small "Config" section of the Memory page exposes the tenant's memory defaults: which strategies are enabled, retention thresholds, and whether Hindsight is wired up. Most tenants leave these at defaults; advanced tenants tune them.

See [Memory](/concepts/knowledge/) for the concept model, [Long-term Memory](/concepts/knowledge/memory/) for the lifecycle, and [Retrieval and Context](/concepts/knowledge/retrieval-and-context/) for how memory plugs into each turn.

## Knowledge bases

**Route:** `/knowledge-bases`

Knowledge bases are Bedrock Knowledge Bases backed by Aurora `pgvector`. The admin UI lets operators:

- **Create** a new knowledge base, naming it and pointing it at an S3 bucket of source documents
- **Upload** documents to the source bucket directly from the admin UI
- **Trigger** a sync (the Bedrock ingestion job that re-indexes the KB)
- **See** last-sync status, document counts, and any ingestion errors

Detail pages at `/knowledge-bases/:kbId` show the document list, retrieval settings (chunk size, overlap, similarity threshold), and the list of agents that have the KB attached.

### Assigning a knowledge base to an agent

Knowledge bases are not automatically attached to every agent. An operator assigns a KB to an agent (or to an agent template, which flows down) from the agent's **Knowledge** tab under [Agents & Templates](/applications/admin/agents/). Retrieval is scoped per-agent: an agent only retrieves from the knowledge bases it has been explicitly given access to.

### Document upload

For small-scale document ingestion, the admin UI has an upload widget that puts the file directly into the knowledge base's S3 bucket and kicks off a sync. For larger ingestion jobs (thousands of documents, automated pipelines), use the CLI or an external ETL job that writes to S3 and calls the sync API.

<Aside type="note">
Because the vector index lives in Aurora (`pgvector`), an Aurora snapshot restore brings back agents, threads, memories, **and** the KB vector index in one shot. The only KB state not in Aurora is the source documents themselves in S3, which have versioning enabled.
</Aside>

## The split: memory vs knowledge

It's worth keeping these straight:

- **Memory** is stateful, agent-produced, and evolves over time. It answers "what does this agent remember about this user and the work so far?"
- **Knowledge** is stateless, operator-curated, and indexed from source documents. It answers "what reference material does this agent have access to?"

Both get assembled into a turn's context, but they come from different places and are managed differently. Memory is mostly automatic; knowledge is mostly manual.
