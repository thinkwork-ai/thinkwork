The Skills Catalog is where tenant skill packs are browsed, imported, exported, and inspected. Skills are reusable capability bundles that agents inherit — a skill can add tool definitions, background knowledge, or behavior instructions, and any agent that's been assigned the skill inherits it at invocation time.

Tenant skill source is stored in the catalog under one folder per skill slug. Installed agent or template copies are separate pinned workspace files under `skills/<slug>/`; catalog edits and imports do not mutate those live copies until an operator applies an update.

## The list view

**Route:** `/settings/skills`

**File:** `apps/web/src/components/settings/SettingsSkills.tsx`

The list is an operator-only `DataTable` backed by the tenant catalog index. Each row shows:

- Name
- Description
- Category
- Tags
- Latest eval score, when the skill has a linked eval dataset

The import action opens a ZIP picker. A valid archive contains exactly one skill folder, or files at the archive root with a slug declared in `SKILL.md` frontmatter. Root-level files are normalized into the declared skill folder during import.

Clicking a row navigates to the skill detail view.

### Underlying API

The settings surface talks to the workspace files API:

| Endpoint                                                                                        | Purpose                                              |
| ----------------------------------------------------------------------------------------------- | ---------------------------------------------------- |
| `POST /api/workspaces/files` with `action: "list"` and `summary: true`                          | Read indexed skill summaries                         |
| `POST /api/workspaces/files` with `action: "import-skill"`                                      | Validate and import a skill ZIP archive              |
| `POST /api/workspaces/files` with `action: "export-skill"`                                      | Export one catalog skill as an import-compatible ZIP |
| `POST /api/workspaces/files` with `action: "get"`, `"put"`, `"delete"`, `"rename"`, or `"move"` | Read and edit files in one catalog skill folder      |

Every action is admin-gated server-side and resolves the tenant from the authenticated request context.

### Importing a skill archive

The importer accepts ZIP archives that follow the Agent Skills package shape:

- `SKILL.md` is required.
- The skill slug must be lowercase kebab-case and must match either the single top-level folder name or the `slug` frontmatter value for root-level archives.
- `WIRING.md` is optional. If it is missing, ThinkWork generates default wiring during import so the skill can still be inspected and exported consistently.
- Supporting files such as `scripts/`, `references/`, `assets/`, and `evals/` are preserved.

If an archive targets an existing skill, the UI asks for replacement confirmation. Replacement updates the catalog source and linked eval data, but installed agent copies keep running the pinned version until an operator applies the update from the skill detail page.

## The skill detail view

**Route:** `/settings/skills/:skillSlug`

**File:** `apps/web/src/components/settings/SettingsSkillDetail.tsx`

The detail view is a full file-tree browser plus editor, sized for real skill authoring work:

- **Header** — skill slug, back breadcrumb, and export action
- **Eval panel** — current score, regression badge, run action, candidate review link, and update apply/override controls
- **File tree** — collapsible folder structure on the left, backed by the catalog-scoped workspace files client
- **Editor pane** — the workspace editor opens `SKILL.md` by default and can inspect generated `WIRING.md` plus supporting files
- **Toolbar** — file actions provided by the shared workspace editor

The export action downloads `<slug>.zip`. Exported archives are compatible with the same import path and include the exact catalog source files, including generated wiring when the importer created it.

### Files and update actions

Operators can edit catalog files through the workspace editor. Those edits change catalog source only. They become active for an installed agent or template after the operator applies the skill update, subject to the eval gate shown on the detail page.

Destructive file actions use the workspace editor's confirmation flows and do not have undo.

## Authoring workflow

Author multi-file skills in a local editor, package the skill folder as a ZIP, and import it through `/settings/skills`. Small edits can happen directly in the detail page after import. Exporting the skill later produces an archive with the same source shape, so it can be moved between tenants or round-tripped through local editing.

## Distinction: workspace skills vs the Skills Catalog

- **Skills Catalog** (this page) — browse, import, export, and edit skill source at the **tenant** level. Skills here exist whether or not any agent uses them.
- **Agent or template Workspace tab** — materialize an editable skill under `workspace/skills/<slug>/SKILL.md` so it can affect a real invocation. See [Agents → Workspace](/applications/web/agents/#workspace).

A workspace skill goes through two steps before it affects a real invocation: import or edit it at the tenant/catalog level, then add or update it in an agent or template workspace.

Built-in tools do not follow this path. Tools such as `web_search` and `send_email` are injected by runtime configuration and should not be copied into `workspace/skills/`.

## Known limits

- **Single-skill archives only.** Import expects one skill per ZIP.
- **Dependency resolution is manual.** If a skill references another skill or MCP server, install or configure that dependency separately.
- **No archival.** Deleted skills are hard-deleted. There's no undo.
- **Replacement is explicit.** Importing over an existing skill requires confirmation and updates catalog source, not already-installed workspace copies.
