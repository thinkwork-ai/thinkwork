The Knowledge page is the operator hub for the context an agent can use. It combines the previous tenant-wide Memory, Wiki Pages, and Knowledge Bases surfaces under one tabbed route, then adds Context Engine configuration and test tooling.

**Route:** `/knowledge`  
**Tabs:** Memory, Wiki, Knowledge Bases, Context Engine

## Why It Exists

Agents can draw context from several sources:

- durable memory, including Hindsight-backed memory when enabled;
- compiled wiki pages from Compounding Memory;
- Bedrock Knowledge Bases attached to agents;
- workspace files scoped to an agent;
- approved MCP context tools.

Before the Knowledge page, operators had to inspect these sources across multiple sidebar pages and then reason separately about what Context Engine would query. The Knowledge page keeps source inspection and Context Engine routing close together without merging the underlying stores.

## Tabs

| Tab | Purpose |
|---|---|
| Memory | Browse and search retained memory records; view the Hindsight memory graph when available |
| Wiki | Inspect compiled wiki pages and graph/search output |
| Knowledge Bases | Manage Bedrock KBs, document uploads, and sync status |
| Context Engine | Configure and verify the provider layer used by `query_context` |

The old standalone routes should redirect into the matching tab:

| Old route | New route |
|---|---|
| `/memory` | `/knowledge/memory` |
| `/wiki` | `/knowledge/wiki` |
| `/knowledge-bases` | `/knowledge/knowledge-bases` |

## Context Engine Test Harness

The Context Engine tab includes a safe query harness for operator verification. It is intended for dogfooding provider behavior before testing through a full agent chat.

The harness controls are:

- **Adapters** — multi-select dropdown for provider families such as Hindsight Memory, Compounding Wiki, Workspace Files, and Bedrock Knowledge Bases.
- **Agent target** — optional agent picker used when workspace files or agent-scoped attachments should participate in the test.
- **Hindsight strategy** — `recall` for direct memory-record retrieval or `reflect` for synthesized memory output.
- **Search** — full-width natural-language query field.

Results show provider statuses and top hits side by side. Clicking a hit opens a formatted markdown preview plus the raw structured hit. Clicking **View full result** opens the complete `query_context` response. Provider failures should be local: a skipped Bedrock KB or slow Hindsight call should not hide successful Wiki or Workspace hits.

## Template Configuration

Agent Templates expose Context Engine as a built-in tool alongside Code Sandbox, Browser Automation, Web Search, and Send Email. Enabling it injects the platform-owned `query_context` tool into supported runtimes. Template configuration can further choose which tenant-approved adapters participate by default and can expose adapter-specific options such as Hindsight strategy.

Normal agent configuration should show Hindsight as a Context Engine adapter, not as a separate peer tool. Raw Hindsight tools may still exist for diagnostics, but the operator-facing path is Context Engine.

## Operational Guidance

- Use the Context Engine tab first when debugging provider behavior. It gives provider-level latency, hit counts, skipped states, and partial failures without model/tool-selection noise.
- Select an agent target when testing Workspace Files. Workspace search needs the agent workspace scope to find files such as `USER.md`.
- Use `reflect` for answer-like memory verification and `recall` when debugging raw memory retrieval.
- Keep mobile Wiki search on the dedicated Wiki path unless a future mobile Context Search screen intentionally replaces it.
