Kestra is an optional self-hosted orchestration runtime that ThinkWork deploys
inside the customer's AWS account. The intended operating model is that users
stay in ThinkWork while agents create, validate, run, and inspect Kestra flows
through a curated control MCP surface.

- **Primary route:** Settings -> Applications -> Kestra
- **Managed app key:** `kestra`
- **Control MCP endpoint:** `/mcp/kestra`

<Aside type="note">
  Kestra is infrastructure, not an end-user connector. Deploy, park, destroy,
  and MCP repair actions always go through the normal ThinkWork deployment
  pipeline and managed-app evidence flow.
</Aside>

## Operator Surfaces

| Surface                                     | Use it for                                                                     |
| ------------------------------------------- | ------------------------------------------------------------------------------ |
| Settings -> General -> Managed Applications | Plan, approve, deploy, park, destroy, and inspect managed-app job evidence     |
| Settings -> Applications -> Kestra          | Inspect runtime status, endpoint metadata, health, and managed MCP repair path |
| Settings -> MCP Servers                     | Confirm the system-managed Kestra Control row and assignment state             |

Operators should not create a second manual MCP row for the same customer
Kestra instance. If the managed row is missing, use the Kestra settings page to
install or repair it.

## Lifecycle

| Action  | Result                                                                                                                          |
| ------- | ------------------------------------------------------------------------------------------------------------------------------- |
| Deploy  | Provisions the Kestra runtime, public endpoint, database state, internal storage, secrets, and MCP path                         |
| Park    | Stops runtime capacity while retaining flows, execution history, storage, secrets, endpoint wiring, and MCP continuity          |
| Destroy | Removes runtime infrastructure, flow data, execution history, app-owned secrets, storage, database state, and MCP configuration |

Parking is the reversible cost-control state. Destroy is the data-removal
state; use it only when orchestration data and history are no longer needed.

## Runtime Contract

The v1 Kestra deployment is intentionally narrow and AWS-native:

- ECS/Fargate standalone Kestra service;
- public HTTPS Application Load Balancer endpoint;
- dedicated PostgreSQL database and least-privilege role;
- encrypted S3 internal storage;
- Secrets Manager credentials for database access and Kestra UI/API basic auth;
- CloudWatch logs and target-group health checks.

Supported v1 flow classes are Fargate-safe orchestration flows: HTTP calls, API
coordination, validation, schedule/control-plane work, and other tasks that run
inside the managed Kestra service without host access.

Unsupported v1 execution classes include Docker socket tasks, Docker-in-Docker,
privileged containers, arbitrary host/container execution, and EC2 worker pools.
The control MCP rejects known unsafe Docker/host task classes before a flow is
upserted.

## Control MCP

ThinkWork reconciles a system-managed MCP server named **Kestra Control** when
Kestra is retained. The MCP row uses tenant API key authentication against the
ThinkWork API, but both the MCP bearer token and the Kestra service credential
live in Secrets Manager. The database stores secret references and runtime
metadata, not plaintext credentials.

The row is assigned to the tenant platform default agent so agents can use the
curated tool surface:

| Tool                      | Purpose                                           |
| ------------------------- | ------------------------------------------------- |
| `kestra_namespaces_list`  | List namespaces visible to the service credential |
| `kestra_flows_get`        | Read a flow by namespace and id                   |
| `kestra_flows_validate`   | Validate a flow and ThinkWork runtime policy      |
| `kestra_flows_upsert`     | Create or replace an allowed ThinkWork flow       |
| `kestra_executions_start` | Start a flow execution                            |
| `kestra_executions_get`   | Inspect execution state                           |
| `kestra_executions_logs`  | Read execution logs                               |
| `kestra_plugins_search`   | Search Kestra plugin metadata                     |

By default, writes are limited to the `thinkwork` namespace prefix. Keep that
prefix unless a follow-up design extends the tenancy policy.

## Catalog MCP Is Different

Public Kestra catalog/docs MCP servers can help agents discover plugins,
blueprints, or examples. They are read-only authoring context. They do not
authenticate to the customer Kestra instance, cannot run customer flows, and do
not replace the ThinkWork-managed `/mcp/kestra` control surface.

Use catalog context to draft an orchestration idea. Use Kestra Control to
validate, upsert, execute, and inspect the managed runtime.

## Verification

Run the managed-app smoke after deploy, park, redeploy, and destroy:

```sh
node scripts/smoke/kestra-managed-app-smoke.mjs

SMOKE_ENABLE_KESTRA_MANAGED_APP=1 \
  SMOKE_TENANT_ID=<tenant-id> \
  SMOKE_EVIDENCE_FILE=deploy-artifacts/kestra-smoke.json \
  node scripts/smoke/kestra-managed-app-smoke.mjs
```

Passing live mode means unprovisioned or parked stages skip explicitly, and
running stages expose an HTTPS endpoint that returns success or an
authentication challenge.

Run the control MCP smoke after the runtime is healthy and the MCP row is
installed:

```sh
node scripts/smoke/kestra-control-mcp-smoke.mjs

SMOKE_ENABLE_KESTRA_CONTROL_MCP=1 \
  SMOKE_API_BASE_URL=<api-url> \
  SMOKE_KESTRA_MCP_BEARER=<tenant-control-mcp-token-or-service-secret> \
  SMOKE_TENANT_ID=<tenant-id> \
  SMOKE_EVIDENCE_FILE=deploy-artifacts/kestra-control-mcp-smoke.json \
  node scripts/smoke/kestra-control-mcp-smoke.mjs
```

By default, live control-MCP mode validates the curated tool surface, confirms a
safe flow is accepted, confirms an unsafe Docker task is rejected, upserts a
safe `thinkwork.smoke` flow, starts one execution, polls it to completion, and
records a logs preview. Set `SMOKE_KESTRA_CONTROL_MUTATE=0` to keep the live
run to initialization, tool listing, and validation-only calls.

## Recovery

If Kestra is running but agents cannot see the tools, check Settings -> MCP
Servers for the **Kestra Control** system-managed row. If it is missing or
stale, open Settings -> Applications -> Kestra and run the install/repair
action. Then rerun the control MCP smoke.

If the runtime is parked, the managed MCP row may remain assigned but calls
should not be treated as healthy until Kestra is deployed again and the smokes
pass.

After destructive destroy, the expected state is no retained Kestra resources,
no app-owned Kestra secrets, no Kestra database state, no S3 internal-storage
objects, and no Kestra Control MCP row or token material.
