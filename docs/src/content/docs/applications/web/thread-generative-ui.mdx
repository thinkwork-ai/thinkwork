Thread Generative UI renders agent-produced interface fragments inside web Thread conversations. The v1 surface consumes bounded `data-json-render` parts that carry complete upstream json-render specs, renders supported catalog components inline on web, and gives mobile the same part with a required fallback summary.

## Scope

Thread GenUI is for ephemeral inline work inside a conversation. It can show a review card, workflow status, key-value list, action form, or analytical display returned by the agent for the current turn.

Durable dashboards, saved reports, refresh schedules, sharing, and dashboard layout ownership belong to the Analytics/Dashboard foundation, not to Thread GenUI.

## Envelope

Persisted Thread GenUI parts use the canonical `data-json-render` envelope:

```json
{
  "type": "data-json-render",
  "id": "json-render:example",
  "data": {
    "schemaVersion": "thread-json-render/v1",
    "catalogVersion": "thread-json-render-catalog/v1",
    "status": "ready",
    "spec": {
      "root": "root",
      "elements": {
        "root": {
          "type": "task.review",
          "props": {
            "title": "Review task",
            "summary": "Confirm the task is ready.",
            "status": "pending"
          },
          "children": []
        }
      }
    },
    "mobileFallback": {
      "title": "Review task",
      "summary": "Confirm the task is ready.",
      "lines": ["Status: pending"]
    },
    "specHash": "json-render-fnv1a:..."
  }
}
```

The `mobileFallback` object is required. It is the compatibility contract for native mobile and any other consumer that cannot render the full web catalog.

Reviewable fixtures are checked in at `docs/fixtures/thread-json-render/`. Use
`valid-card.json` for a known-good primitive card, and use
`invalid-legacy-component.json` plus `invalid-fenced-markdown.md` to verify that
legacy objects and markdown fences are not trusted as generated UI.

## Catalog Ownership

Thread GenUI uses a two-layer catalog. Layer one is upstream
`@json-render/shadcn`; layer two contains ThinkWork domain components used in
conversation:

- `task.review`
- `workflow.status`
- `keyValue.list`
- `form.action`

Analytical charts and tables use the reserved `analytics.display` adapter from `@thinkwork/analytics-display`. Thread GenUI does not define a parallel chart or table catalog. Agents that need inline charts emit an `analytics.display` json-render component inside the same `data-json-render` envelope.

## Rendering

The web Thread renderer accepts persisted message `parts` and validates `data-json-render` through the shared `@thinkwork/thread-json-render` validator before rendering. Unsupported or malformed parts fail closed to the fallback copy rather than executing arbitrary payloads.

The renderer does not scan assistant markdown for UI payloads. A model must use
the platform-owned `emit_json_render_ui` tool so the runtime and API can validate
the complete spec before persistence.

Component props must be bounded data, not code. References to external dashboards or datasets are rejected for inline analytical payloads; the inline part must include the bounded render payload needed for the current Thread view.

## Actions

Actions are declared by the envelope and executed by server-side resolvers. Clients send the action id and constrained params back to the API; the server validates thread visibility, source message, source part, action registration, and rate limits before invoking any effect.

Clients must not treat payload-provided tool names, URLs, or arbitrary params as executable authority.

Durable action submissions go through `handleJsonRenderAction`, not legacy GenUI
mutation names. The server reloads the source message and source part, checks
the current `specHash`, validates the action descriptor, and only then appends a
normal follow-up Thread message.

## Promotion

Ready `data-json-render` parts can be promoted to durable `DATA_VIEW` artifacts. Promotion creates a snapshot of the validated source part and records source thread/message/part metadata, spec hash, status, title, summary, provenance, freshness, diagnostics, and sensitivity metadata where present.

Promotion does not create dashboard persistence, refresh scheduling, or sharing. Those workflows remain separate durable analytics work.

## Mobile Compatibility

Mobile receives the same persisted message `parts` but v1 does not implement the full web renderer. It renders the required `mobileFallback` title, summary, and lines. If a malformed `data-json-render` part is present, mobile shows a stable unsupported state instead of blanking the message or crashing. Legacy `data-genui` payloads are not converted into json-render specs.
