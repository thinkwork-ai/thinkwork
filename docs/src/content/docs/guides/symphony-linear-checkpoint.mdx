The Symphony Linear checkpoint proves that a real Linear issue can enter ThinkWork through a scheduled connector, become Computer-owned work, produce a draft PR, and write status back to Linear without manually invoking the poller.

Use this guide for the normal operator path. Use `docs/runbooks/computer-first-linear-connector-checkpoint.md` when you need SQL fallback, stale-row cleanup, or lower-level scheduler checks.

## What this proves

A passing checkpoint proves exactly this path:

```text
Linear issue with the symphony label
    -> scheduled connector poller
    -> terminal connector execution
    -> completed connector_work Computer task
    -> connector_work_received event
    -> Computer-owned thread
    -> completed delegation
    -> succeeded managed-Agent turn
    -> draft PR
    -> Linear In Review writeback
    -> Symphony Runs lifecycle row
```

It does not prove Slack, Jira, GitHub Issues, Salesforce, Zendesk, automatic actor matching, or custom connector authoring.

## Entry criteria

Before creating a Linear issue, confirm:

- the deployed stack includes the current connector poller, Computer runtime, Symphony UI, and PR-producing connector work harness;
- the target tenant has an active Computer;
- the tenant has an active Linear credential, usually slug `linear`;
- the tenant has an active GitHub credential, usually slug `github`;
- the Symphony connector is active, enabled, type `linear_tracker`, and targets a Computer;
- the connector label filter is `symphony`;
- the scheduled connector poller is enabled;
- no old Symphony process is also watching the same Linear project/label.

<Aside type="caution">
  The issue used for this proof must be fresh and must have only the `symphony`
  label for connector pickup. Do not use `symphony-eligible`.
</Aside>

## Configure the connector

1. Open **Admin -> Symphony -> Connectors**.

2. Edit the Linear Symphony connector.

3. Confirm the structured Linear fields:
   - team key matches the Linear team, such as `TECH`;
   - label is `symphony`;
   - Linear credential is active;
   - writeback states are `In Progress` for dispatch and `In Review` for PR opened.

4. Confirm the target is a Computer, not a direct managed Agent.

5. Confirm **GitHub PR setup**:
   - active GitHub credential;
   - owner;
   - repository;
   - base branch;
   - checkpoint file path.

6. Save without editing Advanced JSON.

If the connector shows **GitHub setup required**, create or reactivate the tenant GitHub credential and save the connector again. The checkpoint should not proceed while that warning is visible.

## Create the Linear issue

Create one real Linear issue in the configured team/project. Use a small safe task such as:

> Add a tiny note to `README.md` or `CHANGELOG.md` proving the Symphony connector can open a draft PR.

Apply only the `symphony` label. Leave the issue in Todo and wait for unattended pickup.

## Verify Linear

After the scheduled poller runs:

1. The issue should move from Todo to `In Progress`.
2. Linear should receive exactly one "Symphony agent is now working..." comment with a branch name.
3. A draft PR should open in GitHub.
4. Linear should receive exactly one "Symphony agent opened a draft PR..." comment.
5. The issue should move to `In Review` only after the draft PR exists.

Duplicate pickup comments, duplicate PRs, or a card stuck in Todo/In Progress means the checkpoint failed and needs debugging.

## Verify Symphony Runs

Open **Admin -> Symphony -> Runs** and find the row for the Linear issue key.

The row should show:

- state `Terminal`;
- completed `connector_work` task;
- completed delegation;
- succeeded thread turn;
- thread link;
- PR link;
- writeback `Linear: In Review`.

Rows should stay single-line and should not require horizontal scrolling for the checkpoint signals.

## If it fails

Start with the visible surfaces:

| Symptom                      | First check                                                      |
| ---------------------------- | ---------------------------------------------------------------- |
| No Runs row                  | Connector enabled, correct label, poller schedule, credential    |
| Linear remains Todo          | Linear writeback state and credential permissions                |
| Linear remains In Progress   | GitHub credential, repo write access, PR creation result         |
| Duplicate comments           | Idempotency regression or another watcher process                |
| No PR link in Runs           | Delegation result metadata or stale admin deploy                 |
| Old cancelled rows are noisy | Toggle **Show cancelled** or use the cleanup path in the runbook |

Then use `docs/runbooks/computer-first-linear-connector-checkpoint.md` for SQL checks and stale-run cleanup.

## Exit criteria

The checkpoint passes when one fresh Linear issue labeled `symphony` reaches `In Review` and shows exactly one terminal connector execution, one completed Computer task, one `connector_work_received` event, one Computer-owned thread, one completed delegation, one succeeded turn, one branch, one draft PR, one dispatch comment, one PR comment, and one Symphony Runs row with `Linear: In Review`.
