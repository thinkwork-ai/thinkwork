Custom connectors bring an external system into ThinkWork without turning that system into a one-off agent integration. A good connector normalizes inbound work, records provenance, hands ownership to a Computer, and writes results back through tenant-scoped credentials.

Use this guide when planning a connector beyond the shipped Linear/Symphony path. If you only need to operate the current Linear connector, start with [Admin: Symphony](/applications/admin/symphony/) and [Run the Symphony Linear Checkpoint](/guides/symphony-linear-checkpoint/).

## Before you build

Decide which connector shape you are building:

| Shape               | Use when                                                                                   |
| ------------------- | ------------------------------------------------------------------------------------------ |
| Tracker connector   | The external system owns work-item lifecycle and ThinkWork reconciles it on a poll cadence |
| Webhook integration | The external system pushes signed events to ThinkWork                                      |
| Channel integration | Users message ThinkWork through a product surface such as chat or email                    |
| MCP tool connector  | ThinkWork workers need tools they can call during a turn                                   |

The current production tracker example is Linear. Future providers should reuse its ownership and idempotency model rather than inventing a provider-specific shortcut.

## Core responsibilities

A connector is responsible for:

1. **Authenticating the provider boundary.** Poll with a tenant credential or verify inbound signatures before processing.
2. **Normalizing provider data.** Convert provider-specific payloads into a small ThinkWork candidate: title, body, external reference, provider metadata, and writeback targets.
3. **Claiming idempotently.** Use a stable external reference so repeated polls or webhook retries do not create duplicate visible work.
4. **Handing work to a Computer.** The normal owner is a configured Computer. Direct managed-Agent, routine, or hybrid-routine targets are advanced paths.
5. **Creating or updating a thread.** The thread is the visible work artifact.
6. **Writing back through credentials.** Provider comments, status transitions, labels, PR links, or replies use tenant credentials, not raw tokens exposed to the worker.
7. **Recording outcomes.** Store enough metadata for the admin UI to show status, errors, branches, PRs, and cleanup state.

## Computer-first target

The happy path for user-owned external work is:

```text
Provider event or poll result
    -> connector execution
    -> connector_work Computer task
    -> Computer-owned thread
    -> optional managed-Agent or routine delegation
    -> provider writeback
```

The external system may still be the canonical owner of lifecycle. Linear, for example, owns the issue state; ThinkWork reconciles and writes back `In Progress` or `In Review` at the right points.

<Aside type="note">
  Do not start a new connector by choosing a default managed Agent. Pick the
  Computer that owns the work, then let the Computer delegate if the task needs
  a managed Agent or routine.
</Aside>

## Credential handling

Use tenant credentials for provider access. A connector should refer to credentials by id or slug in its config, then resolve the secret at runtime through the credential store.

Avoid patterns that copy provider tokens into:

- thread metadata;
- task input visible to workers;
- skill files;
- Lambda environment variables that are not explicitly deployment-owned secrets;
- provider-specific ad hoc storage.

For the Linear/Symphony proof, the connector uses one credential for Linear and one credential for GitHub PR-producing writeback. The setup form makes both explicit so a missing GitHub credential is visible before the connector is enabled for PR work.

## Idempotency

Connector retries are normal. Pollers run repeatedly, webhooks retry, and operators may use "Run now" while debugging.

Design each connector around:

| Concern            | Requirement                                                          |
| ------------------ | -------------------------------------------------------------------- |
| External reference | Stable provider id or composite key for the same item                |
| Claim              | Only one active execution should own a provider item                 |
| Computer task      | Use an idempotency key derived from connector and external reference |
| Thread             | Reuse or link the same provider item to one visible thread           |
| Provider comments  | Do not post duplicate "working" or "done" comments on retry          |
| Provider writeback | State transitions should be safe if repeated                         |

The Linear checkpoint is the reference behavior: repeated polling of the same issue should not create duplicate branches, PRs, comments, tasks, events, or threads.

## Provider writeback

Writeback is part of the connector contract, not a random tool call. The connector config should make provider writeback explicit:

- which credential to use;
- which provider states to write;
- which provider comments to post;
- which external links to attach;
- what to do when the provider rejects the update.

For Linear/Symphony, dispatch writeback moves the issue to `In Progress` after ThinkWork claims it. PR-opened writeback moves the issue to `In Review` only after the draft PR exists.

## Minimal recipe

A custom connector recipe usually needs:

- connector config metadata;
- tenant credential requirements;
- poller or webhook handler;
- candidate normalizer;
- idempotent claim logic;
- Computer handoff;
- thread creation or resume behavior;
- provider writeback adapter;
- admin visibility for status and failures;
- runbook or checkpoint docs.

The `examples/connector-recipe/` directory remains a starting point for custom connector structure, but treat it as a template to modernize for the Computer-first connector model.

## Testing checklist

Before shipping a connector:

- A valid provider event creates exactly one connector execution.
- A duplicate provider event creates no duplicate visible work.
- A missing or inactive credential fails clearly.
- A connector targeting the wrong tenant cannot read or write another tenant's data.
- A successful handoff creates a Computer task/event and a visible thread.
- Provider writeback failure is visible in outcome metadata.
- The admin UI can show success, failure, and cancelled/stale states without SQL.

## Known limits

- **Linear is the only current production tracker connector.** Other providers should follow the model, but they need their own plans and tests.
- **The recipe example is illustrative.** It is not a generated connector SDK.
- **Custom connector UI is not fully catalog-driven yet.** The Symphony form is structured for the Linear proof; broader custom setup UX is future work.

## Under the hood

Connector runtime code lives in `packages/api/src/lib/connectors/`. The scheduled poller entry point is `packages/api/src/handlers/connector-poller.ts`. Tenant credential secret resolution lives under `packages/api/src/lib/tenant-credentials/`. Connector GraphQL resolvers live under `packages/api/src/graphql/resolvers/connectors/`.

The example recipe directory is `examples/connector-recipe/`. If you update it for a real provider, keep its README aligned with this guide and avoid provider-token examples that bypass tenant credentials.
