The ThinkWork React Native SDK (`@thinkwork/react-native-sdk`) gives host apps — LastMile, ThinkWork mobile, any third-party consumer — a uniform hook surface for talking to a ThinkWork stage. It handles Cognito auth, the GraphQL client, AppSync subscriptions, and token lifecycle internally.

**It ships no UI and no host-specific vocabulary.** Consumers build their own chat surface from the hooks.

## What's in the box

| Category | Hooks |
| --- | --- |
| Auth | `useThinkworkAuth` |
| Agents | `useAgents` |
| Threads (read) | `useThreads`, `useThread`, `useUnreadThreadCount` |
| Threads (write) | `useCreateThread`, `useUpdateThread` |
| Messages | `useMessages`, `useSendMessage` |
| Subscriptions | `useNewMessageSubscription`, `useThreadTurnSubscription`, `useThreadTurnUpdatedSubscription`, `useThreadUpdatedSubscription` |

Plus the root `ThinkworkProvider` and exported types for `Agent`, `Thread`, `Message`, `CreateThreadInput`, `UpdateThreadInput`, and event shapes for each subscription.

## Design principles

1. **Hooks-only.** The SDK never renders UI. Host apps build their own visual language on top.
2. **Imperative mutations, not tuples.** `useSendMessage`, `useCreateThread`, `useUpdateThread` return callbacks you invoke at event time. Cleaner in typical `onPress`/`onSubmit` handlers than urql's tuple convention.
3. **Cache coherence by default.** Every hook that reads threads registers the `Thread` typename in urql's document cache, so mutations automatically invalidate dependent queries. No manual refetch plumbing.
4. **Selection and storage stay host-side.** The SDK returns raw data (agents, threads, messages) and trusts hosts to handle UI state (selection persistence, optimistic updates, scroll position). Different hosts want different choices; the SDK doesn't dictate.
