Grounding Context

Codebase context. The eval framework is not a greenfield mess — it is a substantial, recently shipped system (THNK-2 "Trust Core" + skill evals, June 2026) whose pain concentrates in a few specific seams. What exists: an S3-canonical dataset store with versioning and drift detection (packages/api/src/lib/evals/dataset-store.ts); SQS per-case fan-out execution (eval-runner.ts dispatcher → eval-worker.ts → eval-runs-reconciler.ts stall backstop); the settled pass | fail | error verdict taxonomy with five error_cause classes excluded from pass-rate math; a pluggable, engine-neutral ScoringEngine contract whose only live engine is the in-house scorer (deterministic string assertions + LLM rubric judge) — the AgentCore Evaluations adapter is a stub that makes no API calls; per-skill datasets with an isolated eval-baseline agent, advisory-lock serialization, and an update-swap gate (migration 0168); a flagThread flow converting bad production threads into eval cases with workspace-projection snapshots; and a 189-case red-team baseline covering four security categories, seeded at install and never refreshed.

Where it hurts (from five evidence dossiers over the eval code): the framework's only true-crash alarm has alarm_actions = []; every case runs exactly once with no trial concept; every case pays a hardcoded 30-second span-propagation sleep; batch skill installs score only the first skill (busy → marked failed, never retried); the skill gate compares two single-trial pass rates against one per-tenant absolute threshold; flagging is manual-only with flag-time evidence capture that races trace retention; the attribution:fallback quarantine tag is written but consumed by nothing; performance-as-a-category was shipped in U7 and deleted in U19; agent-turn cost is invisible (only evaluator tokens are priced); and the UI surfaces pass-rate only.

Direction of interest. Eric's research comment on THINK-107 (2026-06-30) points at a Harbor-style harness: task = initial environment + instruction + deterministic verifier script; attempts/trials with aggregate scoring; a pluggable AWS-native sandbox provider (AgentCore Code Interpreter vs Lambda MicroVMs); security/functionality/performance/cost in the result schema from the start. External research strongly corroborates the shape: Inspect AI's decoupled Task→Solver→Scorer with pluggable sandboxes, Terminal-Bench's all-or-nothing verifiers averaged over 5 trials, SkillsBench's paired with-skill/no-skill methodology, the LangSmith/Braintrust production-traces→annotation-queue→regression-suite loop, and AWS's own guidance to prefer code-based evaluators over LLM judges for anything deterministic.

Institutional guardrails honored throughout: the verdict taxonomy is settled (don't relitigate); "rated ≠ evaluable" is a named pattern; write-safe replay is a two-layer defense; Trust Core's deferrals (live sampling, deploy gates, AgentCore activation) are sequenced, not rejected; scoring stack stays AWS-native.

1. Deterministic-first scoring ladder

Rank 1scoring-verifiersConfidence 85%Complexity Low→Medium
Description
Make deterministic evidence the default verdict source and demote the LLM judge to the cases that genuinely need semantic judgment — then calibrate the judge itself. Three moves, all behind existing seams: (a) add a verifier ScoringEngine — a deterministic script (sandboxed Lambda) that inspects post-run state: workspace files, the tool-call trace ledger, captured artifacts — as a third engine behind the existing engine-neutral ScoringEngine contract, generalizing the already-shipped workspace-projection-assertions miniature. (b) Emit the write-guard events the replay allowlist already produces (every blocked write attempt, every guarded-prefix read) as first-class scoring inputs, so tool-misuse and data-boundary cases verdict deterministically: pass iff zero violations — no judge call at all. (c) Turn operator verdict overrides (stored immutably next to judge verdicts today, used for nothing) into the judge-calibration corpus: publish a judge-agreement rate, and gate scoring_version bumps on the new judge matching the override ground truth.
Basis
direct: engine.ts:90-94 — "Scoring sits behind this ThinkWork-owned seam so engine-specific concepts can never leak into the dataset format or the verdict taxonomy" (the stub AgentCore engine proves the seam tolerates inert engines); replay-allowlist.ts already classifies and blocks every tool call during eval; in-house.ts:200-213 — the heuristic judge fails closed on non-refusal rubrics when the LLM is unavailable, making the judge the single point of fragility; overrides kept immutable at resolvers/evaluations/index.ts:1413-1425. external: Inspect AI (Task→Solver→Scorer, pluggable sandboxes); Terminal-Bench all-or-nothing verifiers; AgentProp-Bench — judge errors cascade through multi-step trajectories, so step-level deterministic checks beat end-to-end LLM verdicts; AWS's own AgentCore Evaluations guidance: code-based evaluators for anything deterministic.
Rationale
This is the concrete on-ramp to the Harbor direction in Eric's THINK-107 comment without a rearchitecture: task-plus-verifier becomes expressible inside the current runner/worker pipeline, proving the verifier contract before any sandbox-pool investment. It attacks reliability at the root — every verdict moved off the judge is a verdict that cannot be judge-flaky — and an attempted forbidden write is strictly stronger evidence of tool misuse than a judge's reading of the transcript. It also unlocks the functionality category (idea 3's golden cases need verifiers like "the agent actually updated the file," which contains-assertions on prose cannot express).
Downsides
Verifier scripts are a new authoring surface (skill authors and operators must learn to write them); guard-event scoring crosses the eval-worker/Pi-runtime boundary, so both owners are involved; judge calibration needs enough override volume to be meaningful and overrides may be biased toward disagreements.

2. Skill certification pipeline

Rank 2skill-evalsConfidence 82%Complexity Medium→High
Description
Rebuild skill-eval trust as a pipeline with three stages. (a) Eval passport at publish: mint a certification per skill-version content SHA when a skill enters the catalog — WIRING.md present, materializes cleanly into the isolated workspace, smoke fixtures pass, scoring version stamped. Activation and the "Run evals" button check passport validity instead of lazily probing S3 at click time, making the shipped "rated but not evaluable" failure unrepresentable. (b) Paired-condition delta gating: run the same cases with-skill and no-skill (the control arm already exists as the post-purge, pre-install workspace state) and gate skill updates on the delta, not the single per-tenant absolute threshold — the only measurement that can catch a net-negative skill. (c) Concurrency: replace EvalBaselineBusyError → marked failed with, minimally, a per-tenant fair queue (surfacing "queued," retrying automatically), and, fully, a bounded pool of ephemeral isolated eval environments per Eric's sandbox-pool note — turning batch installs from "only the first skill gets scored" into parallel certification.
Basis
direct: the rated≠evaluable pattern doc (docs/solutions/architecture-patterns/skill-eval-rated-does-not-mean-evaluable-2026-06-15.md): "Clicking Run evals now then failed in the backend because the catalog skill had no WIRING.md"; skill-eval-run.ts:207-217 (verified verbatim): busy is treated identically to permanent failure; migration 0168's single per-tenant numeric(5,4) threshold; regression = a two-run pass-rate comparison at skill-eval-run.ts:285-314. external: SkillsBench (arXiv 2602.12670) — paired-condition skill eval as the core primitive; skill value is wildly domain-dependent (+4.5pp on SWE vs +51.9pp on healthcare tasks) so no absolute bar generalizes, and self-generated skills were net negative; Anthropic's skill-authoring guidance — every skill ships its own eval fixture.
Rationale
This is the direct answer to "the skill eval feature has been particularly problematic." Each documented failure maps to a stage: rated-but-not-evaluable → passport; a gate that can't tell "mediocre because the domain is hard" from "actively worse than no skill" → delta; batch installs silently under-delivering → queue/pool. Certification keyed by the same content-SHA drift machinery the catalog already uses makes it auditable, and it becomes the natural enforcement point for requiring future catalog skills to ship eval fixtures.
Downsides
Paired runs double skill-eval cost and duration; a publish-time hard gate changes the skill-contributor workflow (needs a clear authoring error surface, or it just moves the frustration); the ephemeral-pool leg carries real infra cost (IAM, provisioning latency, provider bake-off) — hence the fair-queue interim.

3. Dataset flywheel

Rank 3thread-flagging-dataset-curationConfidence 80%Complexity Medium→High
Description
Turn "flagging threads for evaluation dataset" from a manual button into a closed loop with four stages. (a) Capture at dispatch, not at flag: THNK-10 already writes the workspace projection into thread_turns.context_snapshot at dispatch — extend that to a complete eval-ready capture (history refs, trace pointers, active skills) on every turn, so flagging becomes promoting an existing immutable snapshot and the flag-time evidence-rot gap codes become impossible for new threads. (b) Sentinel sampling: a cheap always-on scorer (deterministic trace checks — failed tool calls, guard blocks, error loops — plus budgeted LLM sampling) over completed production turns routes low scorers into a triage queue; the flag flow becomes review of machine-nominated candidates, not archaeology. This un-defers Trust Core's explicitly "sequenced, not rejected" live-sampling deferral. (c) Reproduce-before-gate quarantine: a flagged case enters as "reported," is replayed once as a positive control, and only counts toward gates once the failure reproduces under its captured snapshot — giving the attribution:fallback tag (written today, consumed by nothing — verified by grep) its promised audit consumer. (d) Golden promotion: flag good threads too — successful production threads become functionality cases via the existing resolution_target / outcome_kind fields, bootstrapping THINK-107's never-scoped functionality category from real tenant workloads instead of hand authoring.
Basis
direct: thread-snapshot.ts — "captures everything replay (U8) needs into self-contained payload objects" (a callable module, currently human-triggered only); flag-time gap code "Trace ledger evidence lookup failed at flag time"; flag-thread.ts:96-104 quarantine intent with zero consumers; dataset-store.ts:351-435 — resolution_target/outcome_kind: security | quality already parse; the 189-case baseline is "seeded at install, never auto-refreshed." external: LangSmith/Braintrust closed loop — online scorers → annotation queue → ~100–200 annotated traces recalibrate the judge → the annotated set becomes the deploy-gating regression suite; Braintrust code-defined dataset pipelines for bulk promotion.
Rationale
Every downstream component (snapshot capture, three-way attribution, S3-canonical sync, drift healing) is already built and fed by human attention alone. Automating discovery is the highest-leverage change to the pipeline Eric named, and quarantine is what makes it safe: without reproduce-before-gate, auto-nominated cases would pour unverified provenance into the datasets that block skill updates. The flywheel compounds — every day of production traffic grows the corpus, and every fixed bug leaves a regression tripwire behind.
Downsides
Always-on scoring is a new cost/privacy surface over production traffic (needs sampling budgets and tenant consent posture); dispatch-time capture increases per-turn storage; quarantine adds a state machine to case lifecycle; snapshot content is still unredacted (a Trust Core deferral this would inherit).

4. Trial economics: the 10-second eval × k-trials

Rank 4run-execution-reliabilityConfidence 80%Complexity Medium
Description
Two moves that only make sense together. (a) Make cases fast: the runner always sleeps 30 seconds for CloudWatch span propagation even when spans are already available — the documented dominant stage at every percentile. Score deterministic assertions against the direct invocation response immediately (a provisional verdict in seconds) and let LLM-judge/span-dependent enrichment refine asynchronously — a two-phase verdict. (b) Then afford trials: add a trial_index to the existing per-case SQS message (already batch-size-1 with advisory-lock dedup), fan out k messages per case, and aggregate binary reward in the unchanged last-writer finalization. Verdict instability across identical trials becomes a new unstable state — quarantined from pass-rate and gate math the same way error already is, extending the settled taxonomy rather than relitigating it.
Basis
direct: docs/solutions/diagnostics/eval-runner-stall-findings-2026-05-16.md (verified by the Phase-3 checker): "the runner always sleeps for the 30 second initial span wait even when spans are available" — ~30s at p50/p95/max; single-trial design confirmed across eval-worker.ts/eval-runner.ts (no trial concept exists); fan-out mechanics at eval-fanout.tf:33-50. external: Terminal-Bench 2.0 — binary reward averaged over 5 trials as the reliability standard for agent evals.
Rationale
Per-case wall time is the hidden tax on every other idea in this document — trials, paired-condition runs, and sentinel sampling all multiply case count, so cutting the floor 3–5× changes what is affordable. And the skill gate currently compares two single-trial pass rates: noise gating noise. Trials make the gate statistically honest, flake quarantine separates "flaky case" from "real regression" (a large share of the felt mess), and attempts/trials is an explicit requirement in Eric's Harbor note.
Downsides
k-trials multiplies compute cost on gate-triggering runs (mitigated by only applying k>1 to LLM-rubric cases — deterministic verdicts don't flake); two-phase verdicts change result-row semantics and need UI treatment for "provisional"; the async enrichment path adds a completion state to reconcile.

5. Verified run bundle

Rank 5run-execution-reliabilityConfidence 85%Complexity Small→Medium
Description
Make run trust visible without re-architecting execution. (a) Pre-flight scope contract: the runner writes a signed manifest — exact case IDs, dataset version, scoring version, eval-workspace content SHA — as the run's first artifact; fan-out is blocked until it validates against the request; finalization diffs results against it, and any delta becomes a loud scope_mismatch terminal state instead of today's console.warn plus an eternally "running" run. The UI renders "N of exactly M contracted cases." (b) Wire the DLQ alarm (alarm_actions = [] today) to SNS→Slack/PagerDuty and auto-run the existing stall-probe's per-phase latency breakdown on trigger, so the operator starts from the bottleneck, not a silent dashboard tile. (c) Simple anomaly signals on error_cause: per-run rates of timeout/throttle/evaluator_error charted with plain thresholds on the existing 30-day trend — the taxonomy exists precisely to separate infra noise, but nothing consumes it as a signal today.
Basis
direct: eval-fanout.tf:90 alarm_actions = [] (verified in-repo by the Phase-3 checker) on the alarm whose own description says "operator must inspect"; eval-runs-reconciler.test.ts:455-460 — a scope discrepancy today produces a warn-and-skip; two shipped silent-scope incidents (a single-case run expanded to 96; a full run silently executed 135/189) documented in docs/solutions/logic-errors/. reasoned: a gate that silently evaluated the wrong thing is worse than no gate — pre-flight echo of effective scope is the only mechanism that catches this class at t=0; the pinning machinery (pinned_case_ids, content SHAs) already computes everything the manifest needs.
Rationale
Scope bugs are the most trust-corrosive failure mode a gating system can have, and this converts the scariest historical bug class into an impossible state mostly by promoting internal state to a verified, surfaced invariant. The alarm fix is the single cheapest reliability win in the framework. Together they answer "reliably run evals" for the runs that already exist.
Downsides
A hard pre-dispatch gate can fail runs that today would (incorrectly) proceed — the team must agree that failing loud beats running wrong; anomaly thresholds need tuning at current low run volumes (full SPC/control-chart statistics were rejected as scale-mismatched — see rejections).

6. Performance + cost as universal telemetry, not authored categories

Rank 6category-coverageConfidence 82%Complexity Medium
Description
Stop treating performance and cost as test-case categories to author — that approach shipped in U7 and was deleted in U19 ("not true red-team"). Instead, attach latency and cost as automatic instrumentation on every eval that already runs: per-tool-call and per-turn spans (tokens, cost, wall time) stored in result evidence; p50/p95 latency and cost-per-case trends computed across the 189 red-team cases and every skill run; regression detected against rolling per-case baselines (2× spike rule). A cost ceiling gate clones the proven presence-gated threshold-table pattern from skill-eval-gate.ts. Skill evals gain a cost-lift axis for free: does this skill make the agent cheaper or 3× more expensive?
Basis
direct: duration_ms is "captured per result but not aggregated or surfaced in run summaries"; "cost aggregation is Bedrock evaluator token-based only; no per-case agent invocation cost tracked"; the U7-shipped/U19-deleted performance category is documented in the plan history; skill-eval-gate.ts:38-53 — gate-row presence = enabled. external: cost-regression tooling convergence (Traceloop, Braintrust, TrueFoundry): span-level token capture per tool call, rolling-baseline spike detection, cost as a fourth CI gate alongside pass-rate/latency/security.
Rationale
Two of THINK-107's four demanded dimensions light up with zero new test cases, sidestepping the exact failure mode that got the U7 attempt deleted. Category becomes a filter over universal metrics instead of a content-creation burden competing with red-team authoring, and the rolling baselines it establishes are prerequisites for any future cost/latency gate.
Downsides
Requires threading token/cost capture through the agent-invocation path (the cost_events wiring that was never finished); latency measured under eval conditions differs from production latency (fine for regression, misleading as an absolute); adds result-row width that the schema should absorb deliberately (see idea 7).

7. Per-category trust budgets replace the pass-rate headline

Rank 7category-coverageConfidence 68%Complexity Medium
Description
Retire the single pass-rate as the tenant-facing headline. Each category gets a budget contract with its own semantics: security = zero successful injections (any fail is a breach — 99% is not a pass); functionality = golden-case delta ≥ 0; performance = p95 ≤ bound; cost = per-case cost ≤ rolling baseline × 2. The dashboard renders contracts met/breached; pass-rate survives only as a drill-down inside categories where averaging is meaningful. This is the reporting layer over ideas 1 and 6, and it decides the result-schema shape Eric's Harbor note says must be right "from the start."
Basis
reasoned: a mean over incommensurable categories destroys information — one successful prompt injection amid 188 passes renders as 99.5% ("excellent") when it is a critical breach, while ten flaky timeouts render as a five-point regression when nothing behavioral changed. The institutional record shows this misreading already burned the team once: the documented 95%→62% slide that triggered Trust Core "blends real failures, brittle assertions, and infra noise indistinguishably." Budget-per-category is how CI already treats security scanners vs perf benchmarks. external: cost-tooling convergence on cost as a fourth gate, not a dashboard.
Rationale
Defines what "MUCH better coverage" actually means to a tenant and forces the right result-schema decision before the other ideas harden it. Without this, every improvement upstream still funnels into a metric that invites the wrong conclusion.
Downsides
A genuine product/UX redesign of the evaluations surface, not a backend change; per-category semantics need agreement (what counts as a security "breach" vs a heuristic false positive); Medium confidence because the *shape* is right but the specific budget definitions need design work with real data.
